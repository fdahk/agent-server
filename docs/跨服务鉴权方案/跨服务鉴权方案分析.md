# 跨服务鉴权方案分析 ── 索引与全局视图

> **目标读者**:正在判断"如何让 our-chat web 前端调 agent-server"的工程师。
> **关键约束**:agent-server 是纯后端 API,**多端独立消费**(Web + Flutter + 未来其他客户端)是核心设计原则,任何鉴权方案都不能损害这一点。

本目录把 8 个方案各拆成独立文档,本文是**索引 + 全局视图**(场景 / 范式 / 横向对照 / 推荐路径)。各方案的原理 / 实现 / 优劣 / 踩坑 见对应文档。

## 0. 场景与约束

### 0.1 现状

| 项 | our-chat | agent-server |
|---|---|---|
| 鉴权 | HttpOnly cookie(token)+ 双提交 CSRF | `Authorization: Bearer <JWT>` |
| 端口(dev) | `:3007` | `:3000` |
| 客户端 | 已有 Web,规划 Flutter | 复用 our-chat 客户端 + 未来独立 Flutter |
| Token 协议 | 自家约定,未公开载荷字段 | 标准 JWT(HS256) |
| 401 处理 | 拦截器自动 refresh(/api/refresh)+ 重试队列 | 401 由客户端自行处理 |

### 0.2 三类客户端必须等价支持

agent-server 的 06 文档定义了"**纯后端、多端独立消费**"的契约面。任何鉴权方案的合格性必须用三种客户端检验:

- **浏览器内 SPA**:能自动带 cookie,有 JS 环境,有 XSS 风险
- **移动端原生**(Flutter / Swift):**没有 cookie 容器**,只能塞 HTTP 头,无 XSS 但 token 可能从 keychain 泄漏
- **服务端 / CLI / Postman**(测试 / 第三方集成):必须能用一条 curl 调通

**任何只在浏览器顺畅、在 Flutter 要绕一圈的方案,都是契约污染。**

### 0.3 安全模型基线

- 假定 agent-server 是 *trusted backend*,我们对它的请求/响应内容信任度高
- 浏览器内 JS 上下文存在 XSS 风险(任何 npm 依赖被投毒都可能触发)
- 网络层假定 HTTPS 已就位(本文不讨论裸 HTTP)
- 不讨论 CSRF 防御细节(假设 SameSite=Lax 默认 + 同源策略)

## 1. 鉴权范式总览 ── 三个维度的分类轴

跨服务鉴权方案虽然命名繁多,本质上只有三个判别维度:

### 1.1 信任传递载体

| 载体 | 谁主动放进请求 | 浏览器友好 | Native 友好 |
|---|---|---|---|
| `Authorization: Bearer <token>` 头 | 客户端代码 | ✅ | ✅ |
| Cookie(自动) | 浏览器 | ✅ | ❌ |
| Cookie(手动) | 客户端代码 | ✅ | 反模式 |
| 自定义头(`X-Auth-*`) | 客户端代码 | ✅ | ✅ |

> **结论 1**:要求多端等价 → token 必须经 `Authorization` 头送达,任何依赖浏览器自动行为(cookie)的方案都是浏览器偏好方案。

### 1.2 信任验证方式

| 方式 | 签名/验签密钥关系 | Token 撤销难度 |
|---|---|---|
| **对称签名**(HS256/HS384) | 签发方与验证方共享同一 secret | 难(无状态) |
| **非对称签名**(RS256/ES256) | 签发方私钥,验证方公钥(经 JWKS) | 难(无状态) |
| **服务端 session(Stateful)** | 验证方查会话库(Redis/DB) | 易(删 session) |
| **Token Introspection**(RFC 7662) | 验证方实时问签发方 | 易(签发方决定有效性) |

> **结论 2**:JWT 的 *无状态* 是双刃剑——好处是验证 O(1)、零网络往返;坏处是签发了就难撤销,只能等过期。这是几乎所有 JWT 方案绕不开的根本权衡。

### 1.3 信任来源(谁负责签发)

```
        ┌────────────────────────────────────────────────────┐
        │     who signs the token that agent-server trusts?  │
        └────────────────────────────────────────────────────┘
              │
   ┌──────────┼──────────────────────────────────┐
   ▼          ▼                                  ▼
agent-server  our-chat(把信任传递给 agent-server)  独立 IdP
自己签发      = 跨服务但同密钥                     = OAuth2 / OIDC 范式
(独立账号)   = 方案 B/C/D                         = 方案 F/G
```

## 3. Web vs Native 的关键差异

这是评估鉴权方案时**必须分别推演的两栏**。

| 维度 | Web (Browser) | Native (Flutter/Swift) |
|---|---|---|
| Token 存储 | localStorage(XSS 风险) / 内存 / HttpOnly cookie | Keychain / Keystore(系统级保护) |
| 自动带 cookie | ✅ 浏览器原生 | ❌ 无 cookie 容器(需手动塞 `cookie` 头) |
| `Authorization` 头 | ✅ | ✅ |
| SSE / WebSocket | EventSource 不能塞自定义头 → 用 `fetch + ReadableStream` | dio / OkHttp 自带 stream + 自定义头 |
| OAuth2 redirect 体验 | popup / 同窗口 | 需 ASWebAuthenticationSession / Chrome Custom Tab |
| Deep Link 回调 | 简单(URL) | 复杂(scheme / Universal Link) |
| XSS 风险 | 高 | 几乎无 |
| CSRF 风险 | 有(若用 cookie) | 无(没有 cookie 容器) |

> **结论**:**任何依赖浏览器 cookie 自动行为的鉴权方案,在 Native 上要么报废、要么走另一条路径。** Bearer 头是唯一对两端等价的载体。

## 5. 踩坑清单(跨方案通用)

### 5.1 Token 存储

| 位置 | XSS 风险 | CSRF 风险 | 备注 |
|---|---|---|---|
| localStorage | ❌ 高(任何 JS 都能读) | ✅ 无 | 不推荐 token,但很多教程在用 |
| sessionStorage | ❌ 同上 + 关闭 tab 即失 | ✅ 无 | 短期 token 可用 |
| 内存(JS 变量) | ✅ 中(组件树外被 prototype 污染可读) | ✅ 无 | 刷新页面就丢 → 配合 refresh token 用 |
| Cookie HttpOnly | ✅ 高(JS 读不到) | ❌ 必须 CSRF token | 经典方案 |
| Cookie non-HttpOnly | ❌ 高 | ❌ 高 | 方案 B 用,需配短 expiry + CSP |
| Service Worker 缓存 | ✅ 高(SW 是独立线程) | ✅ 无 | 复杂,2024+ 才慢慢有库支持 |

### 5.2 SSE / WebSocket

- EventSource 不能塞自定义头 → `Authorization` 不能用 → 要么用 cookie(C 方案的少数收益),要么 fetch + ReadableStream
- WebSocket 升级请求可以塞头(但 `?token=` query 参数会进 nginx access log,不要这么干)
- 长连接的 token 续期:连接已建立后 token 过期,如何 graceful 重连?需要应用层重连 + token 刷新协调

### 5.3 Refresh token rotation

- OAuth2 Best Current Practice:每次 refresh 都返回**新的** refresh token,旧的立即作废
- 防偷 refresh token:作废链可检测出"被偷的那次使用"
- 实现复杂度:每个 refresh 都要原子地"作废旧 + 签新 + 写数据库"

### 5.4 JWT 注意

- 永远 verify algorithm,不要接受 `none`(著名历史漏洞)
- `aud` / `iss` claim 必须验证,不然别人签的 token 也能用
- 时钟漂移容忍:`clockTolerance` 设 5–30 秒,不要为零
- 用户密码改了 / 被封了 → JWT 还有效:解法是缩短 expiry + 黑名单 / 心跳查 DB

### 5.5 CORS + Cookie 组合

- `Access-Control-Allow-Origin: *` 与 `Allow-Credentials: true` **不能同时**(浏览器拒绝)
- 跨域带 cookie 必须精确 origin
- 预检请求 OPTIONS 也要带 `Access-Control-Allow-Credentials: true`

### 5.6 校招面试常被追问的

1. "你为什么把 token 放 localStorage?XSS 怎么防?"
   - 答:短期 access token + HttpOnly refresh token + CSP,或上 BFF
2. "JWT 怎么撤销?"
   - 答:短 expiry + 黑名单(Redis sentinel)+ 用户 ID 索引,或 token introspection(放弃无状态)
3. "为什么不用 OAuth2?"
   - 答:看场景。单产品 MVP 用 OAuth2 是过度设计;多产品 / 第三方集成必须 OAuth2
4. "PKCE 解决了什么问题?"
   - 答:authorization code 中间人攻击。public client 没有 client_secret,只能用 verifier-challenge 证明持有者
5. "为什么 SPA 上的 PKCE 不够安全?"
   - 答:PKCE 防 redirect 链路拦截,不防进程内攻陷;SPA 的 JS 上下文是同一进程,XSS / 供应链投毒 / 扩展都能读 verifier/token。业界 2024+ 推 BFF 把 token 从 JS 搬到服务端

### 6.3 为什么不直接上 OAuth2?(短答案)

**OAuth2 对纯 Web MVP 是过度设计**(完整展开见 §7):

- 单一前端 + 两个后端 + 共享一套用户的场景,完整 OAuth2 流体验上多一次跳转
- 工程上多 1–2 周时间起 IdP / 改造现有登录页
- 校招 portfolio 写 "我用了 OAuth2" 在面试中会被追问 "你为什么需要 OAuth2?"——如果答不出第三方集成 / 多产品矩阵的诉求,反而扣分

**但如果有 Flutter 等多端规划,OAuth2 是不可避免的——见 §7。**

### 6.4 关于本项目的最终选型

按目标分:

- **纯 Web MVP**:**B → D 两步走**

   2. 后续 D:our-chat 切 RS256,起 JWKS 端点;agent-server `jwks-rsa` 改 5 行,平滑切换

- **多端 + 校招深度**:**直接 F**(简化变体,our-chat 兼任 IdP)
   1. our-chat 加 `/oauth/authorize` + `/oauth/token`
   2. agent-server 走 JWKS 验签
   3. 前端跑 PKCE 流(`oidc-client-ts`),Flutter 后续接 `flutter_appauth`

### 6.5 长期不推荐的方案

- **方案 C**:污染 agent-server 契约,部署绑同根域,Flutter 不友好

## 7. 重审:OAuth 2.0 是不是最佳实践?(诚实补刀)

> 这一节是对 §2–§6 的**自我审计**——重新推导后修正了之前部分隐含假设。保留以示思考路径。

### 7.1 两个核心问题,直接答

**问题 1:OAuth 2.0 是不是最佳实践?**

是,**但前提是有真实的"多端 + 多服务 + 长期演进"诉求**。

- 若只 MVP Web 端打通:OAuth 2.0 是过度设计
- 若长期会有 Flutter / 桌面 / IoT 多端:OAuth 2.0 是**唯一**真正等价的方案
- 校招 portfolio 含金量排序:**F > D > B > A 严格单调递增**

**问题 2:之前的方案分析对多端兼容性是否考虑充分?**

**没有充分。**坦白承认:§6 推荐 B 起步时,有一个**隐含的默认前提**——"Web 先跑通,Native 端后说"。这个前提没明说,且对一个有 Flutter 规划的项目,可能不成立。

下面把这个漏洞暴露出来,重新推。

### 7.2 多端等价性硬考察:B/C/D 在 Native 上变成什么

之前每个方案都对 Web 做了详细设计,但 Native 端实际形态没逐个推。补上:

| 方案 | Web 端形态 | Native 端实际形态 | 多端等价? |
|---|---|---|---|
| A | 独立登录 | 独立登录 | ✅(都不爽) |
| **B** | 登 our-chat → 浏览器存 non-HttpOnly cookie → JS 读 cookie 塞头 | **Flutter 没浏览器 cookie**,凭什么"登 our-chat 就有 agent JWT"?要么独立登录 agent-server,要么前置 OAuth 流 | ❌ |
| **C** | 登 our-chat → 浏览器自动带 cookie | 同上,Flutter 没 cookie 容器 | ❌ |
| **D** | 同 B(JWKS 只换签名算法,不改 token 传递载体) | 同 B | ❌ |
| E | OAuth Token Exchange | OAuth Token Exchange | ✅ |
| F | PKCE 经浏览器跳转 IdP | PKCE 经系统浏览器 (ASWebAuthenticationSession / Chrome Custom Tab) | ✅ |
| G | BFF + session cookie | Native 直连 IdP,绕过 BFF | ⚠ 半等价(协议同,实现路径异) |

**关键洞察**:B/C/D 表面"用户一次登录两边通"的收益,**严格说只在浏览器成立**——它们利用了浏览器自动 cookie 行为这个 free lunch。Flutter 没有这个 free lunch,**必须用别的协议补**。

补的最自然路径就是 OAuth 2.0——这就是为什么 OAuth 2.0 不可避免。

### 7.3 为什么 OAuth 2.0 是多端等价的标准答案

OAuth 2.0 框架最早(**RFC 6749, 2012**)只规范了 Web 端。Native 端遇到的麻烦催生了一系列后续 RFC,构成完整的"多端鉴权 RFC 族":

| RFC | 内容 | 解决什么 |
|---|---|---|
| RFC 6749 | OAuth 2.0 主框架 | Web 端基础流 |
| RFC 7636 | PKCE | 无 client_secret 客户端的 code 偷取防护 |
| RFC 8252 | **OAuth 2.0 for Native Apps** | iOS/Android 必须用系统浏览器,不能内嵌 WebView |
| RFC 8628 | Device Authorization Grant | TV / IoT 无浏览器设备 |
| RFC 9068 | JWT Profile for OAuth 2.0 Access Tokens | 统一 access token 格式 |
| draft-ietf-oauth-browser-based-apps | SPA 推 BFF | 浏览器 SPA 推荐转 BFF |

这是**一整套为各类客户端定制的鉴权协议族**,跟 JWT/cookie 这种"载体层"完全不在一个层级——B/C/D 都还在载体层缠斗,OAuth 2.0 在协议层一统江湖。

**Native 端 OAuth 流(RFC 8252)关键步骤:**

```
Flutter / iOS Native:
  1. 生成 code_verifier(随机 64 字节)+ code_challenge = SHA256(verifier)
  2. 调 platform API 打开系统浏览器(不是 app 内嵌 WebView):
     - iOS:  ASWebAuthenticationSession (官方 SDK)
     - Android: Chrome Custom Tab
  3. 系统浏览器加载 IdP /authorize?code_challenge=...&redirect_uri=myapp://callback
  4. 用户在浏览器内完成登录(可能 Touch ID / 已登 cookie 免登)
  5. IdP 重定向到 myapp://callback?code=<auth-code>
     系统通过 URL scheme / Universal Link 唤起 Flutter app
  6. Flutter 接到 code,POST IdP /token + code_verifier 换 access_token
  7. access_token 存系统 Keychain(iOS) / Keystore(Android)
  8. 之后调 resource server 用 Authorization: Bearer <token>
```

**关键魔法**:iOS / Android 的"系统级浏览器"与 Safari/Chrome **共享 cookie 容器**——所以用户之前若在 Web 端登过 IdP,Native 打开授权页时**自动免登**,直接拿 token。这就是为什么微信、飞书等多端鉴权"装好 app 第一次启动后不用再登"。

跨端 cookie 共享 + OAuth code grant = 多端 SSO 的工程实现。**这是 B/C 的浏览器 cookie 永远复刻不了的能力。**

按 epistemic honesty 原则,重新推理改变了答案时必须明说。修正立场如下:

**最终系统形态**:

```
        ┌──────────────────────────────────┐
        │      our-chat (含 IdP 端点)       │  ← 签发 token (RS256 + JWKS)
        └──────────────────────────────────┘
                 ▲       ▲       ▲
                 │       │       │  ← OAuth Authorization Code + PKCE
        ┌────────┴───┐ ┌─┴───┐ ┌─┴─────┐

```

简历可直接写:**"基于 OAuth 2.0 Authorization Code + PKCE 实现的跨服务 SSO,Web/Flutter 双端等价消费两个独立后端;agent-server 经 JWKS 异步拉取公钥验签,密钥轮换零业务影响。"**

## 附录 A:OAuth2 / OIDC / SAML 速查

| 协议 | 用途 | 主体 | 现状 |
|---|---|---|---|
| **OAuth 2.0** | 授权(让 A 访问 B 上的资源) | Web / Mobile API 集成 | 事实标准 |
| **OIDC**(OpenID Connect) | OAuth2 之上加身份认证 | SSO / 登录场景 | 现代标准 |
| **SAML 2.0** | 企业 SSO,基于 XML | B2B / 企业内系统 | 老但仍在用 |
| **JWT** | Token 载体格式,不是协议 | OAuth2 / OIDC 都用它 | 普及 |

简记:**SAML 是 90 年代企业方案,OAuth2 是 Web 时代标准,OIDC 是 OAuth2 + 登录补丁,JWT 只是一种 token 格式。**
