# 13 · 深度 · 下行可靠 —— 离线不丢、增量补拉、gap 检测

> 对应 [10 目标架构](10-目标架构.md) 的 **INV-3**(先持久化再推送)、**INV-4**(单调游标发现并补齐 gap)、**INV-5**(已读位点与已同步位点是两个独立游标)。

> 主题:从服务器到**接收方**这一跳——接收方**离线时不丢、上线后补齐、且不重复**。

## 一、背景与动机:为什么"推送"靠不住,必须能"补拉"

### 1.1 先看会真实发生的场景

```
A 给 B 发了 3 条消息。但此刻:
  B 的手机在地铁里没信号 / B 把 App 划掉了 / B 的 WebSocket 刚好断了一瞬
→ 服务器"推"这 3 条时,B 根本不在线,推送石沉大海
→ B 重新上线,如果系统只靠"实时推",这 3 条永远不会再来。B 永久丢消息。
```

更隐蔽的场景:
```
B 在线,但网络抖动,WebSocket 断了又重连。断的那 0.5 秒里,服务器推的第 5、6 条没送到。
重连后服务器继续推第 7 条。B 看到 …4, 7 —— 中间 5、6 静悄悄丢了,B 自己都不知道。
```

第二个场景最致命:**不是没收到,而是"丢了还不自知"**。这就是为什么需要 gap 检测。

### 1.2 本质:推送是 best-effort,可靠性不能建立在"推一定到"上

上行(12)解决的是"客户端→服务器",这里是"服务器→接收方",但底层困境同源:**网络不可靠 + 接收方可能不在场**。差别在于:

- 上行时,**发送方一直在线**,可以靠"等 ack + 重发"主动兜底。
- 下行时,**接收方可能根本不在线**,服务器没法"重发给一个不在的人"。等它回来时,服务器甚至可能已经忘了要推什么(或不该让服务器无限期持有待推队列)。

所以下行的可靠性不能依赖"服务器把消息推到"。**正确的模型是:**

> **消息一律先持久化在服务器(不丢的根),推送只是"在线时的加速通道"。接收方负责在上线/重连时,用一个游标主动把"我还没拿到的"拉回来。**

这就是 **push + pull 双模型**:push 求快(在线实时),pull 求全(离线/断连兜底)。可靠性押在 pull 上,push 只是优化。**这与 12 的"TCP 只是优化、可靠性在应用层"是同一种思维:不把正确性寄托在 best-effort 的快路径上。**

### 1.3 底层原理:这其实是"消息队列的消费位点(offset)"模型

把"接收方补拉"抽象一下,会发现它和消息队列消费**一模一样**:

```
Kafka 消费者:  记住 committed offset → 重启后从 offset 继续拉 → 不漏不重
IM 接收方:     记住 lastSyncedSeq    → 上线后拉 seq > lastSyncedSeq → 不漏不重
```

- **会话 ≈ 分区(partition)**,**seq ≈ 分区内 offset**(这个类比在 11 §2.3 已立),**接收方 ≈ 消费者**。
- 消费者用 offset 实现"断点续传";接收方用 `lastSyncedSeq` 实现"离线续传"。
- 消费者"至少消费一次 + 幂等处理"避免漏/重;接收方"按游标补拉 + 按 seq 去重"避免漏/重。

理解这个同构,意味着你可以直接借用消息队列几十年沉淀的设计(offset / 位点提交 / gap),而不是从零想。**面试讲到这,等于宣告"我知道这是个已被解决的经典问题,我用的是成熟范式"。**

> 顺带:这也是 09/10 反复说的"IM 离线补拉 和 agent-server 的 SSE 断线回放是同构问题"的根——两者都是"持久化事件 + 按序号续传"。

## 二、概念与底层原理

### 2.1 两个必须分开的游标(INV-5,最易错)

接收端要维护**两个**单调游标,初学者最常把它们合成一个,然后未读数全乱:

| 游标 | 含义 | 谁推进 | 推进时机 |
|---|---|---|---|
| **lastSyncedSeq(已同步位点)** | "我已经**收到并存到本地**到这个 seq" | 设备/客户端 | 每收到(实时推 or 补拉)一条就推进 |
| **lastReadSeq(已读位点)** | "用户**已经看过**到这个 seq" | 用户行为 | 用户真正查看会话时推进 |

为什么必须分开(讲清因果):
- **"收到" ≠ "看过"**。消息推到你手机本地了(synced),但你没点开会话(未 read)。此时**未读数 = lastSyncedSeq − lastReadSeq**。
- 如果只用一个游标:要么"收到就算已读"(未读数永远是 0,错)、要么"没看就不算收到"(那离线补拉就没有位点可用,补不回)。**两个职责强行合并必然有一个被牺牲。**
- 多端场景更明显:你手机收到了(手机 synced 推进)但没看;电脑也得收到(电脑 synced 各自推进);只有当你在**某个端看了**,read 才推进、且要同步给所有端(15 文档)。**synced 是 per-device 的,read 是 per-user 的**——更说明不能合并。

### 2.2 gap 检测:连续 seq 的杀手级用途

因为 seq 在会话内**连续**(11 保证),接收方能做一件全局 id 做不到的事:**发现自己漏了消息**。

```
本地已有最大 seq = 4。实时流推来一条 seq = 7。
→ 4 和 7 之间断了 → 一定漏了 5、6(seq 连续,不可能合法跳过)
→ 立即触发补拉 GET /sync?conv=X&since=4,把 5、6、7 拉齐
```

没有连续 seq,这一步就不存在——这是 11 §3.2 论证"全局 id 做不了 gap 检测"的下游兑现。**gap 检测是"丢了还不自知"(§1.1 场景二)的唯一解药。**

### 2.3 增量同步(incremental sync):只拉差量,不全量

接收方上线时,**不重新拉整个会话历史**(那会拉爆流量),只拉"上次同步之后的增量":

```
GET /sync?conv=X&since={lastSyncedSeq}&limit=N   →  返回 seq > lastSyncedSeq 的消息
```

`since` 是游标,`limit` 是分页。客户端拿到一页就推进 `lastSyncedSeq`,再拉下一页,直到追平。这就是"增量"——**与全量拉取相对,代价从 O(会话总量) 降到 O(离线期间新增量)**。

## 三、现状的坑(基于真实 schema)

```prisma
model UserConversation {
  lastReadMessageId String? @db.VarChar(50)   // 字符串 messageId,且只有"已读",没有"已同步"
  unreadCount       Int     @default(0)        // 直接存了个计数,没有位点支撑
}
```

三个洞:
1. **位点是字符串 messageId**,无法做 `WHERE seq > X` 范围查询(11 §3.3)→ 增量补拉无从下手。
2. **只有"已读"位点,没有"已同步"位点**(违反 INV-5)→ 离线补拉没有可用游标,且未读数无法由两个位点算出,只能靠维护一个易错的 `unreadCount` 计数器。
3. **没有 gap 检测的任何支撑**(没有连续 seq、没有 synced 游标)→ "丢了不自知"无解。

## 四、各方案分析

### 4.1 离线消息怎么存?读扩散(本项目)vs 写扩散收件箱

| 方案 | 离线消息存哪 | 补拉怎么拉 | 取舍 |
|---|---|---|---|
| **读扩散(选)** | 消息只存会话维度一份(`messages` 表) | 各接收方按**自己的 `lastSyncedSeq`** 拉 `seq > 游标` | 省存储;补拉是范围查询。**大群友好** |
| 写扩散收件箱 | 给每个接收方一份"收件箱"副本 | 拉自己收件箱里未拉的 | 写放大(N 人 × M 消息);但每人位点独立、超大群读快 |

本项目选**读扩散**(也是 our-chat 现状方向),补拉 = "在共享的会话消息流上,按各自游标做范围查询"。写扩散作为对比方案在 [14 群扩散](14-深度-群消息扩散-写扩散vs读扩散.md) 详述何时该切。**这里只需知道:读扩散下,"离线不丢"靠的是"消息持久化在会话里 + 接收方游标",不需要给离线的人单独存副本。**

### 4.2 同步的触发点(三个)

补拉不只在"上线"时触发,有三个时机:

| 触发点 | 场景 | 动作 |
|---|---|---|
| **冷启动/上线** | App 启动、重新登录 | 对每个会话 `GET /sync?since=lastSyncedSeq` 追平 |
| **重连后** | WebSocket 断线重连 | 同上,补断连期间漏的(§1.1 场景二) |
| **gap 触发** | 实时流 seq 跳变 | 立即对该会话补拉(§2.2) |

三者用**同一个** `/sync` 接口,只是触发条件不同。统一接口降低复杂度。

### 4.3 分页与背压

离线很久(几千条未读)时,补拉必须**分页**,不能一次性全查:
- `limit` 控制单页;客户端拉一页→处理→推进游标→拉下一页,直到 `返回条数 < limit`(追平)。
- 服务器侧 `WHERE conversation_id=X AND seq > since ORDER BY seq ASC LIMIT N`,走 `(conversation_id, seq)` 索引,**Index Range Scan**,高效。
- 不分页的代价:一个离线半年的用户上线,一条 SQL 拉十万条,OOM + 阻塞(对应 CLAUDE.md"循环/批量"红线的同类问题)。

### 4.4 已同步位点何时推进、谁来确认

- **实时推送路径**:接收方收到 push 并**存入本地后**,推进本地 `lastSyncedSeq`,并向服务器上报(类似下行 ack)。可批量上报(每 N 条或每 T 秒)降低频次。
- **补拉路径**:每拉一页成功落地,推进游标。
- 服务器持久化 `lastSyncedSeq` 用于:计算未读、决定"这个设备还差哪些"。
- **不强求逐条 ack 下行**(写放大),关键是**游标单调推进 + 断点能续**即可——丢的下次补拉自然补回(幂等:按 seq 去重,重复拉到的丢弃)。

### 4.5 补拉的幂等:为什么重复拉不会重复显示

补拉天然可能和实时推**重叠**(gap 触发补拉时,实时流可能也推了同一条)。去重靠 **seq 幂等**:
- 客户端按 seq 维护本地消息,**收到已存在 seq 直接丢弃**(覆盖或忽略)。
- 因为 seq 在会话内唯一且连续,"同 seq = 同一条",去重判断零歧义。
- 这就是 §1.2"push + pull 双模型"能并存的前提:**两条路可能送来同一条,但 seq 去重保证最终恰好一条**(又一次 exactly-once 效果,呼应 12 §1.3)。

## 五、核心机制

### 5.1 `/sync` 范围查询怎么走索引

```sql
-- 拉某会话游标之后的增量,按 seq 升序分页
SELECT id, seq, sender_id, content, type, timestamp
FROM messages
WHERE conversation_id = :conv AND seq > :since
ORDER BY seq ASC
LIMIT :limit;
```

- 命中 11 建的索引 `idx_messages_conv_seq (conversation_id, seq DESC)`(范围查询升序也能用该索引反向扫描,或建对应升序索引)。
- 这是 **Index Range Scan**:定位到 `(conv, since)` 后顺序扫 N 条即停,**不扫全表、不排序额外开销**。这正是"用单调数值游标"相对"字符串 messageId"的性能红利(11 §3.3)。

### 5.2 gap 检测算法(客户端)

```ts
function onRealtimePush(msg: { conversationId: string; seq: bigint }) {
  const localMax = store.maxSeq(msg.conversationId);   // 本地该会话已有最大 seq
  if (msg.seq > localMax + 1n) {
    // gap!中间漏了 (localMax, msg.seq) → 补拉
    syncSince(msg.conversationId, localMax);            // GET /sync?since=localMax
  }
  store.upsert(msg);                                    // 按 seq 幂等写入(§4.5)
  advanceSyncedSeq(msg.conversationId);                // 推进 lastSyncedSeq
}
```

- `msg.seq == localMax + 1`:连续,正常追加。
- `msg.seq > localMax + 1`:gap,补拉。
- `msg.seq <= localMax`:重复(补拉与实时重叠),幂等丢弃(§4.5)。

### 5.3 未读数 = 两个游标之差

```
unreadCount(conv) = lastSyncedSeq(conv, thisDevice/user) − lastReadSeq(conv, user)
```

- **不再维护易错的 `unreadCount` 计数器**(现状那个),而是**由两个单调游标实时算出**。计数器会在并发/多端下漂移,**位点之差天然自洽**(INV-5 的实用价值在此)。

## 六、关键实现

### 6.1 Schema(已在 11 §6.1 加好)

```prisma
model UserConversation {
  lastSyncedSeq BigInt @default(0) @map("last_synced_seq")  // 已同步(per user-conv;多端见 15)
  lastReadSeq   BigInt @default(0) @map("last_read_seq")    // 已读
}
```

### 6.2 服务器:`/sync` 接口

```ts
// GET /sync?conv=X&since=seq&limit=50
async function syncMessages(req) {
  const { conv, since, limit = 50 } = parse(req);            // ① 外部输入 runtime 校验(CLAUDE.md)
  assertMember(req.userId, conv);                            // 必须是会话成员,防越权拉别人会话
  const rows = await prisma.message.findMany({
    where: { conversationId: conv, seq: { gt: BigInt(since) }, isDeleted: false },
    orderBy: { seq: 'asc' },
    take: Math.min(limit, 200),                              // 上限,防一次拉爆(§4.3)
  });
  return { messages: rows, hasMore: rows.length === limit };  // 客户端据此决定是否拉下一页
}
```

### 6.3 客户端:上线/重连追平

```ts
async function catchUp(conv: string) {
  let since = store.lastSyncedSeq(conv);
  while (true) {
    const { messages, hasMore } = await http.get(`/sync?conv=${conv}&since=${since}&limit=50`);
    for (const m of messages) store.upsert(m);               // 按 seq 幂等(§4.5)
    if (messages.length) since = messages[messages.length - 1].seq;
    store.setLastSyncedSeq(conv, since);                     // 推进游标(断点续传:中途断了下次接着拉)
    if (!hasMore) break;                                     // 追平
  }
}
onAppStart(() => activeConversations().forEach(catchUp));
onSocketReconnect(() => activeConversations().forEach(catchUp));
```

## 七、踩坑案例(真实可复现)

### 坑 1:已读位点和已同步位点合成一个 → 未读数错乱
**复现**:只存一个 `lastReadMessageId`,收到消息就推进它当"已读"→ 用户没点开会话,未读数却一直是 0。**根因**:违反 INV-5。**修复**:拆 synced/read 两个游标,未读 = 差值(§2.1、§5.3)。

### 坑 2:断连期间漏的消息"丢了不自知"
**复现**:WebSocket 闪断,期间的消息没补,重连后只继续推新的,中间永久缺失。**根因**:没有 gap 检测、重连不补拉。**修复**:重连触发 catchUp + 实时流 gap 检测(§4.2、§5.2)。

### 坑 3:补拉不分页 → 离线久的用户上线打爆 DB
**复现**:离线半年用户上线,`/sync` 一次返回十万条,内存炸、连接超时。**根因**:无 limit。**修复**:分页 + 服务端 take 上限(§4.3、§6.2)。

### 坑 4:补拉与实时推重叠 → 重复消息
**复现**:gap 触发补拉的同时,实时流也推了同一条 → 列表里出现两条一样的。**根因**:没按 seq 幂等。**修复**:upsert by seq,已存在丢弃(§4.5)。

### 坑 5:`/sync` 不校验会话成员 → 越权拉取
**复现**:`/sync?conv=别人的会话id`,服务器直接返回了别人的聊天记录。**根因**:缺成员校验(外部输入未做权限校验,CLAUDE.md 边界红线)。**修复**:`assertMember`(§6.2)。

### 坑 6:游标用字符串 messageId 续传
**复现**:试图 `WHERE id > 'last_message_id_string'` 续传,字符串比较语义错误、还无法范围扫描。**根因**:沿用现状字符串位点。**修复**:换数值 seq 游标(§三、11 §3.3)。

## 八、Web vs Native 视角

| 维度 | Web(React) | Flutter | 原生 Swift |
|---|---|---|---|
| 本地消息存储 | IndexedDB 按 seq 索引;配额有限,超大历史要裁剪 | sqflite/Drift,容量大 | Core Data/SQLite,容量大 |
| 重连补拉触发 | `visibilitychange`/`online` 事件 + socket reconnect | 应用前台恢复 + reconnect | `applicationDidBecomeActive` + reconnect |
| 后台收消息 | 标签页后台基本收不到(无长驻进程) | 推送通道 + 前台补拉 | APNs 唤醒 + 前台 catchUp |
| 游标精度 | seq 是 BIGINT,**Web 必须 string/BigInt 比较**(11 §八) | Dart int 64 位 OK | Int64 OK |

> **Web 的结构性短板**:浏览器**没有常驻后台进程**,标签页切走/关闭就收不到实时推。所以 Web 端比原生更**依赖"可见性恢复时 catchUp"**——它的"实时"其实是"每次回到前台快速补拉到最新"。原生有 APNs/后台唤醒,离线补拉压力小些。这是"同一套补拉协议,Web 更吃 pull、原生更能 push"的本质差异。
