# 15 · 深度 · 多端一致性

> 对应 [10 目标架构](10-目标架构.md):多端一致依赖 [11 seq](11-深度-消息时序与会话内单调seq.md)、[13 离线补拉](13-深度-下行可靠-离线增量补拉.md) 的两个游标(synced per-device、read per-user),并需要 [16 连接层](16-深度-连接层与可扩展性.md) 的连接注册表与扇出。
> 主题:同一个用户**同时登录 web + 手机 + 平板**时,消息怎么都到、已读怎么跨端同步。

---

## 一、背景与动机:为什么"一个用户多设备"会出问题

### 1.1 先看会真实发生的别扭

```
你手机和电脑同时登着微信。
  场景①:别人发来消息,只有手机响了,电脑半天不刷新 → "消息没同步"
  场景②:你在手机上把一个 99+ 的群点开看完了,回到电脑,那个群还顶着 99+ 红点
          → "我明明读过了,另一个端还显示未读" —— 最招人烦的体验之一
  场景③:你在电脑上发了条消息,手机的那个会话里却没有这条 → "我自己发的都不同步"
```

这三个场景的共同点:**系统把"同一个人的多个设备"当成了互相独立的东西**,没有让它们收敛到一致状态。

### 1.2 本质:一个逻辑用户 = 多个物理连接 + 多份本地状态

多端一致的难点在于一个**身份与连接的错位**:

```
逻辑上:      "用户 B"        (一个人,一份"应该看到的真相")
物理上:  连接1(web) 连接2(手机) 连接3(平板)   (三条独立的长连接,三份本地状态)
```

- 消息的目标是**"用户 B"**(逻辑实体),但投递必须落到**B 的每一条物理连接**。系统得知道"B 现在有哪几条连接、分别在哪"。
- 每个设备有**自己的本地消息副本和位点**(synced 是 per-device 的,13 §2.1),但"已读"是**用户级**的意图(read 是 per-user 的)——你在哪个端读了,都代表"这个人读过了"。
- 所以多端一致 = **两件事**:① **下行扇出到一个用户的所有在线设备**(消息都到);② **用户级状态(尤其已读)在设备间收敛**(红点一起清)。

### 1.3 底层原理:这是"单写多读 + 状态收敛"的分布式同步

- **消息内容**:服务器是唯一权威源(single source of truth),每个设备是它的**只读副本**。副本靠 seq 游标各自追平(13)——这部分天然一致,只要每个设备都能补拉。
- **已读位点**:是用户产生的状态,**任一设备都能写**(在任一端读都算已读)。多个写入端 + 要收敛 → 用**单调推进**化解冲突:`lastReadSeq` 只增不减,`new = max(old, reported)`。**单调性让"并发已读"无需加锁也能收敛**(谁报的位点大,以谁为准),这是把分布式冲突消解掉的关键技巧(CRDT 里 max 寄存器的思想)。
- 所以多端一致不需要复杂的分布式共识,**靠"内容只读副本 + 位点单调 max"两招就够**——能讲清这点是深度。

## 二、概念与底层原理

### 2.1 连接注册表(presence registry)

要"扇给用户的所有设备",先得有个地方记着**"每个用户当前有哪些活跃连接、在哪台服务器副本上"**。这就是连接注册表(presence):

```
user:{B}:devices → {
  deviceA(web)   : { 副本=server-1, socketId=... , lastHeartbeat=... },
  deviceB(手机)  : { 副本=server-2, socketId=... , lastHeartbeat=... },
}
```

- 下行扇出时:查 `user:{B}:devices` → 得到 B 的所有在线连接 → 逐个投递(跨副本的走 pub/sub,16)。
- 它必须是**所有副本共享**的(放 Redis),否则副本 A 不知道 B 还有一条连接在副本 C 上(§四)。

### 2.2 扇出粒度:用户级,不是连接级

- 投递的目标是**用户**,落地是**该用户的每条连接**。
- 发一条消息给会话成员 B,实际动作是:`presence(B).devices.forEach(投递)`。**一个用户三个设备,就投三次(到三条连接)**。
- 这区别于单设备 IM 的"一个用户一条连接"假设——多端把"用户→连接"从 1:1 变成 1:N,所有下行路径都要按这个改。

### 2.3 已读跨端同步:推给"自己的其它端",不是会话对方(最易错)

这是多端最容易写错的一处方向:

```
B 在手机上读了和 A 的会话(read 推进到 seq=100)
  正确:把"B 读到 100"这件事 → 推给 B 的【电脑、平板】(让它们清红点)
        (是否也告诉 A "B 读了" 取决于已读回执功能,那是另一回事)
  错误:把它推给【会话对方 A】当成普通消息,或推给 B 自己刚操作的那个端(回声)
```

- **已读同步的接收者是"同一用户的其它设备"**,目的是让红点/未读数在你所有设备上一致。
- 不要推给**操作端自己**(它已经知道了,回声多余),也不要和"已读回执(告诉对方我读了)"混为一谈——后者是单聊已读功能(14 §4.4),接收者是对方,语义不同。

### 2.4 两个游标的 per-device / per-user 归属(承接 13)

| 状态 | 归属 | 多端行为 |
|---|---|---|
| `lastSyncedSeq` | **per-device**(每个设备各存各的) | 每个设备独立追平;设备 A 同步到 100 不代表设备 B 也到 100 |
| `lastReadSeq` | **per-user**(用户级,所有设备共享一个真相) | 任一端读 → 推进 → 同步给所有端 |

> 这张归属表是多端一致的"地基中的地基"。synced 若误设成 per-user,会导致"手机收到了,电脑就不补了" → 电脑永久缺消息;read 若误设成 per-device,红点永远清不干净。**13 拆两个游标,15 进一步明确它们的设备/用户归属。**

## 三、现状的坑(基于真实 schema 与架构)

- **连接态大概率在单实例 Socket.io 内存**:`io.to(socketId)` 只在本副本有效;多副本下找不到别的副本上的连接(16 详述)。即使单副本,**也没有"按用户聚合多设备"的注册表**。
- **`UserConversation` 是 per-(user,conversation)**,没有 per-device 维度 → `lastSyncedSeq` 现在只能 per-user,**多端各自的同步进度无处存**(§五要补 per-device 同步态)。
- **已读跨端通知通道缺失**:`lastReadMessageId` 改了之后,没有"通知该用户其它端"的机制 → 场景②(红点不清)无解。

## 四、各方案分析

### 4.1 连接注册表放哪:内存 vs Redis

| 方案 | 机制 | 失败模式 | 边界 |
|---|---|---|---|
| 单实例内存(现状) | `Map<userId, Set<socket>>` 在进程里 | 多副本下跨副本不可见;进程重启全丢 | 仅单副本 demo |
| **Redis 共享注册表(选)** | `user:{id}:devices` 存 Redis,所有副本读写 | Redis 单点(需高可用);要处理脏连接清理 | 多副本必需 |

**选 Redis**:多端一致和多副本(16)本就要一起做——只要是多副本,注册表就**必须**共享,否则"连在副本 A 的设备"收不到"从副本 B 发起的消息"。内存方案在多副本下根本不成立。

### 4.2 已读冲突怎么消解:单调 max

- 多端可能**并发上报**已读位点(手机和平板几乎同时读)。
- 解法:`lastReadSeq = max(当前值, 上报值)`,**只增不减**。无需加锁、无需顺序保证——max 满足交换律结合律幂等,**任意顺序到达结果一致**(CRDT 思想)。
- 选错(直接覆盖 `=` 而非 `max`)的代价:乱序到达时,旧的小位点覆盖了新的大位点 → 已读"倒退" → 红点诈尸。

### 4.3 设备下线怎么从注册表摘除:心跳超时

- 设备不会总是优雅断开(息屏、杀进程、断网),**不能只靠 disconnect 事件清理**。
- **心跳 + TTL**:每个设备定期心跳,刷新 Redis key 的 TTL;超时未心跳 → key 自动过期 → 视为下线、摘除。
- 选错(只靠 disconnect)的代价:**幽灵连接**堆积,扇出时往死连接投递,浪费 + 误判在线(影响 14 的"只推在线成员")。

### 4.4 多端发消息的回显:自己发的也要同步到自己其它端

- B 在电脑发消息,**手机端也要出现这条**(场景③)。
- 机制:发送方落库后,扇出不仅给会话**对方**,也给**发送者自己的其它设备**(`presence(senderId).devices` 除当前连接外)。
- 这和 §2.3 已读同步同源:**"自己的其它端"是一类一等公民的投递目标**,贯穿消息回显、已读同步。

## 五、核心机制

### 5.1 Redis 连接注册表结构

```
# 用户 → 设备集合(Hash 或 Set + 每设备一个带 TTL 的 key)
HSET user:{userId}:devices {deviceId} "{副本id}:{socketId}"
EXPIRE 配合心跳刷新(或每设备单独 key: presence:{userId}:{deviceId} 带 TTL)

# 扇出时:
HGETALL user:{userId}:devices → 得到所有 (副本, socketId)
  本副本的连接:直接 io.to(socketId)
  其它副本的连接:发 pub/sub 让对应副本投递(16)
```

### 5.2 下行扇出到用户所有设备

```ts
async function deliverToUser(userId: bigint, payload: any) {
  const devices = await presence.getDevices(userId);            // Redis 注册表
  for (const d of devices) {                                    // 用户级 → 逐设备(§2.2)
    if (d.replica === SELF) localEmit(d.socketId, payload);     // 本副本直接发
    else pubsub.publish(`replica:${d.replica}`, { socketId: d.socketId, payload }); // 跨副本(16)
  }
}
```

### 5.3 已读跨端同步流程

```ts
// 设备上报已读
async function onReadReport(userId: bigint, conv: string, uptoSeq: bigint, fromDeviceId: string) {
  // ① 单调推进(§4.2),并发安全
  await prisma.userConversation.updateMany({
    where: { userId, conversationId: conv, lastReadSeq: { lt: uptoSeq } },
    data:  { lastReadSeq: uptoSeq },
  });
  // ② 通知该用户【其它】设备清未读(§2.3:不含操作端,不发对方)
  const devices = (await presence.getDevices(userId)).filter(d => d.deviceId !== fromDeviceId);
  for (const d of devices) deliverToDevice(d, { type: 'read.sync', conversationId: conv, uptoSeq });
}
```

### 5.4 心跳与摘除

```ts
socket.on('heartbeat', () => presence.refresh(userId, deviceId, TTL));   // 刷新 TTL(§4.3)
socket.on('disconnect', () => presence.remove(userId, deviceId));        // 优雅断开即时摘
// 非优雅断开:TTL 到期自动摘(Redis key 过期)
```

## 六、关键实现(数据模型补充)

读扩散下 `lastSyncedSeq` 需要 **per-device**(13 是 per user-conv,这里细化):

```prisma
// 方案:per-device 同步态单独一张表(read 仍在 UserConversation 保持 per-user)
model DeviceSyncState {
  userId         BigInt
  deviceId       String   @db.VarChar(64)
  conversationId String   @db.VarChar(100)
  lastSyncedSeq  BigInt   @default(0)
  lastHeartbeat  DateTime @updatedAt
  @@unique([userId, deviceId, conversationId])
  @@map("device_sync_state")
}
// UserConversation.lastReadSeq 保持 per-user(§2.4),作为"用户已读"的唯一真相
```

> 取舍:per-device 同步态会增加状态行数(用户×设备×会话)。若设备数可控,可接受;极端可只对"活跃设备×活跃会话"维护,陈旧的随设备登出清理。**这本身是个 per-device 状态成本 vs 一致性精度的小权衡,面试可带。**

## 七、踩坑案例(真实可复现)

### 坑 1:已读同步推给了会话对方 / 操作端自己
**复现**:B 读了,代码把 `read.sync` 当普通消息广播给会话所有成员 → A 收到一条莫名其妙的"已读"消息;或推回 B 操作的那个端产生回声。**根因**:搞错接收者。**修复**:只发给"B 的其它设备"(§2.3、§5.3)。

### 坑 2:已读位点用覆盖而非 max → 红点诈尸
**复现**:手机报 read=100,平板稍后报一个乱序到达的旧值 read=80,直接覆盖 → 已读倒退,清掉的红点又回来。**根因**:`=` 而非 `max`。**修复**:`WHERE lastReadSeq < upto` 条件更新(§4.2、§5.3)。

### 坑 3:synced 误设 per-user → 某个设备永久缺消息
**复现**:把 `lastSyncedSeq` 做成 per-user,手机同步到 100 推进了用户级游标,电脑上线一看游标已是 100 就不补了,但电脑本地只有到 60 → 电脑永久缺 61~100。**根因**:synced 应 per-device(§2.4)。**修复**:per-device 同步态(§六)。

### 坑 4:只靠 disconnect 清理 → 幽灵连接
**复现**:用户息屏/断网没触发 disconnect,注册表里留着死连接,扇出一直往它投递、还被算作"在线成员"。**根因**:无心跳 TTL。**修复**:心跳刷新 + TTL 过期摘除(§4.3、§5.4)。

### 坑 5:自己发的消息不同步到自己其它端
**复现**:电脑发消息,只扇给了会话对方,没扇给自己手机 → 手机那个会话缺自己发的消息。**根因**:扇出漏了"发送者的其它设备"。**修复**:扇出目标包含 `presence(sender).devices` 除当前连接(§4.4)。

## 八、Web vs Native 视角

| 维度 | Web | Flutter | 原生 Swift |
|---|---|---|---|
| deviceId 来源 | 无硬件 id,需生成并存 localStorage(清缓存会变) | 设备标识插件 | `identifierForVendor` 稳定 |
| 同账号多标签页 | **每个标签页是一条独立连接**,相当于"一个设备多连接",注册表要按连接而非仅设备 | 单 App 通常单连接 | 单 App 单连接 |
| 心跳可靠性 | 后台标签页定时器被节流 → 心跳漏发可能误判离线;回前台要立刻补心跳 | 后台挂起,恢复补 | 后台挂起,恢复补 |
| 已读触发 | 标签页 `visibilitychange` + 可见性 | 前台可见回调 | `didBecomeActive` |

> **Web 的特殊性**:同一用户开多个标签页 = 多条连接,本质是"设备内再分裂"。注册表粒度建议到**连接(socketId)**而非仅 deviceId,扇出按连接投递。原生通常一个 App 一条连接,简单些。这是"多端"在 Web 上的额外一层多。
