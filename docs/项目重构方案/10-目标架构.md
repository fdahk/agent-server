# 10 · 目标架构 —— IM 可靠性核心 × AI 横向能力层

> 本文是 [09 校招主线项目规划](09-校招主线项目规划-IM深度与AI助手融合.md) 的架构落地版,目标是给出一份**可被确认、可照着开工**的目标架构:
> 系统全景 → 组件职责与边界 → 消息可靠性协议总览 → AI 横向层解耦边界 → 数据模型增量 → 部署拓扑与扩容 → 选型锁定 → 架构不变量与风险。
> 每个深度点(seq / ack / 离线同步 / 扩散 / 多端 / 连接层)的**字节级细节**在 11–16 各自的深度文档里展开,本文只锁定它们之间的**契约与边界**。

---

## 〇、这份架构要回答的问题

09 已经把方向定死:**不再起新项目,把 our-chat 从"功能多的 IM"打磨成"消息可靠性有硬核 + 鉴权有 OAuth2 + 多端齐 + 带 AI"的深度作品。** 那么架构层面必须先回答清楚四件事,否则开工就会返工:

1. **消息可靠性核心放在哪一层、用什么协议**——这是项目命脉,边界画错后面全乱。
2. **AI 怎么挂上来才能"加多少功能都不动消息核心"**——09 §五的"横向扩展"承诺,要在架构上兑现成一条物理边界,不是口号。
3. **现有 Prisma schema 要动哪几张表**——seq / 幂等键 / 同步游标这些是 schema 级改动,必须提前锁定迁移范围。
4. **从单实例到多副本、再到可选 Go 网关,边界怎么切才能平滑加深**——P1(纯 Node)能独立成立,P2/P5 是叠加而非重构。

本文给出的架构必须让这四个问题各有**唯一确定答案**,这样 09 的"计划文档和架构文档确认好了吗"才能回答"是"。

**一句话读懂这张图**:
- **左侧主干**是 IM,所有"消息不丢不重不乱"的逻辑收敛在 Node 业务后端里的"消息可靠性核心"这一个盒子——它是项目的深度证据。
- **右侧支线**是 AI,走**另一条传输通道(SSE)、另一套后端(agent-server)、另一个存储(Qdrant)**,只通过"共享 JWT + 用户隔离"这条细边界跟 IM 相连。这就是 09 §五说的"加多少 AI 都不动消息核心"在物理拓扑上的体现:AI 流量**根本不流经**消息可靠性核心。
- **Go 网关是可选叠加层**,虚线框,P1 阶段不存在(Node 的 Socket.io 直接收长连接),P5 才把"接入"职责从 Node 里抽出来。

## 二、组件职责与边界(契约表)

把每个组件的**职责、不该管的事、对外契约**钉死。边界清晰是"加 AI 不动核心""抽 Go 网关不动业务"两个承诺能成立的前提。

| 组件 | 职责(必须做) | 边界(明确不做) | 对外契约 |
|---|---|---|---|
| **客户端(三端)** | 维护本地 outbox(发送中消息)、生成 clientMsgId、持有 lastSyncedSeq、收 ack 后落地、gap 检测触发补拉 | 不信任本地时钟做排序、不自行决定消息最终顺序 | 上行带 `clientMsgId`;补拉带 `lastSyncedSeq`;按服务器 `seq` 排序 |
| **Go 接入网关(P5 可选)** | 长连接生命周期、心跳/重连、JWT 校验、连接注册表读写、下行扇出、限流背压 | **无业务逻辑**、不碰 DB、不分配 seq、无状态(连接态全在 Redis) | 校验 our-chat 签发的 JWT;上行透传给业务后端;下行从 pub/sub 收消息扇给本地连接 |
| **消息可靠性核心(Node 内)** | seq 原子分配、`(conv,sender,clientMsgId)` 幂等去重、ack 生成、离线消息持久化、扩散决策、已读位点更新与跨端通知 | 不管好友/群成员关系的业务规则(只读取)、不管 AI | 入参:一条待发消息;出参:`{seq, serverMsgId}` + 向接收方扇出 |
| **业务域(Node 内)** | 好友/群/会话/文件/WebRTC 信令、OAuth2.1 IdP | 不直接写 Message 表的 seq 字段(必须经可靠性核心) | 提供"会话成员列表""权限校验"给可靠性核心 |
| **PostgreSQL** | 消息持久化(seq 权威源)、社交关系、OAuth | 不做实时扇出 | `conversations.next_seq` 是 seq 的最终权威;唯一约束保证幂等 |
| **Redis** | presence(连接注册表)、pub/sub 副本间扇出、(可选)seq 预分配 | 不是消息持久化源(pub/sub 尽力而为,会丢) | `user:{id}:devices` 注册表;`conv:{id}` 频道扇出 |
| **AI 后端(agent-server)** | RAG/Agent、SSE 流式、事件溯源回放 | **不碰 IM 的 Message 表、不参与 seq/ack/扩散** | 验 our-chat 的 JWT;按 `sub` findOrCreate;检索强制按 user_id 过滤 |
| **Qdrant** | 向量检索 | 与 IM 持久化完全隔离 | 仅 AI 后端访问 |

> **关键边界判定**:任何一个"新需求"落地前,先问它落在哪个盒子。
> - "群消息要支持万人大群" → 落在**消息可靠性核心**(扩散策略),是深度改动,要慎重。
> - "AI 助手能总结未读" → 落在**AI 后端**(加一个工具)+ IM 端加一个入口,**消息核心一行不动**。
> 这个判定能力本身就是面试可讲的架构素养。

## 三、消息可靠性协议总览(细节见 11–14)

这是项目深度的总纲。这里只给**协议骨架和不变量**,每条的"为什么、怎么做、坑、面试答什么"在对应深度文档展开。

### 3.1 一条消息的完整生命周期

```
发送方                    服务器(可靠性核心)              接收方
  │                            │                            │
  │─① send(clientMsgId,conv,body)─►│                        │
  │                            │ ② 幂等查重:(conv,sender,clientMsgId)
  │                            │    命中→返回首次结果(不重复入库)
  │                            │ ③ 原子分配 seq(会话内单调)
  │                            │ ④ 持久化 Message(seq, status=sent)
  │◄────⑤ ack(clientMsgId,seq,serverMsgId)──│
  │ ⑥ outbox 标记已发送          │                            │
  │                            │ ⑦ 扩散:查会话成员           │
  │                            │   在线→pub/sub 推 ──────────►│ ⑧ 实时收,按 seq 插入
  │                            │   离线→留库等补拉            │   gap?→触发补拉
  │                            │                            │─⑨ ack/已读上报─►│
```

**先持久化、再推送(④ 在 ⑦ 之前)** 是铁律:推送失败可以靠补拉兜底,但没落库的消息永远丢了。

### 3.2 五个不变量(任何改动不得破坏)

| # | 不变量 | 由谁保证 | 破坏后果 |
|---|---|---|---|
| INV-1 | 同一会话内 seq 严格单调递增、不跳号、不重号 | `conversations.next_seq` 原子自增 + 崩溃恢复策略(见 11) | 客户端排序错乱、gap 检测失效 |
| INV-2 | 同一 `(conv,sender,clientMsgId)` 至多入库一次 | DB 唯一约束 + 命中返回首次结果(见 12) | 网络重发导致消息重复 |
| INV-3 | 消息先持久化成功,才向接收方推送 | 可靠性核心调用顺序(④→⑦) | 推送成功但库里没有 → 补拉补不回 → 真丢 |
| INV-4 | 接收方能用单调游标发现并补齐 gap | `lastSyncedSeq` + `GET /sync?since=`(见 13) | 漏消息且自己不知道 |
| INV-5 | 已读位点(read)与已同步位点(synced)是两个独立游标 | 两个字段分开存(见 13、15) | "已收到"被误当"已读",未读数错乱 |

> 这五条 + §四的 AI 边界,就是本架构的**全部硬约束**。面试问"你怎么保证消息不丢不重不乱",答案就是把 INV-1~5 串起来讲。

### 3.3 协议字段锁定(跨端契约,先定下来免得各端各写一套)

| 方向 | 消息 | 关键字段 |
|---|---|---|
| 上行 | `message.send` | `clientMsgId`(UUIDv4)、`conversationId`、`body`、`extra?` |
| 下行 | `message.ack` | `clientMsgId`、`seq`、`serverMsgId`、`serverTs` |
| 下行 | `message.push` | `serverMsgId`、`seq`、`conversationId`、`senderId`、`body` |
| 上行 | `message.read` | `conversationId`、`uptoSeq`(已读到的 seq) |
| 下行 | `read.sync`(推给同用户其它端) | `conversationId`、`uptoSeq` |
| HTTP | `GET /sync?conv&since=seq&limit` | 返回 `seq > since` 的消息分页 |

## 四、AI 横向层解耦边界(为什么加 AI 不动核心)

这是 09 §五最核心的架构承诺,本节把它钉成一条**物理边界**。

### 4.1 三条隔离线

AI 与 IM 在架构上**三重物理隔离**,所以 AI 功能无论加多少,消息核心的代码路径都不被触碰:

| 隔离维度 | IM 消息核心 | AI 能力层 | 隔离意味着 |
|---|---|---|---|
| **传输通道** | WebSocket(Socket.io) | SSE(agent-server 独立端点) | AI 流量不进 IM 长连接,不抢消息通道、不受扇出逻辑影响 |
| **后端进程** | Node 业务后端 | agent-server(独立 NestJS) | AI 崩了 IM 照常收发;AI 加工具改的是 agent-server,不碰 IM 仓 |
| **存储** | PostgreSQL(Message 表) | Qdrant(向量)+ agent-server 自己的库 | AI 检索/写向量不触碰消息持久化,无双写耦合(除非显式做"聊天记录语义搜索",那条单独定顺序,见 09 §8.3) |

### 4.2 唯一的连接点:共享 JWT + 用户隔离

IM 和 AI **只在一处相连**——身份。其余全部解耦。

```
our-chat 签发 JWT(jose, 含 sub=userId)
        │ 客户端带同一个 JWT 调 AI 端点
        ▼
agent-server 用 our-chat 公钥/共享密钥验 JWT(方案C)
        │ 首次见到该 sub → findOrCreate 用户(零接触同步)
        ▼
所有 AI 检索强制 WHERE user_id = sub(多租户隔离,复用 agent-server/docs/03 §2.3)
```

- **方案C(共享 JWT)**:our-chat 是 IdP,agent-server 是纯资源服务器,不自己管用户密码。细节在 [agent-server/docs/06](06-前端接入与多端规划.md)。
- **零接触同步**:agent-server 不需要 our-chat 推送用户表,第一次拿到 JWT 就 findOrCreate。两个系统的用户库各自独立,只靠 `sub` 对齐。

### 4.3 "AI 助手"在 IM 里的呈现(产品形态)

- our-chat 联系人列表置顶一个**虚拟联系人"AI 助手"**;它在 IM 数据模型里**不是真用户、不进 Message 表的可靠性流程**。
- 点进"AI 助手"会话 → 前端切到 **SSE 流式面板**,直接连 agent-server,不走 Socket.io 消息通道。
- 因此:AI 对话的"消息"不需要 seq/ack/扩散——它们本就不是 IM 消息,是 AI 任务流。**这正是解耦的好处:AI 不必背 IM 的可靠性包袱,IM 也不必懂 AI。**

> **架构结论**:新增任意 AI 能力 = 在 agent-server 加一个工具/接口 + IM 端加一个入口。INV-1~5 一条都不涉及。这就是"横向扩展"的精确含义。

## 五、数据模型增量(基于现有 Prisma schema)

> 现状权威:`our-chat/server/prisma/schema.prisma`。现有 `Message.timestamp DateTime @default(now())` + 索引 `(conversation_id, timestamp DESC)` 是**深度洞**(时钟不单调、同毫秒乱序)。本节锁定要动的字段,**字节级迁移脚本在 11 深度文档**里写。

### 5.1 改动清单(只列增量,不重画全表)

| 表 | 改动 | 用途 | 对应不变量 |
|---|---|---|---|
| `conversations` | **新增** `next_seq BIGINT DEFAULT 0` | 会话内 seq 的权威自增源 | INV-1 |
| `messages` | **新增** `seq BIGINT`;新增唯一约束 `(conversation_id, sender_id, client_msg_id)`;新增 `client_msg_id` | 排序基准 + 幂等去重 | INV-1, INV-2 |
| `messages` | **改索引**:`(conversation_id, seq DESC)` 取代 `(conversation_id, timestamp DESC)`(timestamp 保留作展示,不再用于排序) | 排序与范围查询走 seq | INV-1 |
| `user_conversations` | **新增** `last_synced_seq BIGINT DEFAULT 0`;保留/改造 `last_read_message_id` → 增加 `last_read_seq BIGINT` | 同步游标与已读游标分离 | INV-4, INV-5 |

### 5.2 迁移与回填要点(细节在 11)

- **回填 seq**:对历史消息,按 `(conversation_id, timestamp ASC, id ASC)` 排序回填单调 seq,并把每个会话的 `next_seq` 置为该会话最大 seq + 1。
- **唯一约束的历史数据冲突**:老消息没有 `client_msg_id`,回填时用 `serverMsgId` 兜底填充,避免唯一约束建不上。
- **灰度**:先双写(seq 写入但排序仍读 timestamp)→ 校验 seq 单调无空洞 → 切排序到 seq。**这套灰度本身是 P0 基线脚本的验证对象**(见 09 §P0)。

## 六、部署拓扑与扩容路径

架构的可扩展性叙事分三档,**每一档都能独立演示 + 对应一个面试深挖点**,且后一档是前一档的叠加而非重写。

### 6.1 档位一:单实例(P1 完成态)

```
[1× Node(Socket.io+业务+可靠性核心)] ── [PG] ── [Redis(presence)]
```
- 连接态可暂存单实例内存或 Redis;pub/sub 此时非必需(单实例直接内存扇出)。
- **足以证明 INV-1~5 全套可靠性故事**。09 明确:到 P1 项目就能投,不要等后面。

### 6.2 档位二:多副本(P2 完成态)

```
        ┌─ [Node 副本 A] ─┐
client ─┤                 ├─ [PG]
        └─ [Node 副本 B] ─┘
              │   │
        [Redis: presence 注册表 + pub/sub backplane]
```
- 连接注册表**必须**入 Redis(`user:{id}:devices → 副本`),否则跨副本找不到连接。
- 下行扇出走 **Redis pub/sub**:消息落库后,发布到 `conv:{id}` 频道,所有副本订阅、扇给本地连接。**复用 agent-server/docs/02 §6.1 的 backplane**(事件落库 + pub/sub 实时 + 断线按 seq 回放——IM 与 SSE 同构)。
- 面试点:"连在 A 副本的用户,消息从 B 副本怎么送达"。

### 6.3 档位三:Go 接入网关(P5 可选加深)

```
client ── [Go 网关 ×N(无状态)] ── 内部RPC ── [Node 业务 ×M]
                  │                                  │
            [Redis: presence + pub/sub] ────────────┘
```
- 把"接入"职责(长连接/心跳/JWT 校验/扇出/限流背压)从 Node 抽成独立 Go 服务,**业务仍在 Node**。
- 触发条件:长连接数成为真瓶颈时。goroutine + channel 管理海量连接是 Go 主场,正好接 Node→Go 成长线。
- **是加分不是地基**:P5 砍掉,档位二的故事独立成立(见 09 §8.1)。

> 扩容路径的关键设计:**连接态从一开始就规划放 Redis**(档位二就入),所以档位三抽 Go 网关时,网关天然无状态,不需要重构状态管理。这是"先把边界画对,后面平滑加深"的体现。

## 七、选型锁定(为什么是这些,选错的代价)

| 层 | 锁定选型 | 为什么是它 | 选错的代价 |
|---|---|---|---|
| IM 业务 + 可靠性核心 | **Node(Express + Socket.io,沿用现有)** | 可靠性核心是纯协议 + DB 设计问题,不需要换语言就能做出深度;复用现有全栈资产 | 换语言重写 = 浪费已沉没的 OAuth/文件/WebRTC 硬资产,负 ROI |
| 消息持久化 + seq 权威 | **PostgreSQL(沿用)** | 原子自增(`UPDATE...RETURNING`)+ 唯一约束 + 复合索引一把梭;事务保证 INV-1/INV-2 | 用最终一致的 NoSQL 做 seq 权威 = 单调性难保证,自找麻烦 |
| presence + 扇出 + 可选 seq 预分配 | **Redis** | pub/sub backplane 现成、注册表读写快;与 agent-server 同款 | 用 DB 轮询做扇出 = 延迟高、压垮 DB |
| 接入网关(可选) | **Go** | 高并发长连接的真瓶颈场景,goroutine + channel 契合;接成长线 | 用 Node 抽网关 = 没拉开与现状的差距,白抽 |
| AI 后端 | **agent-server(NestJS,沿用)** | 已是现成 AI 后端,事件溯源 + SSE 回放可复用 | 在 IM 仓里塞 AI = 破坏 §四的进程隔离,AI 故障波及 IM |
| 向量 | **Qdrant(沿用)** | agent-server 已用 | — |

**锁定原则**:除了"可选 Go 网关"是新增语言,其余全部**沿用 our-chat / agent-server 既有栈**。本项目的深度来自**协议与边界设计**,不来自换框架。这与 memory 里"不要再收集框架,缺的是深度"一致。

## 八、架构不变量与风险登记

### 8.1 不变量汇总(改任何代码前对照)

- **INV-1~5**(§3.2):消息可靠性五铁律。
- **INV-6**:AI 流量永不进入 IM 消息核心代码路径(§四三条隔离线)。
- **INV-7**:连接态只存 Redis,接入层无状态(保证档位二→三平滑)。
- **INV-8**:seq 权威只有一个源(`conversations.next_seq`),Redis 预分配只是缓存,崩溃以 DB 为准。

### 8.2 风险登记(写进 README 当面试弹药)

| 风险 | 触发场景 | 缓解 | 深度文档 |
|---|---|---|---|
| seq 跳号/重号 | Redis 预分配后崩溃、DB 自增并发 | 以 DB `UPDATE...RETURNING` 为唯一权威;Redis 仅加速,恢复对齐 DB | 11 |
| ack 丢失致重复 | 网络抖动,客户端重发 | `(conv,sender,clientMsgId)` 唯一约束,命中返回首次结果 | 12 |
| pub/sub 漏消息 | Redis pub/sub 尽力而为 | 双保险:事件落库 + 断线按 seq 回放(同 agent-server 02 §6.1) | 14 |
| 反代 buffering 掐流 | Nginx 默认缓冲 SSE/WS | 显式关 buffering,否则流式与实时全挂 | 14 |
| 多端已读推错对象 | 已读同步推给会话对方 | 只推给"自己的其它端",不推对方 | 15 |
| 双写顺序错乱 | 消息入 PG + 入向量库做语义搜索 | 定顺序(先 PG 后向量)+ 失败补偿(同 agent-server 03 §1.3) | 16 / AI 侧 |
| 大群已读写放大 | N 人 × M 消息逐人已读 | 大群只做"已读人数"不做逐人已读 | 13(扩散) |
