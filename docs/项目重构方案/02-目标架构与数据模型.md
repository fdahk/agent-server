# 02 · 目标架构与数据模型

> 本文把 01 的选型落成具体的模块划分、架构图、关系数据模型和 Milvus collection 设计。

---

## 1. 目标架构总览

单一 NestJS 服务,内部按模块划分;外部依赖 Postgres / Redis / Milvus / Ollama。

```
                          ┌─────────────────────────────────────────────┐
                          │            Client (前端 / 仿微信 bot)         │
                          └───────────────┬─────────────────────────────┘
                                          │ HTTP + SSE
                          ┌───────────────▼─────────────────────────────┐
                          │           NestJS 单体服务 (Port 3000)         │
                          │                                              │
                          │  interfaces 层:Controllers + SSE            │
                          │  ┌────────┬──────────┬──────────┬─────────┐  │
                          │  │  auth  │ documents│   chat   │  agent  │  │
                          │  │ 模块   │  摄取     │  RAG对话  │ 工具编排 │  │
                          │  └───┬────┴────┬─────┴────┬─────┴────┬────┘  │
                          │      │         │          │          │       │
                          │  ┌───▼─────────▼──────────▼──────────▼────┐  │
                          │  │  shared:run-engine(FSM+事件溯源+SSE)    │  │ ← 复用现有资产
                          │  │           rag(检索) / llm(Ollama)       │  │
                          │  └───┬─────────┬──────────┬──────────┬────┘  │
                          │      │         │          │          │       │
                          │  ┌───▼───┐ ┌───▼────┐ ┌───▼────┐ ┌───▼────┐  │
                          │  │Prisma │ │ ioredis│ │ Milvus │ │ Ollama │  │
                          │  └───┬───┘ │+BullMQ │ │ client │ │ client │  │
                          │      │     └───┬────┘ └───┬────┘ └───┬────┘  │
                          └──────┼─────────┼──────────┼──────────┼───────┘
                                 ▼         ▼          ▼          ▼
                            ┌────────┐ ┌──────┐ ┌────────┐ ┌────────┐
                            │Postgres│ │Redis │ │ Milvus │ │ Ollama │
                            │(关系)  │ │(队列)│ │ (向量) │ │ (GPU)  │
                            └────────┘ └──────┘ └────────┘ └────────┘
```

要点:
- **关系数据在 Postgres,向量数据在 Milvus**,两者通过 `chunk` 的 `vector_id` 关联。
- **`shared/run-engine` 是从现有 Java 移植的 FSM + 事件溯源 + SSE**,被 documents(摄取进度)和 agent(任务进度)共用。
- Worker(BullMQ 消费者)与 HTTP 同进程内启动(也可独立进程,见 04)。

## 2. 模块划分(NestJS modules)

| 模块 | 职责 | 主要来源 |
|---|---|---|
| `auth` | 注册/登录/JWT/Guard,用户隔离的入口 | 移植 Java auth + **新增注册** |
| `documents` | 文件上传、摄取流水线触发、文档/分片管理 | **全新** |
| `ingestion`(worker) | 解析→切分→embedding→入 Milvus 的异步作业 | 复用资源采集逻辑 + **全新 RAG plumbing** |
| `chat` | 会话管理、RAG 检索、流式生成、引用 | **全新** |
| `agent` | 复合任务编排、工具调用(LangGraph) | 重构现有编排 + **新增工具** |
| `shared/run-engine` | FSM、事件溯源、SSE Hub | **移植自 Java(核心资产)** |
| `shared/rag` | 检索封装(embed query → Milvus search → rerank) | **全新** |
| `shared/llm` | Ollama chat / embedding 抽象 | 移植 Ollama provider + 新增 embedding |

## 3. 关系数据模型(Postgres / Prisma)

在现有 3 表基础上**新增 4 张核心表**,并**复用/泛化**运行表。下面是 Prisma schema 草案(字段可在落地时微调):

```prisma
// ============ 用户 ============
model User {
  id           BigInt   @id @default(autoincrement())
  username     String   @unique @db.VarChar(64)
  passwordHash String   @db.VarChar(255)   // bcrypt,不再用 {noop}
  displayName  String   @db.VarChar(128)
  roleCode     String   @db.VarChar(32)
  createdAt    DateTime @default(now())
  documents     Document[]
  conversations Conversation[]
  runs          Run[]
}

// ============ 文档(用户上传的资料)============
model Document {
  id          BigInt   @id @default(autoincrement())
  userId      BigInt
  filename    String   @db.VarChar(255)
  mimeType    String   @db.VarChar(128)
  sizeBytes   BigInt
  storagePath String   @db.VarChar(512)    // 落盘路径 / 对象存储 key
  // uploaded → parsing → chunking → embedding → ready → failed
  status      String   @db.VarChar(32)
  errorMsg    String?  @db.Text
  chunkCount  Int      @default(0)
  createdAt   DateTime @default(now())
  user        User     @relation(fields: [userId], references: [id])
  chunks      DocumentChunk[]
  @@index([userId, createdAt])
}

// ============ 文档分片(向量在 Milvus,这里存元数据)============
model DocumentChunk {
  id           BigInt   @id @default(autoincrement())
  documentId   BigInt
  userId       BigInt              // 冗余,便于按用户过滤/级联
  chunkIndex   Int                 // 在文档内的顺序
  content      String   @db.Text   // 原文(用于拼上下文 + 展示引用)
  tokenCount   Int
  vectorId String  @db.VarChar(64)  // 对应 Milvus 里的 point id
  createdAt    DateTime @default(now())
  document     Document @relation(fields: [documentId], references: [id])
  @@unique([documentId, chunkIndex])
  @@index([userId])
}

// ============ 会话 ============
model Conversation {
  id        BigInt   @id @default(autoincrement())
  userId    BigInt
  title     String   @db.VarChar(255)
  createdAt DateTime @default(now())
  user      User      @relation(fields: [userId], references: [id])
  messages  Message[]
  @@index([userId, createdAt])
}

// ============ 消息(多轮对话历史)============
model Message {
  id             BigInt   @id @default(autoincrement())
  conversationId BigInt
  role           String   @db.VarChar(16)   // user / assistant / system
  content        String   @db.Text
  // 该条 assistant 回答引用了哪些 chunk(JSON: [{chunkId, documentId, score}])
  citationsJson  String?  @db.Text
  createdAt      DateTime @default(now())
  conversation   Conversation @relation(fields: [conversationId], references: [id])
  @@index([conversationId, createdAt])
}

// ============ 运行(泛化:摄取作业 + agent 任务都用它)============
model Run {
  id          BigInt   @id @default(autoincrement())
  runId       String   @unique @db.VarChar(64)
  userId      BigInt
  kind        String   @db.VarChar(32)   // ingestion / agent_task
  refId       String?  @db.VarChar(64)   // 关联的 documentId / conversationId
  task        String   @db.VarChar(255)
  status      String   @db.VarChar(32)   // queued/running/completed/failed
  progressMsg String?  @db.VarChar(255)
  startedAt   DateTime?
  completedAt DateTime?
  createdAt   DateTime @default(now())
  user        User     @relation(fields: [userId], references: [id])
  events      RunEvent[]
  @@index([userId, createdAt])
}

// ============ 运行事件(事件溯源,移植自 agent_run_events)============
model RunEvent {
  id         BigInt   @id @default(autoincrement())
  runId      String   @db.VarChar(64)
  sequenceNo BigInt
  eventType  String   @db.VarChar(64)
  payloadJson String  @db.Text
  createdAt  DateTime @default(now())
  run        Run      @relation(fields: [runId], references: [runId])
  @@unique([runId, sequenceNo])
  @@index([runId])
}
```

### 与现有表的对应关系

| 现有表 | 目标 | 说明 |
|---|---|---|
| `users` | `User` + 新增 bcrypt/注册 | 去掉 `{noop}` 明文 |
| `agent_runs` | **泛化为 `Run`** | 加 `kind`(ingestion/agent_task)+ `refId`,一套引擎服务两种异步场景 |
| `agent_run_events` | `RunEvent` | 字段基本不变,直接移植 |
| —— | `Document` / `DocumentChunk` / `Conversation` / `Message` | **全新,愿景的核心数据骨架** |

> 设计要点:把 `agent_runs` 泛化为通用 `Run`,是因为"文档摄取进度"和"agent 任务进度"是同一种东西——长耗时异步作业 + 事件流 + SSE。一套引擎复用,避免重复造。

## 4. Milvus Collection 设计

```
Collection: knowledge_chunks   (consistency_level: Strong)
├── id          VarChar(64) PRIMARY KEY   # 用 UUID,与 DocumentChunk.vectorId 一一对应
├── vector      FloatVector dim=768       # 必须 == embedding 模型维度(nomic-embed-text=768)
│                 index: AUTOINDEX / metric COSINE
├── user_id     Int64   # ★ 多租户隔离的关键字段(检索强制按它过滤)
├── document_id Int64   # 按文档删除/重试清场按它过滤
├── chunk_id    Int64   # 对应 Postgres DocumentChunk.id
└── chunk_index Int64
```

> Milvus 与 Qdrant 的两点运维差异:① 检索/删除前集合必须 **load** 进内存
> (`ensureCollection` 建集合+索引后轮询到 Loaded 才返回);② 标量字段直接用表达式过滤
> (`user_id == X`),不需要像 Qdrant 那样单独建 payload 索引。

### 多租户隔离(安全底线)

**每一次检索都必须带 `user_id` 过滤**,否则会泄露其他用户的资料:

```ts
await milvus.search({
  collection_name: 'knowledge_chunks',
  data: queryEmbedding,
  limit: 6,
  filter: `user_id == ${currentUserId}`,   // ★ 多租户隔离
  output_fields: ['user_id', 'document_id', 'chunk_id', 'chunk_index'],
  consistency_level: 'Strong',
});
```

- 这个过滤条件封装在 `shared/rag` 的 `searchByUser` 里强制注入,**禁止**让上层 controller 自己拼 filter(避免某次忘了写就越权)。

### Postgres ↔ Milvus 一致性

- 写:摄取 worker 先写 Postgres `DocumentChunk` 拿 `chunk.id`,再带 `chunk_id` 把点 upsert 进 Milvus(payload 用 UUID 作主键)。**固定顺序**,失败整 job 重试。
- 重试幂等:开写前按 `document_id` 把 Milvus 与 Postgres 两边旧 chunk 清场,再重写。
- 删:删文档时,既删 Postgres 的 chunks(级联),也按 `document_id` 过滤删 Milvus points。
- 检索回显:Milvus 只返回标量字段(含 `chunk_id`),**原文从 Postgres 的 `DocumentChunk.content` 取**(向量库不适合存大文本)。

## 6. 部署拓扑与横向扩容(web 池 + worker 池)

单体不等于"只能单副本"。同一份代码库以两种**角色**启动,各自独立扩容:

```
                       ┌──────────── Load Balancer / Nginx ────────────┐
                       │              (TLS 终结 + 路由)                  │
              ┌────────▼─────────┐                  ┌──────────────────▼──────────┐
              │   Web 池 × N      │                  │      Worker 池 × M           │
              │  start:server        │                  │  start:worker                │
              │  • auth / 注册     │                  │  • 消费 BullMQ "ingest" 队列  │
              │  • 上传接口        │   Redis(队列)    │  • 解析→切分→embedding        │
              │  • chat / 检索     │◄────────────────►│  • upsert Milvus + 写库       │
              │  • SSE 推送        │   Redis(pub/sub) │  • 发运行事件                 │
              └───┬───────┬───────┘                  └────────┬──────────┬─────────┘
                  │       │          同一份代码,不同入口         │          │
                  ▼       ▼                                    ▼          ▼
              Postgres  Milvus                              Ollama    (写回)Postgres/Milvus
```

- **Web 池**:无状态(JWT 无状态 + 不在本地内存存业务态),可任意多副本。
- **Worker 池**:消费 BullMQ 队列,摄取压力大就加副本,与 web 独立伸缩。
- **角色由启动入口区分**:`start:server` 起 HTTP 监听;`start:worker` 用 `createApplicationContext()` 起无 HTTP 的消费进程,二者 `import` 同一批 module。

### 6.1 横向扩容的真实障碍:SSE 必须加 Redis backplane

现有 `SseEmitterHub` 是**内存态**——这是单体多副本下唯一真正会出错的点(拆不拆服务都一样):

> 某个 run 的 SSE 客户端连在 **web 副本 A**,但事件由 **worker** 产生 → worker 怎么把事件送到正好持有该连接的副本 A?

**解法:事件经 Redis pub/sub 扇出。**

```
worker 产生事件
   → ① 落库 RunEvent(事件溯源,保底)
   → ② PUBLISH 到 Redis 频道 run:{runId}
        ↓
   所有 web 副本都 SUBSCRIBE run:{runId}
        ↓
   持有该 SSE 连接的那个副本收到后,推给客户端;其余副本忽略
```

- 你正好已经有 Redis(BullMQ 也用它),零新增组件。
- **双保险**:① Redis pub/sub 负责实时扇出;② 事件落库 + `sequence_no` 支持**断线重连后从数据库回放**(现有设计已具备)。即使某条 pub/sub 漏了,重连也能补齐。
- 这把 `shared/run-engine` 从"单机内存推送"升级为"多副本可扩展推送",是这次重构在可扩展性上的关键一笔。

### 6.2 各依赖的扩容方式(对照)

| 组件 | 怎么扩 | 是否本次重点 |
|---|---|---|
| Web 层 | 无状态多副本 + LB | ✅ 设计成无状态即可 |
| Worker 层 | 多副本消费同一队列 | ✅ BullMQ 天然支持 |
| Ollama/推理 | GPU 层独立扩(本地单实例即可) | 正交,非本次重点 |
| Postgres | 主从读写分离(后期) | 暂不需要 |
| Milvus | 单节点足够,后期可集群 | 暂不需要 |
| Redis | 单实例足够 | 暂不需要 |

> **小结**:模块化单体 + web/worker 角色拆分 + SSE 的 Redis backplane,既保住横向扩容,又实现"轻重分离、各扩各的",且不付微服务的税。详细论证见 `01` 第 1.1 节,**具体用什么工具把它部起来(Compose vs k8s vs ...)见 `07-部署演进与扩容路径.md`**。
