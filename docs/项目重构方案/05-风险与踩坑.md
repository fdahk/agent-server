# 05 · 风险与踩坑

> 落地时真会踩的坑、缓解办法,以及校招定位的诚实提醒。

---

## 1. RAG / 向量相关坑

### 1.1 embedding 维度必须与 Qdrant collection 一致(最常见)

- Qdrant collection 创建时写死了 `size`(向量维度)。`nomic-embed-text`=768,`bge-m3`=1024。**换 embedding 模型 = 维度变 = 旧 collection 全废**,要重建并重新摄取所有文档。
- **缓解**:选定模型前先定维度;collection 名带模型标识(如 `chunks_nomic768`);把维度写进配置,启动时校验。

### 1.2 切分策略直接决定检索质量

- chunk 太大 → 检索命中但上下文噪声多、塞爆窗口;太小 → 语义被切碎、召回差。
- **缓解**:用 token 级切分(不是字符),起步 `chunkSize≈500 token, overlap≈50`,按效果调;保留 chunk 在原文的位置以便回显。

### 1.3 多租户隔离漏过滤 = 越权(安全红线)

- 任何 Qdrant 检索漏了 `user_id` 过滤,就会把别人的资料检索出来。
- **缓解**:检索只允许走 `shared/rag` 单一入口,过滤写死在里面(见 03 §2.3);给 `user_id` 建 payload index;写一个测试:A 用户绝不能检索到 B 的 chunk。

### 1.4 Postgres ↔ Qdrant 双写不一致

- 一边写成功一边失败 → 向量与元数据对不上(检索到 point 但查不到原文,或反之)。
- **缓解**:固定写入顺序 + 失败整 job 重试 + 重试前按 `document_id` 幂等清理(见 03 §1.3);删文档两边都删。

## 2. LangChain.js 工具包相关坑(只剩这些)

> 我们已锁定**只用** `@langchain/community` 的 loader 和 `@langchain/textsplitters` 这两个工具包,不用 chain/runnable/agent 框架(01 §5)。所以坑面收窄到这两个包。

### 2.1 loader/splitter 个别格式不稳

- 某些边角 PDF(扫描件、复杂版式)或 docx 解析可能出空内容/乱码。
- **缓解**:每个 loader 都包一层 try/catch,失败把文档标 `failed` 并存 `errorMsg`(用户能看到原因);极端格式上 unstructured.io 或后期上 Python parser sidecar(B 方案,文档里已留位置)。

### 2.2 版本破坏性变更

- LangChain 历史上 minor 版常有 breaking change(虽然现在我们只用 2 个工具包,影响面小)。
- **缓解**:锁版本(pin),升级前读 changelog,别盲目 `^`。

### 2.3 不要被"加一个 chain 抽象就更优雅"的诱惑反复

- 真实诱惑:实现 chat 时看到 `RunnableSequence`、`ChatPromptTemplate` 觉得"用一下挺方便",慢慢就把 chain 引进来了。
- 后果:① 越权过滤可能被遮蔽(L3 安全);② 调试要穿透框架;③ M5 的 agent 循环开始觉得"是不是也该上 LangGraph";一步步回到"框架接管一切"。
- **缓解**:`shared/rag` 和 `agent` 模块**约定不 import** `@langchain/core`、`langchain`、`@langchain/qdrant`、`@langchain/openai`、`langgraph`——把这些写进 ESLint 规则或 code review checklist。

## 3. 流式 / SSE 相关坑

### 3.1 反向代理缓冲会"卡住"流式输出

- Nginx 默认 `proxy_buffering on`,会把流式响应攒起来一次性发,逐 token 体验全无。
- **缓解**:SSE/流式路由设 `proxy_buffering off`;Nginx 还需 `proxy_http_version 1.1` + 关闭对该路径的缓存;若用 Caddy 默认不 buffer。

### 3.2 多副本下 SSE 送不到(已在 02 §6.1 设计)

- 内存态 SSE Hub 在多 web 副本下,worker 的事件可能推不到持有连接的副本。
- **缓解**:Redis pub/sub backplane + 事件落库回放(双保险)。**单副本开发时不会暴露这个坑,上多副本才炸,要提前设计。**

## 4. 模型层 / 性能相关坑

- 本地 Ollama 并发吞吐有限,批量 embedding 大文档会慢——这是接受的开发期成本,生产换第三方 API 即可缓解。
- **缓解**:embedding 批处理(`input` 接数组,一次请求多条)+ 限制 worker 并发(BullMQ concurrency)+ embedding 结果缓存(同内容不重复算);摄取本就异步,慢可接受。
- **切第三方 API 后的新坑**:① 计费按 token,大批量 embedding/长文档要估算成本上限;② 限流(qps/TPM),超了要退避重试;③ 数据出本机,涉敏感资料要看合规;④ 不同厂商对 OpenAI 协议的兼容**不是 100% 严格一致**(stream 格式、function call 形态偶有差异),切换后要跑端到端冒烟测试。
- **不要在 Mac 上跑 vLLM/TGI**(无 CUDA)。本地 = Ollama,生产 = 第三方 API 或独立 GPU 主机上的 vLLM,切换全部通过 `LLM_BASE_URL` 完成(见 01 §7)。

## 5. 工程 / 安全坑

| 坑 | 缓解 |
|---|---|
| 文件上传安全 | 限大小/类型/扩展名;文件名做路径穿越校验;存到 web 根目录之外;病毒/超大文件防护 |
| JWT 真实 IP / 越权 | Guard 全局生效,资源查询都带 `where userId`;别只信前端传的 id |
| 大 PDF 解析 OOM | 解析/切分设上限;超大文件分段;worker 设内存上限 |
| BullMQ job 堆积 | 设并发上限 + 失败重试上限 + 死信处理;监控队列长度 |
| Prisma 连接数 | 多 worker 副本共享 DB,配 connection pool 上限,别打爆 Postgres |
| 密码安全 | bcrypt(去掉 `{noop}`);注册做密码强度校验 |

## 6. 范围 / 时间风险

- **这是个大重构**:删 Java + 补整条 RAG 主线,工作量不小。
- **缓解**:严格按 04 的里程碑推进,**M0–M4 是 MVP,先把"上传→对话"打通**;M5(agent)、M6(打磨)是加分,时间紧可裁剪。
- **别同时改太多**:一次只推一个 Mx,每个都能独立演示,避免"全都开了头、全都没完成"。
- **复用优先**:run-engine、采集、Ollama provider、prompt 都是现成的,移植别重写,省下的时间投到新能力(RAG)上。

## 7. 校招定位的诚实提醒

> 这条是策略层面的,不影响技术方案,但你应该清楚。

- **RAG / agent 知识库是 2026 最卷的方向之一**:几乎人人都在做"上传文档 + 和文档对话"。光有这个 surface,在简历里区分度有限。
- **你的差异化不在"做了个 RAG 助手",而在工程深度**:
  - 事件溯源 + 断线重连的运行引擎(可审计、可回放)
  - web/worker 角色拆分 + Redis backplane 的可扩展设计(知道瓶颈在哪、何时不拆微服务)
  - 多租户向量隔离(安全意识)
  - 模型层可热插拔(LLM 工程化)
  - 从 polyglot 主动收敛为模块化单体的**架构判断力**(讲清"为什么不拆/为什么这么拆")
- **面试讲法**:别讲"我做了个 ChatGPT",讲"我设计了一个可观测、可扩展、多租户隔离的知识助手后端,并在架构上做了 A/B/C 的取舍"。后者才是大厂想听的。
- **提醒**:你此前的方向是图形渲染(高壁垒、区分度高)。切到 agent 后端意味着放弃那条差异化赛道,进入更卷的方向——这是你的选择,我尊重,但定位上的代价要心里有数(详见你之前确认的方向记录)。

## 8. 风险速查表

| 风险 | 严重度 | 何时暴露 | 关键缓解 |
|---|---|---|---|
| 多租户漏过滤越权 | 🔴 高 | 多用户测试 | 检索单一入口 + 测试 |
| embedding 维度不匹配 | 🟠 中 | 换模型时 | 配置校验 + collection 命名带维度 |
| 多副本 SSE 送不到 | 🟠 中 | 上多副本 | Redis backplane(提前做) |
| 双写不一致 | 🟠 中 | 摄取中途失败 | 固定顺序 + 幂等重试 |
| 流式被代理缓冲 | 🟡 低 | 上线经代理 | proxy_buffering off |
| 范围失控 | 🟠 中 | 全程 | 严守 M0–M4 MVP 边界 |
| 定位过卷 | 🟡 低(策略) | 投简历时 | 用工程深度差异化 |
