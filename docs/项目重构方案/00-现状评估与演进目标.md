# 00 · 现状评估与演进目标

> 本目录是 agent-server 的重构/演进方案,基于对现有代码的逐文件分析写成,不是凭空规划。
> 阅读顺序:本文(总览)→ 01 选型 → 02 架构 → 03 流程 → 04 路线图 → 05 风险。

---

## 1. 一句话定位

把现在这个"**资源整理批处理器**"演进成一个"**ChatGPT 式个人知识助手后端**":用户上传资料 → 系统自动解析/切分/向量化/组织 → 用户可以基于自己的资料多轮对话。

而要做到这一点,技术上同时做两件事:

1. **降复杂度**:砍掉 Java + Nest 双后端的多语言负担,统一为单一 Node/TS 服务。
2. **补能力**:引入 Qdrant(向量库)+ LangChain/LangGraph(RAG 与 agent 编排)+ embedding,把缺失的"检索增强生成(RAG)"主线补齐。

---

## 2. 现状诚实评估(基于代码,不美化)

现有项目是一个 **polyglot monorepo**:`apps/java-server`(Spring Boot 核心)+ `apps/node-server`(NestJS AI 网关),两者用 HTTP + `X-Internal-Token` 通信。

### 2.1 真实可用的部分(值得保留的资产)

| 模块 | 状态 | 说明 |
|---|---|---|
| **认证** | ✅ 可用 | `POST /api/auth/login` 真实校验 + 签发 JWT;`JwtAuthenticationFilter` 真实拦截校验。**但没有注册接口**,只有一个 `{noop}` 明文种子管理员。 |
| **Agent 运行编排** | ✅ 可用 | `AgentRunController` → 应用层 → `@Async` 异步执行 → 调 Node → 回放事件。FSM 状态 `QUEUED→RUNNING→COMPLETED/FAILED` 真实流转。 |
| **事件溯源 + SSE** | ✅ 可用 | 每步事件逐条落 `agent_run_events` 表,`SseEmitterHub` 真实长连接推送,支持断线后从数据库回放。**这是整个项目最有价值的工程资产。** |
| **资源采集** | ✅ 可用 | Node 侧真实递归扫本地目录(17 种文本后缀)、真实用 axios+cheerio 抓网页正文,有文件数/大小/总量限流。 |
| **Ollama 调用** | ✅ 可用 | 真实调 Ollama:生成计划 / 逐条摘要 / 主题聚类 / 生成报告,每步都有"AI 失败优雅降级"。 |

### 2.2 看起来完整、实则缺失的部分(愿景的关键缺口)

| 能力 | 状态 | 对愿景的影响 |
|---|---|---|
| **文件上传** | ❌ 没有 | 现在只能"扫服务器本地目录",用户根本无法上传自己的资料。这是 ChatGPT 式产品的第一步。 |
| **文档解析(PDF/DOCX)** | ❌ 没有 | 只读纯文本/代码文件,真实资料大量是 PDF/Word/PPT。 |
| **文本切分(chunking)** | ❌ 没有 | 现在是粗暴截断到 12000 字符,无法做检索粒度控制。 |
| **Embedding 向量化** | ❌ 没有 | 没有任何 embedding 调用。 |
| **向量库 / 检索** | ❌ 没有 | 没有 Qdrant/pgvector/FAISS,无相似度检索。**这是"和你的资料对话"的命脉。** |
| **多轮对话** | ❌ 没有 | 没有 conversations/messages,没有 chat 接口,没有上下文记忆。 |
| **工具调用(tool use)** | ❌ 没有 | agent 无法调用检索/整理等工具,谈不上"助手"。 |
| **Token 流式输出** | ⚠️ 部分 | SSE 只用于"运行进度事件",没有用于"模型逐字回答"。 |

### 2.3 摆设型基础设施(Infra Theater)

- **Redis**:docker-compose 和 Spring 配置里都声明了,**代码里零调用**(无缓存、无会话、无队列)。
- **RabbitMQ**:同样声明了,**代码里零调用**。

> 这两个是典型的"配置了但没接进业务"的装样子中间件。重构中要么**让它们真正承重**(Redis 变 BullMQ 队列后端 + 缓存),要么**删掉**(RabbitMQ)。

### 2.4 数据模型现状

只有 3 张表:`users`、`agent_runs`、`agent_run_events`。完全没有 documents / chunks / conversations / messages —— 也就是说,**愿景所需的数据骨架还没起**。

## 3. 当前模型 vs 目标模型的本质差异

这是整个重构最需要想清楚的一点:

| | 现在(资源整理器) | 目标(知识助手) |
|---|---|---|
| 交互形态 | **一次性批处理**(下任务→跑完→出报告) | **持续对话**(随时问,带记忆) |
| 数据生命周期 | 处理完即丢,只留报告 | **持久化为可检索的知识库** |
| 资料来源 | 服务器本地目录 / URL | **用户上传** |
| AI 用法 | 摘要 + 聚类 + 生成报告 | **检索增强生成(RAG)+ 工具调用** |
| 存储 | 关系表存运行记录 | 关系表 + **向量库** |

**关键判断**:现有的"扫描→摘要→聚类→报告"流水线,不是要丢,而是要**降级为愿景里的一个子能力**——即"上传后自动组织/打标签"的增强步骤。真正的新主干是 **摄取(ingestion)→ 检索(retrieval)→ 对话(chat)**。

而那套"异步运行 + 事件溯源 + SSE 进度"引擎,正好可以**原样复用**到"文档摄取进度"上(上传一个大 PDF,解析/切分/向量化是慢操作,需要异步 + 进度推送)。这是现有代码给重构最大的红利。

## 4. 演进总目标(可验收)

重构完成后,系统应当能做到:

1. 用户**注册/登录**,数据按用户隔离。
2. 用户**上传**一份 PDF/Word/Markdown,系统**异步**完成解析→切分→embedding→入 Qdrant,期间通过 **SSE 实时显示进度**(复用现有引擎)。
3. 用户对自己的知识库**多轮提问**,系统做 **RAG 检索**并**流式**返回带**引用来源**的回答。
4. (进阶)用户让 agent 执行复合任务(如"把我上传的这些论文按主题整理成一份综述"),agent 通过 **工具调用** + 多步推理完成,全程可观测。
5. 整个系统是**单一 Node/TS 服务**,本地 `docker compose up` 起 postgres + redis + qdrant + ollama 即可跑通。

---

## 5. 设计原则(贯穿后续所有文档)

1. **复用胜过重写**:FSM、事件溯源、SSE、资源采集、Ollama provider、各类 prompt —— 这些逻辑跟语言无关,移植而非重造。
2. **降复杂度优先**:能在进程内调用的,绝不拆成跨服务 HTTP;能用一个中间件解决的,不引第二个。
3. **框架用在刀刃上**:LangChain/LangGraph 用于它真正省事的地方(文档加载、切分、向量库对接、agent 循环),但**不让框架接管整个应用**,尤其是你已有的事件溯源审计层要自己掌控(见 05)。
4. **多租户安全是底线**:任何检索都必须按 `user_id` 过滤,绝不能让 A 用户检索到 B 用户的资料。
5. **每个决策都要能在面试里自圆其说**:写清"为什么选它、放弃了什么、什么时候不该这么选"。
