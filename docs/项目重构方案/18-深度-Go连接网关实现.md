## 一、背景与动机:为什么要单独写这一篇

### 1.1 这篇和 16 的分工

16 已经把连接层的**理论**讲透了:长连接有状态与无状态扩展的矛盾、backplane、C10K/C10M、pub/sub 为什么会丢。但 16 对 Go 网关只给到 §4.3/§5.4 的**概念级伪代码**——"每连接一读一写 goroutine,有界 channel 表达背压"。

概念到代码之间有一整层真实工程问题被概念跳过了:

- 握手时 JWT 从哪来、怎么防 `alg=none` 降级?
- 网关用自己一套连接表,Node 的读扩散(14 §6③ `filterOnline`)怎么还能看见网关上的在线用户?
- 上行透传给 Node 到底走 gRPC 还是 HTTP,身份信任边界画在哪?
- 下行 pub/sub 频道是全局一个还是每副本一个,各自的代价?
- 有界 channel "满了逐出",**逐出动作本身会不会阻塞扇出循环**?
- bigint 的 `seq` 在 JSON 里怎么不溢出?

**这篇就是把这些"概念没说但落地必踩"的问题逐一拆开**,并诚实复盘第一版 PoC 写薄在哪(§三)。

### 1.2 方案 A(平行接入)vs 方案 B(替换 socket.io)

落地第一个岔路:Go 网关是**接管现有 socket.io 的连接**,还是**作为一条平行的新接入端**?

| 维度 | 方案 A:平行接入(选) | 方案 B:替换 socket.io |
|---|---|---|
| 改动面 | 新增 `gateway/` 模块 + Node 一个内部端点,socket.io 一行不动 | 前端协议、握手、所有实时事件全部迁到网关协议 |
| 回归风险 | 零——P1~P4 的可靠性链路(seq/ack/sync/读扩散)完全不受影响 | 高——多端一致、已读同步、通话信令全要重测 |
| 叙事完整性 | "我能用 Go 扛连接"独立成立,且能和 socket.io **同库对照** | 把已验证的东西推倒重来,叙事反而变弱 |
| 适用前提 | 加深项/PoC,证明能力,不赌生产 | 连接数真成瓶颈、要 all-in Go(16 §4.3 的触发信号) |

**选 A 的本质**:16 §4.3 明确 Go 网关是"瓶颈出现才上"的加深项。在没有真实瓶颈信号时,把生产级 socket.io 推倒去换一套未经实战的协议,是用确定的回归风险换不确定的收益——典型的过度工程。平行接入让"Go 高并发承载"作为一条**可独立演示、可量化对照**的能力线存在,这正是校招深度项目要的"两层叙事都能独立成立"(16 §1.3)。

### 1.3 成功标准(可测试,不是"我觉得做完了")

把模糊的"写个 Go 网关"转成两条可验证的验收线(对齐 16 §4.3/§4.4):

- **无状态**:网关进程持有的可发现状态(用户↔连接)100% 在 Redis;杀掉网关、客户端重连重建 + `/sync` catchUp,Node 业务侧零感知。
- **背压不崩**:一个只连不读的慢消费者,在持续高频下行下被**逐出**,而进程与同副本其它连接安然无恙。

这两条都落成了真 Redis 的 Go 测试(`gateway/test/`),不是注释里的承诺。

## 二、概念与底层原理(实现层)

### 2.1 为什么海量长连接是 Go 的主场:三种并发模型的底层差异

16 §2.5 给了结论(goroutine 适合),这里补足"为什么",把主流语言/运行时按并发模型摊开对照——同一个问题(一台机扛多少条长连接),不同语言给出的答卷:

| 模型 | 代表 | 每连接成本 | 瓶颈 / 落点 |
|---|---|---|---|
| 一连接一(OS)线程 | 老式 Java BIO、Python `threading`、Apache prefork | 线程栈 **~1MB** + 内核调度 | 1 万连接 ≈ 10GB 栈 + 上下文切换风暴,C10K 即崩;**Python 还叠加 GIL**(见下) |
| 单线程事件循环 | Node(libuv)、Nginx、Python `asyncio`、Redis | 一个 fd + 回调/协程帧,极省内存 | **单线程吃不到多核**:扇出/序列化是 CPU 活,打满一个核就到顶 |
| 多 Reactor 事件循环(线程池) | **Java Netty**、nginx 多 worker、Seastar | 一个 fd + handler 对象,EventLoop 绑核 | 能用满多核,但回调式编程(`ChannelHandler` 链)心智负担重,业务逻辑被打散成事件 |
| **M:N 用户态协程 + netpoller** | **Go** goroutine、Elixir/BEAM 进程、**Java 虚拟线程(Loom)** | 栈 **2KB~KB 级**、按需增长 | 内存(百万连接的缓冲)、fd 上限;**写法是同步阻塞、跑起来是异步** |

关键 insight:IM 接入层的负载画像是**"海量连接 + 每连接少量计算 + 偶发 CPU 密集扇出"**。逐个语言看它怎么应对这个画像:

- **Node(单线程事件循环)**:对"海量连接 + 少量计算"极省——一个 fd 一个闭包,百万连接的内存占用比线程模型低一两个数量级。但"扇出"这一步是 CPU 密集(每条群消息要对 N 个在线成员各做一次 JSON 序列化 + 投递),**单线程一旦在序列化上打满一个核就到顶,加机器只能加进程不能加核**——这正是 16 §4.3 说的"Node 副本 CPU 打满"。
- **Java**:经历了完整三级跳。① BIO 一连接一线程,C10K 撞墙;② **NIO + Netty** 转向"多 Reactor 事件循环 + EventLoopGroup 线程池",底层一样是 epoll,能用满多核——这是 Java 扛长连接的工业标准(Spring WebFlux、网关、IM 后端大量用 Netty)。代价是 Netty 的 `ChannelPipeline`/`ChannelHandler` 回调式模型,业务逻辑被切成事件碎片,和 Node 的回调地狱同源。③ **Project Loom 虚拟线程(JDK 21 正式)**:JVM 在用户态调度的 M:N "绿色线程",栈按需增长,**和 goroutine 几乎是同一套东西**——于是"一连接一(虚拟)线程"的写法又活了过来:同步阻塞地 `socket.read()`,但底层被 carrier thread + park 接管,不占 OS 线程。
- **Python**:① `threading` 受 **GIL**(全局解释器锁)钳制——同一时刻只有一个线程执行字节码,**多线程对 CPU 密集的扇出/序列化等于没用**(只在等 IO 时放锁),且仍背着 OS 线程的 1MB 栈成本,是长连接接入层最差解。② `asyncio` 是 Python 的事件循环答卷,和 Node 同构:单线程、协程、`async/await`,省内存但同样吃不到多核,扇出 CPU 打满即顶(FastAPI/Starlette WebSocket 就在这条线上)。③ **free-threaded Python(3.13 实验性 no-GIL 构建)** 想拆掉 GIL 让多核真正并行,但目前有性能回退、C 扩展兼容性未稳——还不是生产可依赖的接入层方案。所以 Python 写高并发网关,现实路径是 asyncio 单进程 + 多进程(`gunicorn -w N`)横向铺核,本质是用进程数绕开 GIL。
- **Go**:goroutine 调度在用户态(**GMP** 模型),栈 2KB 起按需增长,M 个 OS 线程(P 个逻辑处理器)跑 N 个 goroutine,**天生用满多核**;底层 I/O 由 **netpoller(epoll/kqueue)** 统一管理,一条连接阻塞在 `ReadMessage` 上时 goroutine 被 park、让出 M,不占 OS 线程。所以"每连接两个 goroutine"(§2.2)在 Go 里不是负担——它们大多数时间挂在 netpoller 上,不消耗 CPU,又能在扇出时吃满所有核。**同步写法 + 多核 + 省内存**三者同时拿到,正是 Go 在这个负载画像上的甜点。

> **"那 Java 有 Loom 了,为什么这个网关还选 Go?"**(高频面试追问,§九有剧本)——诚实回答:在**虚拟线程 vs goroutine 这一层,二者已高度同质**,Loom 之后"Java 扛不了海量长连接"的旧论断已不成立。选 Go 的理由不在"只有 Go 能做",而在**工程权重**:① 网关是独立小服务,Go 的部署形态(单静态二进制、无 JVM、冷启动毫秒级、常驻内存小)更贴合"接入层 sidecar";② **有界 channel + `select-default` 把背压做成语言原语**(§2.3),Loom 生态要表达同样的背压得自己拼 `BlockingQueue`/Reactor 背压算子;③ netpoller 与 channel 是同一套调度器的原生公民,组合无缝。**反过来,如果整个后端是 JVM 技术栈,用 Netty 或 Loom 写这个网关同样正确**——这不是"Go 赢了 Java",是"在已有技术栈与部署约束下的选型",把这点讲清楚比强行贬低 Java 更显判断力。

> **诚实的天花板**:goroutine-per-connection(以及 Loom 的 vthread-per-connection)在百万连接量级会撞内存墙(1M 连接 × 2 goroutine × 2KB ≈ 4GB 仅栈,加上读写缓冲更多)。真正的 C10M(如 Centrifugo)会放弃 goroutine-per-conn,改用 `gobwas/ws` + 手写 epoll 事件循环——回到"少量事件循环线程管海量 fd"的 Netty 式范式,只是手写得更极致。本网关用 gorilla 的 goroutine 模型,定位是 **C10K~C100K 的工程演示**,而不是 C10M 的极限压榨——这条边界要讲清楚,否则面试官一问"百万连接呢"就露怯(见 §九)。

### 2.2 每连接两个 goroutine + 有界 channel:为什么这样解耦

gorilla/websocket 有一条硬约束:**同一连接的并发写不安全**(并发 `WriteMessage` 会损坏帧)。所以不能让"下行扇出的多个调用方"直接写连接。

解法是经典的"每连接一个写 goroutine + 一条 channel 串行化写":

```
                       ┌─ readLoop  (1 goroutine)  收上行帧 → 透传 Node / 处理心跳
一条 *Conn  ───────────┤
                       └─ writeLoop (1 goroutine)  ←── send chan(有界)←── 所有下行来源
```

- **写串行化**:所有下行(backplane 扇出、ack 回投、错误帧)都只往 `send chan` 里塞,由唯一的 `writeLoop` 取出后串行写连接——天然满足"单写者"约束,无需对连接加锁。
- **读写解耦**:读慢(客户端发得慢)不影响写,写慢(客户端读得慢)不影响读;两个方向各自的反压互不串扰。
- **channel 即背压支点**:`send chan` 有界,这条边界就是 §2.3 的全部魔法所在。

### 2.3 有界 channel 为什么"天生就是背压"

背压(backpressure)的本质是:**当下游(客户端)消费速度跟不上上游(下行扇出)生产速度时,系统必须有一个明确的"满了怎么办"策略,而不是无限缓冲到 OOM**(16 §4.4)。

Go 的有界 channel 把这件事变成一个语言原语:

```go
// gateway/internal/hub/conn.go
func (c *Conn) enqueue(payload []byte) bool {
    select {
    case c.send <- payload:
        return true   // 缓冲未满,入队成功
    default:
        return false  // 缓冲已满 → 慢消费者,交给调用方逐出
    }
}
```

`select { case ch<-x: ... default: ... }` 是**非阻塞发送**:能塞进去就塞,塞不进(缓冲满)立刻走 `default` 返回 false。这一个 `default` 分支就是背压策略的落点——满了不等待、不扩容,而是上报"这是个慢消费者"。

**为什么必须非阻塞**——这是 16 概念伪代码没点破、但落地一旦写错就致命的点:下行扇出是**一个 goroutine 循环投递给该用户的所有连接**(`hub.RouteToUser`)。如果 `enqueue` 用阻塞写 `c.send <- payload`,那么**只要其中一条连接的缓冲满了,整个扇出循环就卡在这条连接上**,后面正常的连接全被它拖住——一个慢客户端瞬间放大成"拖垮整批投递"。`select-default` 把"慢消费者"的影响**隔离在它自己这条连接上**,这才是背压真正生效的样子。

> 对照 Node:socket.io 没有 channel 这种原语,背压要手动盯 `socket.conn.bufferedAmount`,超阈值再决定丢弃/断开——能做,但要显式写、容易漏。Go 把"有界 + 满了走 default"做成了三行,这是选 Go 写网关在工程表达力上的一个真实收益。

### 2.4 presence 镜像:网关凭什么和 socket.io"互相看得见"

平行接入(§1.2)埋了一个隐患:现在有**两套接入层**——socket.io 和 Go 网关。如果用户 X 连在网关、用户 Y 连在 socket.io,Y 发群消息时,Node 的读扩散(14 §6③)调用 `filterOnline(成员)` 来决定推给谁——它查的是 Redis 里的 presence 注册表(15 §5.1)。**只要网关把自己的连接也写进同一套 presence 键,Node 就能无差别地把网关上的在线用户算进在线子集**,无需知道这条连接物理上挂在哪套接入层。

这就是为什么网关的 `presence` 包**必须逐字节对齐** `server/src/realtime/presence.ts` 的键结构:

```
presence:{userId}        ZSET   member=deviceId, score=过期时刻(ms)   ← 用 score 当 per-device TTL
presence:{userId}:meta   HASH   field=deviceId,  value="{replica}:{socketId}"
```

镜像不是"复制一份数据",而是**共享同一份事实**:网关和 socket.io 写的是同一对 Redis 键。这把"连接在哪"这个状态彻底从两套进程内存里抽离到唯一的共享真相(16 §1.2 的核心 insight 落到双接入层场景)。键格式差一个字节,两套接入层就互相瞎了——这也是 §七坑 3 的来源。

## 三、PoC 的边界与已知缺口(诚实复盘)

> 第一版我把网关写成了"能跑的最小骨架",在深度项目的标准下偏薄。把缺口显式列出来,既是工程诚实,也是面试时"你还能怎么深挖"的弹药库——知道边界比假装完整更有说服力。

- **上行只落了 `message.send` + `heartbeat`**:`read.report`(已读多端同步)、通话信令未透传。原因是已读同步要带"排除发送设备"的下行语义(socket.ts 里的 `.except(socket.id)`),而当前下行频道按 userId 路由、不区分设备;补它要在下行帧里加 `excludeDeviceId` 并在 hub 路由时跳过——是确定的增量,不是难点。
- **下行用全局频道 `gw:downlink` 而非 per-replica**:每个网关副本都订阅同一个频道,收到每条下行都要反序列化再判断"这个 userId 在不在我本地"。副本多了,这是无谓的反序列化放大。16 §5.1 的范式是 per-replica 频道 `replica:{id}`,由发布方查 presence 的 `replica` 字段定向发布。当前选全局是为 PoC 省掉"发布方查副本"的一跳;演进方向见 §十。
- **没有上行限流**:16 §4.4 的背压(下行)做了,但限流(上行令牌桶,防单连接刷消息)没做。生产必须补。
- **ack 经内部 HTTP 同步往返**:上行透传是 `网关 → HTTP → Node 落库 → 返回 ack` 一个同步 RTT,`UplinkDuration` 指标量的就是它。高并发下 HTTP 连接复用/超时要调;真要压榨延迟会换 gRPC 流或 Redis stream(§4.3)。
- **无 pprof / 无基准压测**:`/metrics` 有了,但 `net/http/pprof` 和 k6 连接风暴压测没接,"能扛多少连接"目前是推算不是实测。

这些不是 bug,是**有意识的范围划定**:PoC 先把"无状态 + 背压"两条主验收打穿,其余作为标注清晰的演进项。把它们藏起来假装完整,才是真正的质量问题。

## 四、各方案分析(实现选型)

### 4.1 WebSocket 库:gorilla vs coder vs gobwas

| 库 | 模型 | 取舍 |
|---|---|---|
| **gorilla/websocket(选)** | 每连接读写各一 goroutine,API 成熟稳定 | 生态最广、文档全、踩坑资料多;**每条消息有分配**,百万连接级会成 GC 压力 |
| coder/websocket(原 nhooyr) | context 原生、API 更简洁 | 更现代,但 `ReadMessage` 一次性读全帧,流式控制略弱;生态比 gorilla 小 |
| gobwas/ws | 零拷贝、零分配、可不绑 goroutine | C10M 极限选择(Centrifugo 用它);**要自己管 epoll 事件循环**,复杂度高一个量级 |

**选 gorilla 的本质**:网关定位是 C10K~C100K 的工程演示(§2.1 的边界),这个量级 gorilla 的 goroutine 模型完全够用,而它最成熟、最易讲清、踩坑最少。gobwas 的零分配只有在真撞百万连接的 GC 墙时才有意义——为没到的瓶颈选最复杂的库,是另一种过度工程。**知道 gobwas 存在、知道什么时候才需要它**,比一上来就用它更能体现判断力。

### 4.2 下行 backplane 频道:全局 vs per-replica vs per-room

| 设计 | 发布方 | 订阅方 | 取舍 |
|---|---|---|---|
| **全局 `gw:downlink`(PoC 选)** | 落库后按 userId 发一条 | 所有副本都收,各自判本地有没有这人 | 实现最简(发布方不查副本);代价是副本数 × 反序列化放大 |
| per-replica `replica:{id}`(16 §5.1) | 查 presence 拿 `replica`,定向发 | 只订自己的频道 | 无放大;代价是发布方多一次 presence 查询 + 跨副本要先知道副本号 |
| per-room/per-conversation | 按会话发 | 副本订自己持有成员的会话 | 适合超大群广播;订阅关系管理复杂 |

演进路径清晰:PoC 用全局打通链路,副本数上来后切 per-replica(presence 的 `meta` 已经存了 `replica`,发布方分组发布即可)。**先让它对、再让它快**——在链路还没跑通时就上 per-replica 的定向发布,是在没有瓶颈证据时优化,违背 16 §4.3 的"瓶颈才上"原则。

### 4.3 上行透传:内部 HTTP vs gRPC vs Redis stream

- **内部 HTTP(选)**:网关 `POST /internal/gateway/uplink`,Node 同步处理返回 ack。最简单、零新依赖、Node 侧就是一个 express 路由复用既有 `persistMessage`。代价是每次一个 HTTP RTT。
- **gRPC 双向流**:网关与 Node 之间一条长流,上行/ack 在流上跑,省掉每条消息的连接握手,延迟更稳。代价是引入 protobuf + 服务定义,Node 侧要起 gRPC server。
- **Redis stream**:上行写 stream,Node 消费。天然解耦 + 可重放,但 ack 要异步回(网关要等 Node 处理完再回投发送方),链路变长。

**选 HTTP 的本质**:16 §4.3 把网关定义为"业务仍在 Node、网关只透传"。透传量 = 上行消息量,在演示规模下 HTTP 的 RTT 完全可接受,而它让 Node 侧改动**收敛到一个能复用既有落库逻辑的路由**(`internal.ts` 直接调 `persistMessage`,seq/幂等/读扩散一行没改)。gRPC 是"延迟真成问题"时的下一步,不是起点。

### 4.4 JWT 校验:HS256 共享密钥 vs RS256/JWKS

这里有个容易混的点,要和 P3 的 JWKS(项目记忆里的硬约束)分清:

- **网关验的是 our-chat 自己的登录 token**(用户登录后种在 cookie 里的 `token`,HS256,密钥 `JWT_SECRET`)。网关和 Node 同属 our-chat,共享同一对称密钥天经地义——这是**同一信任域内**的验签。
- **P3 的 JWKS(RS256)是跨服务**:our-chat 作为 OAuth IdP 签发 token,agent-server 作为独立服务拉 our-chat 的公钥验签。**跨信任域才需要非对称**(签发方持私钥,验证方只拿公钥,不共享秘密)。
- 所以网关用 HS256 不是"图省事降级",而是信任域决定的正确选择:同域共享密钥,跨域才上 JWKS。

实现上有一个必须守的安全红线——**强制校验签名算法**:

```go
// gateway/internal/auth/jwt.go
if _, ok := t.Method.(*jwt.SigningMethodHMAC); !ok {
    return nil, fmt.Errorf("非预期的签名算法: %v", t.Header["alg"])
}
```

不校验的话,攻击者可以把 token 头部改成 `alg=none`(无签名)或换成 `alg=RS256` 然后用公钥当 HMAC 密钥伪造——经典的 JWT alg 混淆攻击(§七坑 1)。

### 4.5 身份信任边界:为什么 `X-User-Id` 能走头部

直觉上"把用户 id 放 HTTP 头让 Node 信任"很危险——这不就是 socket.ts 里反复强调的"绝不信任客户端自报 userId"吗?

区别在**信任边界画在哪**:

```
客户端 ──(不可信)──► 网关 ──(可信内网,带 X-Gateway-Token)──► Node
        网关在这里验签 JWT,                身份已由网关验签确立,
        把验签结果作为权威身份               Node 信任的是"网关",不是客户端
```

- 客户端到网关这一跳:网关**验签 JWT** 得到身份,绝不信任帧里自报的 `senderId`(`internal.ts` 落库时 `senderId = BigInt(userId)` 取自头部,丢弃帧内 `senderId`)。
- 网关到 Node 这一跳:走**内网 + 共享内部令牌 `X-Gateway-Token`**。Node 校验这个令牌来确认"调用方确实是我们的网关,不是外部伪造"。令牌对了,就信任网关注入的 `X-User-Id`。

所以 `X-User-Id` 走头部是安全的——因为它的可信度由"网关已验签 + 内部令牌确认调用方"共同担保,信任链没有断点。把内部端点暴露到公网、或不校验内部令牌,才是真漏洞(§七坑 7)。

## 五、核心机制(源码走读)

### 5.1 握手:先验签后升级

```go
// gateway/internal/ws/server.go(节选)
cookie, err := r.Cookie("token")               // 与 Node authCookies.ts 的 TOKEN_COOKIE 一致
if err != nil || cookie.Value == "" { 401 }
ident, err := auth.Verify(cookie.Value, secret) // HS256 验签,得到 {UserID, Username}
if err != nil { 401 }
conn, err := h.upgrader.Upgrade(w, r, nil)       // 验过签才升级协议
```

**顺序很重要**:先验签、不过直接 401,**验过才做协议升级**。反过来(先升级再验)会白白完成一次 WebSocket 握手再断开,既浪费又给未授权连接留了打开窗口。`deviceId` 取握手 query,缺省回落随机 `socketID`(每连接唯一),与 socket.ts 的设备语义对齐(15 §5.1)。

### 5.2 连接生命周期与背压逐出

读循环把"连接存活"和"应用层心跳"用两层探测叠起来:

```go
// gateway/internal/hub/conn.go(节选)
c.ws.SetReadLimit(1 << 20)                                  // 单帧上限 1MB,防超大帧打爆内存
c.ws.SetReadDeadline(time.Now().Add(c.hub.heartbeatTimeout))
c.ws.SetPongHandler(func(string) error {                   // 协议 pong → 续读截止
    c.ws.SetReadDeadline(time.Now().Add(c.hub.heartbeatTimeout)); return nil
})
for {
    _, raw, err := c.ws.ReadMessage()
    if err != nil { return }                                // 断开/超时统一走 defer close
    c.dispatch(raw)
}
```

写循环串行化所有下行,并定时发协议 ping:

```go
pingInterval := c.hub.heartbeatTimeout * 2 / 5             // 60s 超时 → 24s 探一次,窗口内必探到
for {
    select {
    case payload, ok := <-c.send:
        if !ok { return }                                  // send 被 close() 关闭 → 连接下线
        c.ws.SetWriteDeadline(time.Now().Add(10*time.Second))
        if c.ws.WriteMessage(TextMessage, payload) != nil { return }
    case <-ticker.C:
        c.ws.SetWriteDeadline(time.Now().Add(10*time.Second))
        if c.ws.WriteMessage(PingMessage, nil) != nil { return }
    }
}
```

逐出在 `hub.RouteToUser` 里发生:`enqueue` 返回 false(缓冲满)即关闭这条慢连接,**并计入 `Evicted` 指标**——背压"在生效"是可观测的,不是黑箱:

```go
// gateway/internal/hub/hub.go(节选)
for _, c := range targets {
    if c.enqueue(payload) { metrics.Downlink.WithLabelValues("delivered").Inc() } else {
        metrics.Evicted.Inc(); c.close()                    // 慢消费者:逐出,但不影响同用户其它连接
    }
}
```

### 5.3 hub 注册表:配额、同设备踢旧、close 幂等

```go
// gateway/internal/hub/hub.go(节选)
h.mu.Lock()
if h.countLocked() >= h.maxConns { h.mu.Unlock(); return nil, ErrOverQuota } // 配额硬上限(坑6)
old := devices[deviceID]; devices[deviceID] = c                              // 同设备:新连接覆盖旧
h.mu.Unlock()
if old != nil { old.close() }                                               // 踢掉旧连接,close 幂等
```

- **配额**:`countLocked() >= maxConns` 在锁内判定,超限拒绝握手——防 fd/内存爆(16 坑 6)。
- **同设备踢旧**:同一 `(userId, deviceId)` 重连时,新连接覆盖旧映射并 `old.close()`,避免幽灵连接堆积(弱网下同设备反复重连)。
- **close 幂等**:`closeOnce sync.Once` 保证摘 presence / 关 channel / 关 ws 只执行一次——读循环出错、写循环出错、被踢、被逐出可能并发触发 close,幂等才不会重复关 channel 引发 panic。
- **unregister 的身份校验**:`if devices[c.deviceId] == c { delete }`——只在"索引里仍是我这条连接"时才删,避免同设备重连后,旧连接的延迟 close 误删了新连接(坑 5)。

### 5.4 presence 镜像(键格式对齐)

```go
// gateway/internal/presence/presence.go(节选)
pipe := r.rdb.TxPipeline()
pipe.ZAdd(ctx, "presence:"+uid, redis.Z{Score: expireAtMs, Member: deviceID})
pipe.HSet(ctx, "presence:"+uid+":meta", deviceID, replica+":"+socketID)
pipe.Exec(ctx)
```

与 `presence.ts` 的 `register`(MULTI 里 ZADD+HSET)逐字段等价。`Refresh`(心跳)只 `ZAdd` 推后 score、不动 meta;`Remove`(优雅断开)`ZRem`+`HDel`。过期摘除是**惰性**的——靠 `getDevices`/`filterOnline` 读取时清 score < now 的项(15 §5.1),网关侧不需要自己跑清理。

### 5.5 上行透传 + 下行 publish(跨进程接缝)

完整一条上行的生命周期,横跨网关与 Node:

```
客户端 ──WS帧{type:"message.send",...}──► 网关 readLoop.dispatch
  └─ 非 heartbeat → upstream.Forward(POST /internal/gateway/uplink, 头:X-Gateway-Token / X-User-Id)
       └─ Node internal.ts: 校内部令牌 → persistMessage(复用 seq/幂等/落库,12)
            ├─ 单聊直推双方 / 群聊 filterOnline(14 §6③)
            ├─ 对每个在线目标 publish gw:downlink {userId, frame:{type:"receiveMessage", data:message}}
            └─ 返回 {type:"message.ack", clientMsgId, seq, serverMsgId}
       └─ 网关把 ack 原样 enqueue 回发送方连接
  其它副本/本副本的 backplane 订到 gw:downlink → RouteToUser → 投给该用户在本副本的所有连接
```

两个信任锚点:Node 侧 `senderId` 取自 `X-User-Id`(网关验签身份)而非帧内自报;去重命中(`deduped`)时**不重复扇出**、只回 ack——和 socket.ts 的 `persistAndBroadcast` 完全同一套语义,只是把 `io.to().emit` 换成 `redis.publish`。

### 5.6 心跳双层:协议 ping/pong + 应用层 heartbeat

网关同时跑两层心跳,各管一件事:

- **协议层 ping/pong**(writeLoop 定时发 ping、PongHandler 续读截止):管"TCP 通道是否假死"(16 §2.2 的中间设备回收/假死)。
- **应用层 `{"type":"heartbeat"}` 帧**(dispatch 里 `Refresh` presence + 续读截止):管"presence TTL 续约",让 Node 的 `filterOnline` 持续认为这台设备在线。

两层都会续读截止,但**只有应用层帧续 presence TTL**——因为 presence 是给 Node 业务看的在线状态,得由客户端**主动声明**"我还在收消息",而不是底层 TCP 没断就算在线(TCP 没断不代表 App 还在前台收消息,16 §2.2)。

## 六、关键实现要点(部署 / 运维)

- **同一个 Redis**:网关的 presence + backplane 必须连 Node 同一个 Redis 实例,否则 §2.4 的"互相看得见"不成立。`REDIS_URL` 两侧同值。
- **订阅连接独立于命令连接**:go-redis 里 `rdb`(命令:presence 读写)和 `subRdb`(订阅:`gw:downlink`)是两个 client——进入订阅态的连接不能再发普通命令(坑 3)。
- **指标**:`/metrics` 暴露连接数 gauge、握手/上下行/逐出 counter、上行端到端耗时 histogram。`gateway_evicted_total` 上扬 = 有慢消费者(背压在生效);`gateway_connections` 接近 `MaxConns` = 该扩副本。
- **优雅退出**:`SIGINT/SIGTERM` → `signal.NotifyContext` 取消 ctx → backplane 退订 + `http.Server.Shutdown` 排空在途请求 + 关 Redis。因为无状态,粗暴 kill 也只是让客户端重连(验收线 1),优雅退出是锦上添花。
- **反向代理**:Nginx/Ingress 要转发 `Upgrade`/`Connection` 头、关 buffering(同 16 §六、坑 1)。
- **fd 与配额**:`MaxConns`(进程级软配额)要配合 `ulimit -n`(OS 级 fd 硬上限)一起调,两者取小。

## 七、踩坑案例(真实可复现,Go 实现特有)

### 坑 1:JWT alg 混淆 —— 不校验算法等于没鉴权
**复现**:`jwt.Parse` 的 keyFunc 直接 `return secret, nil` 不判 `t.Method`。攻击者把 token 头改 `alg=none` 或 `alg=RS256`,用公钥/空签名伪造任意 userId。**根因**:信任了 token 自报的算法。**修复**:keyFunc 里强制 `t.Method.(*jwt.SigningMethodHMAC)`,非 HMAC 直接拒(§4.4)。

### 坑 2:bigint 的 seq 在 JSON 里溢出/报错
**复现**:`seq`/`serverMsgId` 是 BIGINT,Node 直接 `JSON.stringify(message)` 会抛 `Do not know how to serialize a BigInt`。**根因**:JS 原生不序列化 BigInt。**修复**:Node 侧 `bigint-json.ts` 给 `BigInt.prototype` 挂 `toJSON → Number`,下行帧里 seq 才能正常编码(网关只透传字节,不碰这层)。**延伸**:seq 超过 `2^53` 时 Number 会丢精度——演示规模够用,生产要改成字符串编码。

### 坑 3:订阅连接复用命令连接 → 命令全部失败
**复现**:用同一个 go-redis client 既 `Subscribe` 又 `ZAdd`,进入订阅态后 presence 写命令报错。**根因**:Redis 连接进订阅态后只能收发订阅消息。**修复**:`subRdb` 专做订阅,`rdb` 专做命令(§六)。

### 坑 4:WriteMessage 不设 deadline → 一条慢连接卡死 writeLoop
**复现**:写不设 `SetWriteDeadline`,客户端 TCP 窗口满(读得慢)时 `WriteMessage` 无限阻塞,writeLoop 永远卡住、send chan 再也排不出去。**根因**:无写超时。**修复**:每次写前 `SetWriteDeadline(10s)`,写超时即返回触发 close——和有界 channel 一起构成完整背压(写阻塞超时 + 缓冲满逐出双重保护)。

### 坑 5:同设备重连,旧连接的 close 误删新连接
**复现**:同设备快速重连,新连接已登记,旧连接的 close 才慢半拍执行,把 map 里的新连接删了 → 新连接收不到下行。**根因**:close/unregister 不校验身份。**修复**:`unregister` 里 `if devices[deviceId] == c` 才删(§5.3)。

### 坑 6:扇出用阻塞写 → 一个慢消费者拖垮整批投递
**复现**:`RouteToUser` 里 `c.send <- payload` 阻塞写,某连接缓冲满,整个扇出循环卡在它上面,该用户其它设备全收不到。**根因**:背压点用了阻塞语义。**修复**:`enqueue` 用 `select-default` 非阻塞,满了逐出该连接、循环继续(§2.3)——这是背压能否真正隔离慢消费者的命门。

### 坑 7:内部端点没有令牌校验 → 任意人伪造任意身份发消息
**复现**:`/internal/gateway/uplink` 不校验 `X-Gateway-Token`,外部直接 POST 带任意 `X-User-Id` 即可冒充任意用户落库。**根因**:内部端点当成可信却暴露在外。**修复**:校验共享内部令牌,且生产应把该端点限制在内网/不经公网网关(§4.5)。

## 八、Web vs Native 视角(连接到 Go 网关的差异)

| 维度 | Web | Flutter | 原生 Swift |
|---|---|---|---|
| 连到网关的载体 | 浏览器 WebSocket(`/ws`,带 cookie) | `web_socket_channel` / IO socket | `URLSessionWebSocketTask` / Starscream |
| 携带 JWT 的方式 | **HttpOnly cookie 自动带**(同源/CORS withCredentials) | 自管 token,握手 header 注入 | 自管 token,header 注入 |
| `deviceId` | 不稳定(无持久设备 id),多用随机回落 | 可取设备指纹,稳定 | `identifierForVendor`,稳定 |
| 应用层心跳 | 后台标签页定时器被节流 → 回前台立即重连 | 系统挂起即断,恢复重连 | 系统挂起即断,APNs 唤醒 |
| 网关重启的体感 | 回前台/`online` 事件触发重连 + catchUp | 前台恢复重连 | `didBecomeActive` 重连 |

> **结构性差异在"JWT 怎么到网关"**:Web 端 token 是 HttpOnly cookie,握手时浏览器**自动带上**,网关从 `Cookie` 头取——这要求网关与前端同源或正确配 CORS 凭据(本实现从 cookie 取 `token` 正是为此)。原生端没有 cookie 自动机制,token 自管、握手时手动塞 header。但**两端验签逻辑在网关侧是同一套** HS256——接入层差异收敛在"token 怎么送达",不扩散到鉴权核心。设备 id 的稳定性差异则直接影响多端去重:Web 随机 deviceId 意味着每次重连可能被当新设备,presence 里会短暂多一台,靠 TTL 收敛。
