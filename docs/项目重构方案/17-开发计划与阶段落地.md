## 二、P0 · 工程地基

**目标**:让"可靠性"这件事可被自动化验证 —— 没有集成测试,P1 的不丢不重就只是口号。

### 步骤
1. **DB 集成测试harness**:vitest 增加一个跑真实 PG 的测试 project(连 `pnpm docker:pg` 的库),区别于现有 mock-prisma 单元测试。事务/唯一约束/seq 单调只能在真库验证。
2. **接口契约**:引入 OpenAPI(REST,如 `/sync`)+ Socket.io 事件 schema(用 zod 在运行时校验上行入参,满足 CLAUDE.md"外部输入 runtime 校验")。zod schema 同时作为前后端对齐的单一真相,可由 `zod-to-openapi` 导出 OpenAPI 文档。
3. **测试数据工厂**:`test/factory.ts` 造 user/conversation/message,集成测试复用。

### 交付物
- `vitest.config.ts` 拆 `unit` / `integration` 两个 project;`test/integration/setup.ts`(每用例 truncate 相关表)。
- `src/contracts/`:zod schema(`sendMessage` 入参、`/sync` query、ack/push 出参)。
- `docs` 或 `/openapi.json` 端点导出契约。

### 验收
- `pnpm test` 同时跑单元+集成,集成用例能连真库建表、读写、断言。
- 非法上行入参被 zod 拒绝并回 `error` 事件,不进落库路径。

## 三、P1 · 可靠性命脉(本项目的"深度"主线)

**目标**:把 11/12/13 落成代码 —— 消息**不丢、不重、可补齐、未读准**。这是面试主战场,必须做到机制级扎实。

### 步骤(严格按依赖序)

**3.1 Schema 迁移(11 §6.1 / 13 §6.1)**
- `Conversation.nextSeq BigInt @default(0)`。
- `Message.seq BigInt` + `clientMsgId String?` + 索引 `(conversationId, seq DESC)` + 唯一约束 `(conversationId, senderId, clientMsgId)`。
- `UserConversation.lastSyncedSeq / lastReadSeq BigInt @default(0)`。
- 迁移含**存量回填**:`row_number() OVER (PARTITION BY conversation_id ORDER BY timestamp ASC, id ASC)` 给历史消息补 seq,再 `next_seq = MAX(seq)`;`lastReadSeq` 从旧 `lastReadMessageId` 映射(查不到则 0)。
- 旧 `unreadCount` 保留列但不再权威(未读改由两游标算),迁移不删以防回滚。

**3.2 落库 + 发号 + 幂等(11 §5 / 12 §五)**
- 抽 `persistMessage(input)`:单事务内 `UPDATE conversations SET next_seq = next_seq + 1 WHERE id=:conv RETURNING next_seq` 拿 seq → INSERT message(带 seq + clientMsgId)。
- 幂等:捕获唯一约束冲突(PG 23505)→ `findUnique` 取首次结果返回 `{seq, serverMsgId, deduped:true}`,不报错。
- 会话不存在时先 upsert(保留现有逻辑)。

**3.3 上行 ack 协议(12 §4.3 / §6.2)**
- `socket.on('message.send')`:zod 校验 → `persistMessage` → **落库后**回 `socket.emit('message.ack', {clientMsgId, seq, serverMsgId})`。
- 真异常才回 `error`,去重路径回正常 ack(幂等)。
- 保留旧 `sendMessage` 事件一段时间兼容前端,内部转调新逻辑(P1 末或 P2 切前端后移除)。

**3.4 下行 /sync + gap(13 §5/§6)**
- REST `GET /user/sync?conv=X&since=seq&limit=N`:`assertMember` 越权校验 → `WHERE conversationId=X AND seq>since AND isDeleted=false ORDER BY seq ASC LIMIT min(N,200)` → `{messages, hasMore}`。
- 实时推送 payload 带 seq,供客户端 gap 检测(客户端逻辑在前端阶段,后端只保证 seq 连续 + /sync 可补)。

**3.5 未读 = 两游标之差(13 §5.3)**
- 已读上报 `POST /user/read`:单调推进 `UPDATE ... WHERE lastReadSeq < :upto`(防倒退)。
- 会话列表的未读由 `lastSyncedSeq − lastReadSeq` 算出(synced 上报接口或随 ack 推进)。

### 交付物
- 1 个 prisma migration(含回填 SQL)。
- `src/services/message.ts`(`persistMessage`)、`src/routes/sync.ts`、socket handler 改造。
- 集成测试:并发同 clientMsgId 重发只入库一次、seq 会话内严格 +1、/sync 范围查询正确分页、gap 场景补齐、未读=差值、越权 /sync 被拒。

### 验收(可测试成功标准)
- [ ] 并发 50 次同 `(conv,sender,clientMsgId)` 重发 → `messages` 恰好 1 行,50 次 ack 的 seq 全一致。
- [ ] 单会话并发写 100 条 → seq 为 1..100 无洞无重。
- [ ] 离线后 `/sync?since=k` 精确返回 `seq>k`,分页 `hasMore` 正确。
- [ ] 乱序已读上报(80 晚于 100 到达)→ `lastReadSeq` 不倒退。
- [ ] `/sync` 拉非成员会话 → 403。
- 全绿 + typecheck + lint + build。

## 四、P2 · 多端一致 + 多副本(可扩展性证据)

**目标**:落 15/16 —— 一个用户多设备都收到、已读跨端清、加副本仍可达。

### 步骤
1. **presence 注册表入 Redis**(15 §5.1):`user:{id}:devices` Hash + 每设备 TTL key;引入 `ioredis`。心跳刷新 TTL,断开/超时摘除。
2. **pub/sub backplane**(16 §5.1):副本订阅 `replica:{self}`,跨副本下行经 Redis 转投;本副本直投。可先用 `socket.io-redis-adapter` 起步,再讲清内部即 pub/sub。
3. **用户级扇出**(15 §5.2):下行目标从"对方一条连接"改为"对方所有设备 + 发送者其它设备"(消息回显)。
4. **已读跨端同步**(15 §5.3):`read.sync` 推给同用户其它端,单调 max。
5. **per-device synced 态**(15 §6):新增 `DeviceSyncState` 表。

### 交付物
- `src/realtime/presence.ts`、`src/realtime/backplane.ts`、`DeviceSyncState` migration、`ioredis` 依赖。
- docker-compose 加 redis 服务。
- 集成测试:双副本(两个 io 实例 + 共享 Redis)跨副本投递可达;多端已读 max 收敛;幽灵连接 TTL 摘除。

### 验收
- [ ] 副本 A 的连接能收到从副本 B 发起的消息(pub/sub 代投)。
- [ ] 同用户两设备并发已读 → 位点取 max,不倒退。
- [ ] 自己发的消息同步到自己其它端。

## 五、P3 · AI 横向集成(不动消息架构)

**目标**:按 10 §四 AI 横向层解耦边界 + 06,把 agent-server 作为横向能力接入,**消息可靠性架构零改动**。

> 边界纠偏:AI 对话**不是 IM 消息**,不走 seq/ack/扩散,agent-server **不碰 Message 表**(10 §四、行 177/179)。
> "加 AI 不动核心"的横向扩展叙事,正依赖 AI **不**穿过消息可靠性核心。AI 回复有自己的持久化
> (agent-server 的 `messages`/`run_events`),续传靠事件溯源 + `Last-Event-ID`,与 IM 的 seq 续传**同构但不共用**。

### 步骤
1. 共享 JWT(方案 JWKS,非对称):our-chat 自身即 OAuth2.1/OIDC IdP,经 `/.well-known/jwks.json` 暴露公钥;
   agent-server 用 `jwks-rsa` 按 `kid` 拉公钥验签 our-chat 签发的 access_token,严格校验 `iss`/`aud=agent-server`。
   token 经 `agent-server` scope 派生该 audience。两侧此机制均已落地,P3 只需配置串联 + 端到端验证。
2. zero-touch 用户同步(06 §3):our-chat 用户首次访问 agent-server 时,按 JWT `sub`/`username` 在
   agent-server 本地 `User` 表 findOrCreate(id 对齐 our-chat userId,满足会话/文档的 FK),后续请求命中缓存不重复写。
3. SSE 透传 + 断线回放:AI 回复走 agent-server 的 SSE 流(`token`/`done`/`error`);断线用 `Last-Event-ID`
   从 `run_events` 回放该序号之后的事件衔接实时流(06 §2.3、16 §2.3),agent-server 自有,不依赖 IM seq。
4. our-chat 接入入口(06 §4):联系人列表置顶虚拟联系人「AI 助手」,点进去是特殊会话窗口,消息流走
   agent-server SSE 而非 socket.io。

### 验收
- [ ] our-chat 签发的 access_token(含 `agent-server` scope)被 agent-server 的 JWKS 路径校验通过。
- [ ] 我方用户首次访问 agent-server 自动在其本地建账(zero-touch),会话/文档 FK 不报错。
- [ ] AI 流式回复断线可经 `Last-Event-ID` 续传,不重复持久化;全程 IM 的 Message/seq 表零改动。

## 六、P4 · 群扩散 + 已读聚合(可砍/简化)

**目标**:落 14 读扩散。主体读扩散(写 1 份),在线成员 pub/sub 扇出,离线补拉;大群只做"已读人数"聚合不做逐人逐条;@提醒走旁路。

### 步骤
1. 群会话成员表读取 + `getMembers` / `presence.filterOnline`。
2. `fanoutGroupMessage`:落库 1 条 → 在线成员扇出 → 离线靠 /sync。
3. @提醒:`Message.mentions` 旁路标记(`hasMention` 或 mention 表)。
4. 已读聚合:大群只存"已读人数",批量游标上报。

### 验收
- [ ] 群消息只落 1 行,各成员按自己 synced 补拉。
- [ ] @我 的消息单独可查/高亮,不淹没在普通未读。

## 七、P5 · Go 网关 + 可观测性(加深项,时间紧直接砍)

**目标**:仅当长连接/扇出成真瓶颈时才上(16 §4.3)。无状态 Go 网关:连接生命周期 + 心跳 + JWT 校验 + 订阅 pub/sub 下行 + 限流背压;业务仍在 Node。

### 步骤(可选)
1. Go 网关 PoC:WebSocket 接入 + 有界 channel 背压 + 连接数配额。
2. 上行透传 Node(gRPC/内部 HTTP),下行经 pub/sub 回网关。
3. 可观测:Prometheus 指标(连接数、扇出延迟、ack 时延)+ 结构化日志。

### 验收(若做)
- [ ] 网关无状态:重启后客户端重连重建 + catchUp,业务不受影响。
- [ ] 背压:慢客户端不拖垮进程(有界 channel 丢弃/断开)。

## 八、测试策略(贯穿全程)

| 层 | 工具 | 覆盖 |
|---|---|---|
| 单元 | vitest + mock prisma(现状沿用) | 纯逻辑、校验、边界 |
