# 09 · 校招主线项目规划 —— IM 深度 × AI 助手融合

> 本文不是 agent-server 单仓的重构方案,而是把 **our-chat(IM)** 与 **agent-server(AI 后端)** 两个已有仓库,
> 收敛成**一个能撑校招、能扛面试深挖的全栈主线项目**的规划 + 开发计划。
> 前置阅读:本系列 00–08(agent-server 重构) + our-chat/docs/06(前端多端接入)。

---

## 〇、先纠正一个认知:你不缺项目,你缺深度

你说"定一个全栈项目帮我拿校招 offer"。但盘完代码后,真实结论是:

> **你已经有了那个全栈项目(our-chat),而且它的广度已经超过绝大多数双非学生作品。校招缺的不是再起一个新项目,是把它从"功能堆得很多"打磨成"有一个能被面试官往下挖三四层、你每层都答得出的硬核"。**

这条判断直接决定了下面所有安排的方向:**收敛 + 补深度 + 接 AI**,而不是再铺新功能。再起新项目 = 把现有的 OAuth2 授权服务器、四端、分片上传、WebRTC 这些已沉没的硬资产浪费掉,是负 ROI。

## 一、现有家底盘点(基于代码,不美化)

### 1.1 our-chat —— 真·全栈 IM,资产很硬

| 层 | 现状 | 校招信号 |
|---|---|---|
| **后端** | Express 4 + Socket.io 4 + Prisma 6/PostgreSQL + JWT(jose)+ bcrypt | 标准 Node 全栈后端 |
| **鉴权** | **完整 OAuth 2.1 / OIDC IdP**:authorize/token/introspect/revoke/userinfo/discovery + **PKCE(S256)** + **refresh token 轮换 + family_id 复用检测** + JWKS | **这是最硬的资产**——多数在职工程师都没手写过授权服务器 |
| **数据建模** | MySQL+MongoDB 双库 → **统一到 PG**;Message 用 JSONB(灵活字段)+ 复合索引 `(conversation_id, timestamp DESC)` + DISTINCT ON 命中 Index-Only Scan | 体现真实 DB 性能意识,不是 CRUD |
| **IM 功能** | 好友(申请/拉黑)、群(角色/上限)、会话(单聊/群)、消息(编辑/撤回/@、JSONB extra)、已读/未读/置顶/免打扰/归档、typing 指示 | 功能面接近一个真 IM |
| **文件** | 分片上传 / 断点续传 / 秒传(MD5)/ 流式 / 图片压缩(sharp) | 真实工程难点,已踩过坑(见 record.md) |
| **实时音视频** | WebRTC 语音,Socket.io 做信令(offer/answer/ICE 转发) | P2P + 信令,差异化亮点 |
| **前端** | React 19 + antd 5 + Redux Toolkit + redux-persist + socket.io-client + crypto-js;**size-limit + Lighthouse CI 性能预算** | 前端工程化成熟度高于一般学生 |
| **多端** | web + **mobile-flutter** + **mobile-swift(原生,Tuist 工程)** | 一套后端撑三端,叙事强 |

### 1.2 agent-server —— 现成的 AI 后端(就是"那一点 AI")

| 能力 | 现状 |
|---|---|
| 形态 | NestJS 单体,**设计上就是 our-chat 的"系统助手"后端**(见 agent-server/docs/06) |
| AI 主线 | RAG(上传→解析→切分→embedding→Qdrant)+ 自研 agent 工具循环(OpenAI tool calling,~200 行,不绑框架) |
| 工程资产 | **FSM + 事件溯源 + SSE**(可断线重连回放)、Redis pub/sub backplane、BullMQ 摄取队列、OpenAI 兼容模型层(Ollama 本地 / 第三方 API 一键切) |
| 与 IM 的接法 | 06 已设计:our-chat 联系人列表置顶虚拟联系人"AI 助手",共享 JWT 密钥(方案 C),agent-server 按 JWT `sub` findOrCreate 用户 |

**关键红利**:agent-server 的"事件溯源 + SSE 断线回放"和 IM 的"消息可靠投递 + 离线补拉"是**同构问题**——下面会复用同一套思路。

### 1.3 诚实的深度缺口(这就是要补的)

盘完代码,IM 的"广度"够了,但有几个**面试官一挖就见底**的深度洞:

| 缺口 | 现状证据 | 为什么是深度洞 |
|---|---|---|
| **消息时序靠 `timestamp` 排序** | Message 排序索引是 `(conversation_id, timestamp DESC)`,`timestamp = now()` | 多端/服务器时钟不单调,高并发同毫秒会乱序——**经典错误**,应改为**会话内单调 seq** |
| **可靠投递语义不完整** | `Message.status` 有 `sent` 字段,但看不到客户端幂等键 / ack / 重传 / 去重的闭环 | "消息会不会丢/重/乱"是 IM 面试第一深挖点,目前答不出闭环 |
| **离线增量同步靠 messageId 位点** | `lastReadMessageId` 是字符串 id | 没有单调游标无法做"拉取 seq > X 的消息"和 **gap 检测补拉** |
| **群扩散模型未刻意设计** | 消息只存会话维度(读扩散),MessageRef 标注为遗留兼容 | 写扩散 vs 读扩散是必问的权衡题,需要能讲清你为什么这么选 |
| **多端一致性** | 同用户多设备的连接注册、多端扇出、已读位点跨端同步未见 | "我手机读了,电脑的未读怎么清"——多端一致是深挖点 |
| **连接层耦合在单体 Socket.io** | 连接态大概率在单实例内存 | 多副本下"连在 A 副本的用户,消息从 B 副本怎么送达"——可扩展性深挖点 |
| **AI 未接入** | agent-server 与 our-chat 尚未联调 | "带一点 AI"还没落地 |
| **IM 侧可观测性** | 未见指标/链路 | 收尾加分项 |

> 这张表就是开发计划的来源。**每一行都是一个面试官想问、你做完就能答的深挖点**——这正是"全栈深度"的定义。

## 二、校招定位:补深度,不加功能

### 2.1 这个组合最终证明什么

> "我做了一个支持多端(web/Flutter/原生 iOS)的 IM,自己实现了 **OAuth 2.1 授权服务器**做统一登录;消息层做到了**不丢、不重、不乱序、离线可补、多端一致**;并把一个**自托管 RAG/Agent 的 AI 助手**作为系统联系人融进会话流。我能讲清每一个可靠性/扩散/一致性决策的取舍。"

这比"我用 XX 框架做了个聊天 demo"高好几档,因为它命中校招后端/全栈面试的**标准深挖剧本**:长连接、消息可靠性、时序、扩散、多端一致、鉴权、可扩展性。

### 2.2 优先级(和 memory 里的定位一致)

破双非的主武器是**大厂实习 + 内推**,项目是**面试深挖弹药**。所以:

- **能讲通 > 功能全**。每个里程碑结束都要能演示 + 能深挖,而不是攒一堆半成品。
- **深度叙事押在消息核心**(可靠性/时序/一致性),这是 ROI 最高、面试官最爱挖的地方——它是项目的"深度证据"。
- **AI 是横向扩展能力层,不是深度证据,但不设功能上限**。RAG 保留,Agent 工具可持续加;它通过干净的解耦边界挂在 IM 上,**加多少都不动消息核心架构**(见 §五)。区别只是:消息核心用来证明"我有工程深度",AI 用来证明"我跟得上趋势、会做 AI 工程化",两者分工不同,不互相挤占。

## 三、目标架构

在现有 our-chat 基础上,新增/强化三块:**① 消息可靠性核心(Node,必做)② 接入网关(Go,可选加深)③ AI 助手集成(复用 agent-server)**。

```
                         ┌──────────────────────────────────────────────┐
                         │   客户端三端:web(React) / Flutter / Swift     │
                         └───────────────┬──────────────────────────────┘
                                         │ WebSocket(长连接) + HTTP + (AI)SSE
                ┌────────────────────────┼─────────────────────────────────┐
                │  【可选·T2 加深】Go 接入网关                              │
                │  长连接保活/心跳/重连 · 连接注册表(Redis) · 下行扇出      │
                │  限流/背压 · 鉴权校验(验 our-chat 签发的 JWT)            │
                └────────────────────────┬─────────────────────────────────┘
                                         │ gRPC / 内部 HTTP(上行业务)
        ┌────────────────────────────────┼────────────────────────────────┐
        ↓                                ↓                                ↓
┌─────────────────┐          ┌─────────────────────┐         ┌─────────────────────┐
│ our-chat 业务后端 │          │  消息可靠性核心       │         │  AI 助手(agent-server)│
│ Express+Socket.io│          │  seq 分配/幂等去重/   │         │  RAG/Agent + SSE      │
│ 好友/群/会话/文件 │          │  ack/重传/离线队列/   │         │  共享 JWT(方案C)      │
│ WebRTC 信令       │          │  扩散/已读           │         │  作"系统联系人"        │
│ OAuth2.1 IdP      │          └──────────┬──────────┘         └──────────┬──────────┘
└────────┬─────────┘                     │                               │
         ↓                  ┌─────────────┼──────────────┐                ↓
   ┌──────────┐         ┌───▼───┐     ┌───▼────┐     ┌────▼────┐     ┌─────────┐
   │PostgreSQL│         │ Redis │     │PostgreSQL│   │ pub/sub │     │ Qdrant  │
   │(消息/社交)│        │presence│    │(消息持久)│   │(下行扇出)│    │(向量)   │
   └──────────┘         └───────┘     └─────────┘    └─────────┘     └─────────┘
```

**分层取舍**:
- **T1 全在 Node 内做**(Express+Socket.io 现有栈):消息可靠性核心是纯协议 + 数据库设计问题,不需要换语言就能做出深度。**先把这层做扎实,项目就能投。**
- **T2 才考虑 Go 接入网关**:当你要讲"高并发长连接 + 多副本扇出"时,把接入层抽成 Go 服务最有说服力,也正好接上你 Node→Go 的成长线(见 memory)。**但它是加深项,不是前置依赖**,时间紧直接砍,T1 的可靠性故事独立成立。

## 四、深度硬核规格(项目的命脉)

这一节是"全栈深度"的全部含金量所在。每个子问题给:**目标 → 怎么做 → 坑 → 面试能答什么**。

### 4.1 会话内单调序列号(seq):时序的地基

- **目标**:每条消息在其会话内有一个**服务器分配的单调递增 seq**(不是全局,是 per-conversation),客户端按 seq 排序、检测空洞。
- **怎么做**:发消息落库时,在会话维度原子自增(Postgres `UPDATE conversations SET next_seq = next_seq + 1 RETURNING`,或 Redis `INCR conv:{id}:seq` + 落库)。Message 表加 `seq BIGINT`,索引改为 `(conversation_id, seq DESC)`。
- **坑**:① 别用 `timestamp` 排序(时钟不单调 + 同毫秒并发会乱);② 自增要原子,高并发下两条消息不能拿到同一个 seq;③ Redis 分配 + DB 落库要保证不跳号/不丢号(崩溃恢复)。
- **面试能答**:"为什么不用时间戳排序""seq 怎么保证单调且不重""分配 seq 和落库的原子性怎么处理"。

### 4.2 上行可靠:不丢、不重(客户端 → 服务器)

- **目标**:网络抖动/重发下,消息不丢也不重复入库。
- **怎么做**:客户端生成 `clientMsgId`(UUID,幂等键)随消息上行 → 服务器入库成功后回 **ack(带分配的 seq + 服务器 msgId)** → 客户端收到 ack 才标记"已发送";超时未收 ack 则**重发同一个 clientMsgId**;服务器对 `(conversationId, senderId, clientMsgId)` 做**唯一约束**,重复直接返回首次结果(幂等)。
- **坑**:① ack 丢了客户端会重发,服务器必须靠幂等键去重而不是再插一条;② 唯一约束冲突要"返回已存在的那条"而不是报错;③ 发送中状态要持久化到客户端本地,杀进程重开不丢。
- **面试能答**:"at-least-once + 幂等 = exactly-once 效果"的经典论证。

### 4.3 下行可靠:离线不丢 + 增量补拉(服务器 → 接收方)

- **目标**:接收方离线时消息不丢;上线后能增量补齐,且不重复。
- **怎么做**:消息一律**先持久化再推送**;接收方在线则实时推 + 等其 ack;**离线或未 ack 的消息**靠"接收方已同步到的 seq 游标"补:客户端上线时带 `lastSyncedSeq` 调 `GET /sync?conv=X&since=seq` 拉取 `seq > lastSyncedSeq` 的消息。客户端发现实时流里 seq 跳变(gap)也触发同样的补拉。
- **坑**:① `lastReadMessageId`(现有)是 messageId,要补一个**单调游标**(seq)才能做范围查询和 gap 检测;② 补拉要分页(大量离线消息);③ 已读位点 vs 已同步位点是**两个不同的游标**,别混。
- **面试能答**:"离线消息怎么补""客户端怎么发现自己漏了消息""为什么读取位点和同步位点要分开"。

### 4.4 群消息扩散:写扩散 vs 读扩散(必问权衡)

- **目标**:能讲清你为什么这么扩散,以及大群怎么办。
- **怎么做(建议混合)**:中小群用**读扩散**(消息只存会话一份,各成员按 seq 游标拉,省存储)——你现状就是这个,顺势讲清;在线成员走实时推。把"写扩散"(给每个成员一份收件箱)作为**对比方案讲取舍**,并说明何时该切(超大群、每人不同的已读/免打扰状态需要独立时)。
- **坑**:① 读扩散下"每个成员的未读数/已读位点"要按 `UserConversation` 单独维护(现有表已支持);② 群已读回执是**写放大**(N 人 × M 消息),大群只做"已读人数"不做"逐人已读";③ @提醒要单独通知通道。
- **面试能答**:"写扩散读扩散怎么选""微信/钉钉大群为什么这么设计""你的群已读为什么只显示人数"。

### 4.5 多端一致性

- **目标**:同一用户多设备,消息都到、已读位点同步。
- **怎么做**:连接注册表从"单实例内存"升级为 **Redis 维护 `user:{id}:devices → 连接所在副本`**;下行消息扇出到该用户**所有在线设备**;已读上报时更新 `UserConversation.lastReadMessageId` + seq 游标,并通过下行通道**通知该用户其它设备**清未读。
- **坑**:① 多副本下连接注册表必须共享(Redis),否则跨副本找不到连接;② 已读同步是"自己的其它端"要收到,别推给会话对方;③ 设备离线要从注册表清理(心跳超时摘除)。
- **面试能答**:"多端未读怎么同步""连接状态放哪""一个用户连了三个设备,消息怎么找到这三条连接"。

### 4.6 连接层与可扩展性(T2 的故事,可先用 Node 占位)

- **目标**:长连接保活、断线重连、多副本下消息可达。
- **T1(Node)**:Socket.io + 心跳 + 重连;连接注册表入 Redis;下行用 **Redis pub/sub** 做副本间扇出(复用 agent-server/docs/02 §6.1 同款 backplane——"事件落库 + pub/sub 实时扇出 + 断线按 seq 回放",IM 和 SSE 是同构问题,直接搬)。
- **T2(Go,可选加深)**:把"接入层"(长连接管理 + 鉴权校验 + 下行扇出 + 限流背压)抽成独立 **Go 网关**,业务仍在 Node。这是"高性能服务端 + Node→Go 迁移"的硬证据;goroutine + channel 管理海量连接是 Go 的主场。
- **坑(Go 网关)**:① 连接数上去后 fd 上限/内存;② 没有背压会被下行打爆;③ 网关无状态,连接态全在 Redis;④ Go↔Node 的内部协议(gRPC)序列化成本。
- **面试能答**:"长连接怎么保活""10 万连接单机扛不扛""为什么接入层用 Go 而业务用 Node"。

## 五、AI 能力层(横向扩展,不影响核心架构)

agent-server 已是现成 AI 后端。关键定位:**AI 是挂在 IM 上的横向能力层,通过解耦边界接入,加多少功能都不改动消息核心架构**。RAG 保留,Agent 工具可持续加——它们是"往一个稳定的接入口上插能力",不是改主干。

### 5.1 为什么 AI 不影响核心架构(解耦边界)

AI 能力全部走**一条固定的接入边界**,IM 消息核心对"接的是哪种 AI"无感知:

```
IM(our-chat) ──共享 JWT(方案C) + SSE 流式 + 多租户隔离──► AI 后端(agent-server)
   ↑ 消息核心(seq/ack/扩散)完全不感知 AI 是 RAG 还是 Agent 还是翻译
```

- **鉴权**:our-chat 签的 JWT,agent-server 直接验;首次 `findOrCreate` 用户(零接触同步)。
- **传输**:AI 会话/任务走 SSE 流式(agent-server 已有事件溯源 + 断线回放),不占用 IM 的 Socket.io 消息通道。
- **隔离**:所有 AI 检索强制按 `user_id`/`conversation_id` 过滤(复用 agent-server/docs/03 §2.3)。
- **结论**:新增任何 AI 能力 = 在 agent-server 内加一个工具/接口 + 在 IM 端加一个入口,**消息核心的 seq/ack/扩散/一致性逻辑一行不用动**。这就是"横向扩展"的含义。

### 5.2 必做:AI 助手作为系统联系人 + RAG(复用 06 设计)

- our-chat 联系人列表置顶虚拟联系人"AI 助手";点进去是特殊会话,消息流走 agent-server(SSE 流式)。
- **RAG 保留**:上传文档→解析→切分→embedding→Qdrant→基于自己资料流式问答带引用(agent-server 00–08 已规划)。
- **验收**:IM 内和"AI 助手"对话,上传文档后基于自己资料带引用回答;断线重连能回放。

### 5.3 能力清单(可持续横向加,按需排期,不互相阻塞)

| 能力 | 复用什么 | 类型 | 备注 |
|---|---|---|---|
| **RAG 知识问答** | embedding + Qdrant + 检索 | 已规划 | 必做,5.2 |
| **Agent 工具循环** | 自研 ReAct 循环 + run-engine(03 §3) | 已规划 | 可加 MCP、更多工具 |
| **聊天记录语义搜索** | 把消息异步入向量库(强制隔离) | IM 原生 | 和 IM 强相关,推荐早做 |
| **长群未读摘要** | LLM 拼未读 prompt | IM 原生 | 轻、演示效果好 |
| 智能回复建议 / 实时翻译 | LLM(+ 现有 i18n) | IM 原生 | 按兴趣加 |

> 这些能力**彼此独立、可任意先后落地**,因为它们共用同一条接入边界(5.1)。想加就加,不构成架构债。**唯一的纪律**:深度叙事(面试深挖)仍押在消息核心,AI 数量再多也不替代"我有工程深度"这件事——它证明的是另一件事(AI 工程化 + 跟得上趋势)。

## 六、开发计划与里程碑

原则:**每个里程碑可独立演示 + 对应一个面试深挖点**;关键路径是 P1→P2→P3,P4/P5 可按时间裁剪。

### P0 · 基线与度量(1 周)
- 拉起 our-chat 全栈本地环境(server + web + PG + Redis),补一个**最小压测/灰度脚本**(模拟 N 客户端发消息),作为后续"可靠性改造前后对比"的基准。
- 引入结构化日志 + 基础指标(连接数、消息收发计数、端到端延迟),为 P5 可观测打底。
- **验收**:能一键起;能跑出一组"改造前"的基线数据(丢消息率/乱序率/延迟)。

### P1 · 消息可靠性核心(2~3 周)★ 项目深度命脉
- 落 **会话内单调 seq**(4.1)+ Message 表/索引改造 + 数据迁移。
- 落 **上行幂等 ack/重传/去重**(4.2):clientMsgId 唯一约束 + ack 协议。
- 落 **离线增量同步**(4.3):同步游标 + `GET /sync?since=seq` + gap 补拉。
- **验收**:压测下**零丢失、零重复、零乱序**;离线一段时间后上线能补齐;杀客户端进程重开不丢"发送中"消息。**到这里项目就能带去投实习/校招了,不要等后面。**

### P2 · 多端一致 + 多副本可达(2 周)
- 连接注册表入 Redis(4.5);下行扇出到用户所有在线设备。
- **Redis pub/sub backplane**(4.6 T1):起 2 个 server 副本,跨副本消息可达;断线按 seq 回放。
- 已读位点多端同步。
- **验收**:同账号 web+手机同时在线,消息两端都到;一端读了另一端未读清零;2 副本下消息不丢。

### P3 · AI 助手集成(1.5 周)★ "带一点 AI"落地
- 共享 JWT 联调 our-chat ↔ agent-server(方案 C);"AI 助手"虚拟联系人 + 流式会话面板(web)。
- 选做其一:**聊天记录语义搜索** 或 **长群未读摘要**(5.2)。
- **验收**:IM 内和 AI 助手对话流式带引用;选做功能可演示;多租户隔离验证(查不到别人的)。

### P4 · 群扩散打磨 + WebRTC 收尾(1.5 周,可裁剪)
- 群读扩散下的未读/已读人数/@提醒(4.4);把写扩散作为"对比方案"写进文档。
- WebRTC 语音稳定性(ICE 失败重连、信令边界),时间够可加视频/群通话。
- **验收**:群聊未读/已读/@ 正确;1v1 语音稳定。

### P5 · Go 接入网关 + 可观测(2~3 周,加深/可砍)★ 高性能服务端证据
- 抽 **Go 长连接网关**(4.6 T2):连接管理 + JWT 校验 + 下行扇出 + 限流背压;Node 业务不动。
- 压测:Go 网关 vs Node 直连的连接数/内存/延迟对比(**要有数据曲线**)。
- 接入指标面板(连接数/消息延迟/扇出耗时/网关排队深度);链路追踪可选。
- **验收**:Go 网关扛住 N 万连接不 OOM;有"引入网关前后"的对比数据;面板能看核心指标。

### 依赖与裁剪
```
P0 ─► P1(命脉)─► P2 ─► P3(AI)
                  └─► P4(群/RTC,可裁) 
                  └─► P5(Go网关/可观测,加深,可砍)
```
- **最小可投版本 = P0+P1+P3**:可靠 IM + AI 助手,已经是强作品。
- P2 显著加深(多端/可扩展),建议做。
- P4/P5 按时间和实习节奏裁剪;**实习窗口优先级高于把 P5 做完**(memory:实习 > 项目)。

## 八、取舍、范围控制与踩坑

### 8.1 Go 网关到底上不上
- **不是必须**。T1(P1)的可靠性深度**不依赖 Go**,纯 Node 就能做出面试官爱挖的全套。
- 上 Go 的唯一理由:你要"高性能服务端 + Node→Go"这条简历线(与 memory 的成长方向一致)。**它是加分,不是地基**——P5 没时间就砍,P1 的故事独立成立。这条与 agent-server 系列"单体优先、有真瓶颈再抽"的判断一致,只是 IM 的长连接是它真正的瓶颈场景。

### 8.2 范围红线(防止又一次范围爆炸)
- **AI 不限数量但限形态**:AI 能力横向加无妨(§五),但保持"挂在 IM 上的增强",**不要把它单独抽成一个对外的知识库/工作流平台产品**(06/07 已论证那条独立赛道供大于求)。区别在落点:AI 是 IM 的功能,不是另起的产品。
- 不追求 IM 功能"全":朋友圈/小程序/表情商城这类**与深度无关的功能不做**。
- 三端不必都追平:web 做全,Flutter/Swift 至少各跟到"登录 + 收发消息 + 离线补拉"能演示多端一致即可。

### 8.3 全局踩坑预警(写进 README 当弹药)
- seq 分配与落库的原子性:崩溃恢复时不能跳号/重号。
- 反向代理/网关对 SSE 与 WebSocket 的 buffering:要关,否则流式与实时全挂。
- Redis pub/sub 是"尽力而为":必须配合"事件落库 + seq 回放"双保险,漏了能补(同 agent-server 02 §6.1)。
- 多端已读同步别推错对象(推给自己其它端,不是会话对方)。
- 移动端 SSE/长连接进后台会断:前台用 `Last-Event-ID`/seq 游标重连(our-chat/docs/06 §5.3 已记录)。
- 双写(消息入 PG + 入向量库做语义搜索)要定顺序 + 失败补偿(同 agent-server 03 §1.3)。
