# 08 · 测试策略与工程化

> 本文回答两个问题:**"测试怎么写?"** 与 **"测试框架选什么?"**
> 核心原则:**按层分级覆盖,不追求行覆盖率,关键路径 100%**——尤其是 05 风险文档里标 🔴 的安全/数据完整性项。

---

## 1. 测试金字塔(分层策略)

```
                ┌──────────────┐
                │   E2E 测试    │  少而精,验关键端到端流程(auth/RAG/SSE)
                └──────────────┘
              ┌──────────────────┐
              │  集成测试         │  每个 shared module 至少 1 个 happy path
              │  (testcontainers)│  真实 Postgres/Redis/Qdrant,不 mock
              └──────────────────┘
            ┌──────────────────────┐
            │    单元测试           │  纯逻辑、prompt 拼装、工具函数
            │   (mock 一切 IO)     │  快,数量多
            └──────────────────────┘
```

**比例**:数量 = 单元 ≫ 集成 > E2E;**信心**:集成 + E2E ≫ 单元(单元再多也证明不了"系统真能跑")。

---

## 2. 框架选型:Vitest(否定 Jest)

经评估,**用 Vitest 而不是 Jest**。完整对比与改判理由见 commit 4 引入的 `vitest.config.ts` 注释,要点:

| 维度 | Vitest | Jest |
|---|---|---|
| 速度 | esbuild/Vite,2-5× 快 | ts-jest 编译重 |
| ESM | 原生 | 历史包袱 |
| TS | 原生 | 需 ts-jest/babel |
| Jest API | **高度兼容**(describe/it/expect/vi.mock) | —— |
| 覆盖率 | V8 原生,快 + 准 | istanbul 插桩 |
| Watch 模式 | 极快 + UI | 慢 |
| NestJS 集成 | 用 `unplugin-swc` 处理装饰器,社区方案成熟 | 官方默认(历史路径) |

**关键澄清**:NestJS 默认 cli 模板用 Jest 不代表 NestJS 绑定 Jest——`@nestjs/testing` 的 `Test.createTestingModule()` 在 Vitest 下完全工作。

**Vitest 何时不该选**:依赖某个只有 Jest 适配器的旧插件——本项目核心(testcontainers/supertest/openai SDK mock)都框架无关,不踩。

## 3. 工具栈完整清单

| 类别 | 选 | 用途 |
|---|---|---|
| 测试运行器 | `vitest` | 全部测试 |
| NestJS 装饰器编译 | `unplugin-swc` | 让 Vitest 跑通 NestJS 的 `@Injectable`/`@Controller` 等 |
| HTTP E2E | `supertest` | 启动 NestJS app,跑真实请求 |
| 覆盖率 | `@vitest/coverage-v8` | V8 原生覆盖率 |
| 容器化集成测试 | `testcontainers`(node 版) | 自动起 Postgres/Redis/Qdrant 容器 |
| Postgres 容器 | `@testcontainers/postgresql` | 模块化封装 |
| Redis 容器 | `@testcontainers/redis` | 同上 |
| Qdrant 容器 | `GenericContainer`(qdrant/qdrant 镜像) | 社区无 typed 模块,用通用接口 |
| Mock LLM | 自写 fake openai client | 不调真 Ollama |
| Pre-commit | `husky` + `lint-staged` | 提交前自动跑 lint + 相关 unit test |

## 4. testcontainers 配置策略

### 4.1 启动开销摊销

testcontainers 首次启动一个容器 20-60 秒,但有几个机制能把开销摊薄:

1. **镜像本地缓存**:第一次拉取后,后续测试用本地镜像,秒级启动。
2. **会话复用**(默认行为):同一个 Jest/Vitest 进程内,容器**只起一次**,所有测试共享。
3. **`globalSetup`**:在所有测试前起容器、设环境变量、跑迁移;`globalTeardown` 收尾。
4. **本地 Ryuk 守护**:容器在进程退出时自动被回收,无残留。

### 4.2 集成测试模板(以 Postgres 为例)

```ts
// test/integration/setup-postgres.ts
import { PostgreSqlContainer, StartedPostgreSqlContainer } from '@testcontainers/postgresql';
import { execSync } from 'node:child_process';

let container: StartedPostgreSqlContainer;

export async function setup() {
  container = await new PostgreSqlContainer('postgres:16-alpine').start();
  process.env.DATABASE_URL = container.getConnectionUri();
  // 跑 Prisma 迁移到测试库
  execSync('pnpm prisma migrate deploy', { stdio: 'inherit', env: process.env });
}
export async function teardown() {
  await container?.stop();
}
```

```ts
// vitest.config.ts(节选)
import { defineConfig } from 'vitest/config';
export default defineConfig({
  test: {
    globalSetup: ['test/integration/setup-postgres.ts'],
    pool: 'forks',                 // 避免多 worker 抢容器
    testTimeout: 60_000,           // 容器启动留余地
    coverage: { provider: 'v8', thresholds: { lines: 80, branches: 80, functions: 80 } },
  },
});
```

Redis/Qdrant 用相同模式各起一个 setup 文件。

## 6. 覆盖率分级策略

| 层 | 阈值 | 备注 |
|---|---|---|
| 整体 lines/branches/functions | **≥ 80%** | jest/vitest config 设阈值,不达不通过 |
| **关键路径**(见 §7) | **100%**(严格) | 安全和数据完整性,漏一项 = 风险 |
| Boilerplate(DTO / 纯配置 provider / 单行 getter) | 不计入 | `collectCoverageFrom` 排除 |
| 第三方 SDK 封装薄层(纯 forward) | 不计入 | 测了等于测 SDK |

**触发逻辑**:`vitest --coverage` 输出 + CI 把 `coverage/coverage-summary.json` 解析,低于阈值挂掉构建。

## 7. 关键路径必测清单(对照 05 风险文档)

这些**必须**有专属测试,且测试要看得出"漏了会发生什么":

| 路径 | 测什么 | 测试位置 |
|---|---|---|
| 🔴 **多租户向量隔离** | A 用户检索绝不命中 B 的 chunk | `e2e/rag-tenant-isolation.e2e.test.ts` |
| 🔴 **JWT 鉴权** | 无 token/过期/篡改全部 401/403 | `e2e/auth.e2e.test.ts` |
| 🔴 **Postgres↔Qdrant 双写一致** | 写一半失败可幂等重试,不留孤儿 | `integration/ingest-consistency.integration.test.ts` |
| 🔴 **事件溯源 sequence_no 单调** | 并发写不会重号,断线回放完整 | `integration/run-events.integration.test.ts` |
| 🟠 **文件上传安全** | 路径穿越、类型/大小限制、超大文件 | `e2e/upload.e2e.test.ts` |
| 🟠 **embedding 维度校验** | 启动时维度不匹配直接拒绝启动 | `unit + e2e` |
| 🟠 **SSE 多副本扇出**(M6 之后,启 2 副本测) | worker 发的事件正确路由到 web | `e2e/sse-backplane.e2e.test.ts` |

**这些测试漏一个 = 生产留地雷**,5 风险表里的 🔴 全部对应一条测试用例。

## 8. Mock 策略

什么时候 mock,什么时候用真容器,**有明确分界**:

| 测的对象 | 策略 |
|---|---|
| 纯函数 / 业务逻辑 | **全 mock 依赖**,只测函数行为 |
| Service 方法(调 Prisma/Redis/Qdrant) | 单元测试 mock 依赖;**集成测试用真容器**(两层都要) |
| LLM 调用 | **mock**(假 openai SDK):AI 不该参与单元/集成测试的"输出正确性"判断,只测调用契约 |
| HTTP 客户端(网页抓取) | mock(`nock` 或自写 mock server) |
| 时间 | `vi.useFakeTimers()`,避免测试依赖墙上时钟 |
| 随机/UUID | 注入或 mock(确定性) |

**关键原则**:**LLM 输出永远 mock**——既因为慢和不稳定,也因为"测 AI 的输出对不对"本身就是错误的测试目标(那应当是离线 eval 的事,见 §11)。

## 9. pre-commit 钩子与 CI 联动

### 9.1 本地 pre-commit(快速反馈)

`husky` + `lint-staged`,**只跑快测试**:

```jsonc
// package.json
{
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --fix",
      "vitest related --run --reporter=dot"   // 只跑改动文件相关的 unit test
    ]
  }
}
```

- pre-commit 不跑集成/E2E(慢,容器要起)
- pre-commit 失败 = 不许 commit(不允许 `--no-verify` 绕过,见系统约束)

### 9.2 CI(GitHub Actions,全量)

```yaml
- run: pnpm install
- run: pnpm tsc --noEmit       # 类型检查
- run: pnpm lint
- run: pnpm test:unit          # 全部单元
- run: pnpm test:integration   # testcontainers(Docker 在 Action runner 上原生支持)
- run: pnpm test:e2e           # 端到端
- run: pnpm test:coverage      # 阈值校验
```

`pnpm` scripts 拆得对应分明,本地也能单独跑。

## 10. 实战示例骨架(三层各一)

### 10.1 单元测试(mock 全部依赖)

```ts
// src/shared/rag/retriever.spec.ts
import { describe, it, expect, vi } from 'vitest';
import { Retriever } from './retriever';

describe('Retriever', () => {
  it('强制注入 user_id 过滤,不会越权', async () => {
    const qdrant = { search: vi.fn().mockResolvedValue([]) };
    const llm    = { embed: vi.fn().mockResolvedValue([0.1, 0.2]) };
    const prisma = { documentChunk: { findMany: vi.fn().mockResolvedValue([]) } };

    const r = new Retriever(qdrant as any, llm as any, prisma as any);
    await r.retrieve(42n, 'hello');

    expect(qdrant.search).toHaveBeenCalledWith('knowledge_chunks', expect.objectContaining({
      filter: { must: [{ key: 'user_id', match: { value: 42 } }] },
    }));
  });
});
```

### 10.2 集成测试(真容器)

```ts
// test/integration/prisma.integration.test.ts
import { describe, it, expect, beforeAll } from 'vitest';
import { PrismaClient } from '@prisma/client';

let prisma: PrismaClient;
beforeAll(() => { prisma = new PrismaClient(); });

describe('Prisma 迁移与基本 CRUD', () => {
  it('能创建并查回 user', async () => {
    const u = await prisma.user.create({
      data: { username: 'alice', passwordHash: 'x', displayName: 'A', roleCode: 'USER' },
    });
    const got = await prisma.user.findUnique({ where: { id: u.id } });
    expect(got?.username).toBe('alice');
  });
});
```

### 10.3 E2E 测试(启全栈)

```ts
// test/e2e/health.e2e.test.ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { Test } from '@nestjs/testing';
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';
import { AppModule } from '../../src/app.module';

let app: INestApplication;
beforeAll(async () => {
  const mod = await Test.createTestingModule({ imports: [AppModule] }).compile();
  app = mod.createNestApplication();
  await app.init();
});
afterAll(async () => { await app.close(); });

describe('GET /health', () => {
  it('200 + 各依赖 status=up', async () => {
    const res = await request(app.getHttpServer()).get('/health');
    expect(res.status).toBe(200);
    expect(res.body.details.postgres.status).toBe('up');
    expect(res.body.details.redis.status).toBe('up');
    expect(res.body.details.qdrant.status).toBe('up');
  });
});
```

## 11. AI 输出的离线 eval(不是测试,放这提一下)

> **"测 LLM 输出对不对" ≠ 单元/集成测试**。

后期(M5/M6)可以做**离线 eval**:
- 准备一个 fixture 集(问题 + 期望答案要点)
- 跑 RAG 流程,用更强的 LLM("LLM as judge")给分
- 跟踪召回率/answer 质量随 prompt/chunk size 变化的趋势

这是**评估**,不是 CI 测试——慢、贵、有随机性,不挡 commit/merge。本项目 MVP 不强求,但要在简历能讲到。

## 12. 测试踩坑

1. **testcontainer 启动卡 Docker daemon**:CI 上要确保 runner 装了 Docker;本地 mac 上 Docker Desktop 起着。
2. **测试间数据污染**:每个测试用 `beforeEach` 清表或用事务回滚;集成测试不要假设顺序。
3. **NestJS 装饰器在 Vitest 不工作**:用 `unplugin-swc`,确保 `swcrc` 启用 `decoratorMetadata`。
4. **覆盖率阈值锁太紧**:遇到正当原因(如纯类型文件)用 `collectCoverageFrom` 排除,不要用 `// istanbul ignore next` 偷绕。
5. **e2e 测试启动慢**:`globalSetup` 复用容器 + NestJS app;避免每测一次起一次。
6. **mock LLM 失真**:mock 一定要返回**真实结构**的 JSON,别返回 `{}`,否则测过了生产挂。
7. **不要追覆盖率**:有些代码(错误路径、降级分支)不容易测——宁可显式标注"未覆盖原因",也别写假测试凑数。

> **按层分级,关键路径 100%,boilerplate 不强求覆盖**;**Vitest + testcontainers + supertest** 是 2026 的 Node/TS 最佳实践;**LLM 输出永远 mock,留给离线 eval**;每个 feature commit 带测试,**绿了才提**。
