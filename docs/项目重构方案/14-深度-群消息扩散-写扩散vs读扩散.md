# 14 · 深度 · 群消息扩散 —— 写扩散 vs 读扩散

> 对应 [10 目标架构](10-目标架构.md):扩散策略是"消息可靠性核心"的一部分,直接决定 [13 离线补拉](13-深度-下行可靠-离线增量补拉.md) 是"会话流范围查询"还是"各自收件箱拉取"。
> 主题:一条群消息要送达 N 个成员,**消息到底存几份、在写时还是读时扇出**。这是 IM 面试**必问的架构权衡题**。

---

## 一、背景与动机:为什么"扩散"是个问题

### 1.1 单聊很简单,群聊一上来就要做选择

单聊:一条消息,两个人,存一份、推一份,没什么可纠结。

群聊不一样。一条群消息,要让**群里 N 个人都能看到**,而且:
- 每个人有**自己的未读数**(我未读 3 条,你未读 0 条)。
- 每个人有**自己的已读位点**(我读到第 100 条,你读到第 50 条)。
- 每个人有**自己的免打扰/置顶/归档**状态。
- 有人在线(要实时推),有人离线(要离线补)。

于是一个朴素的问题冒出来:**这一条消息,在系统里到底存几份?是写进来时就给 N 个人各复制一份,还是只存一份、等每个人来读时再分发?** 这就是**扩散(fan-out)**问题,而它有两个对立的答案——这正是面试官想听你权衡的地方。

### 1.2 本质:扇出(fan-out)—— 一份输入变 N 份投递

"扩散"的学名是 **fan-out(扇出)**:一个生产者的一条消息,要"扇"给 N 个消费者。这个问题在所有"一对多分发"系统里都存在——**它不是 IM 独有的,而是 Feed 流(微博/推特时间线)、订阅推送、广播的共性问题**。理解它在别处怎么解,就理解了 IM 怎么选。

核心矛盾是一个永恒的**时间 vs 空间(存储)/ 写 vs 读**的权衡:

> 你要么在**写的时候**多做功(把消息复制 N 份,读时很快)——**写扩散**;
> 要么在**读的时候**多做功(消息存一份,每个人读时去聚合)——**读扩散**。
> 没有免费的午餐,只能根据"写多还是读多、群大还是群小"选。

### 1.3 为什么这题是面试硬通货

- 它是**没有标准答案的权衡题**,正好考"你能不能讲清取舍",而不是"你会不会写 CRUD"。
- 它直接关联**可扩展性**:大群/超大群下,选错扩散模型会让存储或读路径爆炸。
- 它和"微博大 V 发一条,千万粉丝怎么收到"是**同一道题**,面试官可以无限往下挖(混合扩散、推拉结合)。

**能把读/写扩散的机制、成本、何时切换讲清,本身就是"我有架构权衡能力"的硬证据。**

## 二、概念与底层原理

### 2.1 写扩散(fan-out on write / push model)

**写的时候就把消息复制到每个接收者的"收件箱"。**

```
群里 Alice 发 1 条消息(群有 200 人):
  写时:服务器把这条消息 INSERT 进 200 个人各自的收件箱(inbox)
  读时:每个人只读自己的 inbox,seq 是 inbox 内的位点,极快、各自独立
```

- 类比:**把信复印 200 份,分别塞进 200 个人的信箱**。
- 存储:1 条逻辑消息 → N 条物理记录(**写放大 N 倍**)。
- 读:每个人读自己的收件箱,**O(1) 定位、互不干扰**,天然支持"每人不同的已读/位点"。

### 2.2 读扩散(fan-out on read / pull model)

**消息只存一份(会话维度),每个人读的时候,从这份共享流里按自己的位点拉。**

```
群里 Alice 发 1 条消息:
  写时:只 INSERT 进会话的 messages 表 1 条(seq 是会话内位点)
  读时:每个成员按自己的 lastSyncedSeq 从同一条会话流拉 seq > 游标 的消息
```

- 类比:**信只贴在公告栏一份,每个人自己来看,各自记住"我看到第几条了"**。
- 存储:1 条逻辑消息 → 1 条物理记录(**无写放大**)。
- 读:每个人都查同一张会话消息表,**用各自游标过滤**(13 文档的 `/sync?since=`)。

### 2.3 两者的对称性(一张表看懂)

| 维度 | 写扩散(push) | 读扩散(pull) |
|---|---|---|
| 消息存几份 | N 份(每人一份) | 1 份(会话一份) |
| 写成本 | **高**(写放大 N 倍) | **低**(写 1 次) |
| 读成本 | **低**(读自己 inbox) | 较高(共享流 + 各自游标过滤) |
| 每人独立位点/状态 | 天然支持(inbox 内位点) | 需在 `UserConversation` 单独维护 |
| 大群代价 | 写爆炸(200 人 = 200 倍写) | 读时聚合,但写不爆 |
| 典型适用 | 读远多于写、群不大、要强个性化 | 写不能放大、超大群、省存储 |

**关键直觉**:写扩散把成本压在**写**和**存储**;读扩散把成本压在**读**。选哪个,看你的负载"写重还是读重、群大还是群小"。

### 2.4 同构题:微博/推特的 Feed 扩散

这题在 Feed 流里是**明星问题**,IM 直接借鉴:

- **普通用户发微博 → 写扩散**:粉丝不多,发的时候推进每个粉丝的 timeline,粉丝刷新很快。
- **大 V 发微博 → 写扩散会爆炸**:1000 万粉丝,一条微博要写 1000 万次 timeline,不可行。
- **推特/微博的解法:混合**——普通用户写扩散,大 V 改读扩散(粉丝刷新时实时去拉大 V 的最新),两者在读时合并。

把这个映射回 IM:**小群 ≈ 普通用户(写扩散可承受),万人大群 ≈ 大 V(必须读扩散)**。所以 IM 的答案也往往是**混合**(§4.3)。

## 三、现状(基于真实 schema)

```prisma
model Message {
  conversationId String   // 消息存在"会话"维度,一条群消息只有一条记录 → 读扩散
  seq            BigInt   // 会话内位点(11 加)
}
model UserConversation {   // 每个成员一行,维护各自的位点/未读/免打扰 → 配合读扩散
  lastSyncedSeq BigInt
  lastReadSeq   BigInt
  isMuted/isPinned/isArchived ...
}
// MessageRef:注释标注"保留作兼容",是 MySQL/Mongo 时代遗留,不是有意的写扩散收件箱
```

- 现状是**读扩散**:消息存会话一份,`UserConversation` 给每个成员维护独立位点/状态。
- `MessageRef` 是历史遗留的轻量引用表,**不是**刻意设计的写扩散收件箱——文档里要讲清这一点,免得被面试官问"你这 MessageRef 是干嘛的"时答不上。
- **顺势讲清**:现状选读扩散是对的(省存储、大群友好),本文要做的是把"为什么这么选、什么时候该切写扩散"补成一个有意识的权衡,而不是"碰巧这样"。

## 四、各方案分析

### 4.1 写扩散:成本、收益、失败模式

- **收益**:读极快(读自己 inbox)、每人位点/状态天然隔离、离线补拉简单(拉自己 inbox 未拉部分)。
- **成本/失败模式**:
  - **写放大**:N 人群发一条 = N 次写。1 万人群每条消息 1 万次写,**写入与存储双爆炸**。
  - **成员变更代价**:有人入群,要不要回灌历史到他 inbox?退群要不要清理?
  - **一致性窗口**:N 份副本不是原子写完的,扩散中途崩溃要补偿。
- **适用边界**:成员数可控(几百内)、读远多于写、要强个性化 inbox 的场景。

### 4.2 读扩散:成本、收益、失败模式

- **收益**:写轻(1 次)、省存储(1 份)、超大群友好(发消息成本与群大小无关)、成员变更便宜(改 `UserConversation` 即可)。
- **成本/失败模式**:
  - **读时聚合**:每个成员读都查共享会话流;热门大群同一张表读并发高,要靠 `(conversation_id, seq)` 索引 + 缓存扛。
  - **每人状态要单独表**:未读/已读/免打扰必须在 `UserConversation` 维护(现状已有)。
  - **已读回执是写放大点**(§5.2)。
- **适用边界**:写不能放大、群可能很大、存储敏感——**IM 群聊的主流默认**。

### 4.3 混合方案(本项目建议)

不是二选一,而是**按群规模/场景分流**:

| 场景 | 选择 | 理由 |
|---|---|---|
| 单聊 / 中小群(默认) | **读扩散** | 写轻省存储,现状即此,顺势做扎实 |
| 超大群(万人级) | **读扩散**(更要) | 写扩散在这里直接爆炸,只能读扩散 |
| 需要强个性化 inbox / 每人不同的复杂投递规则 | 写扩散(局部) | 个性化收益 > 写放大成本时 |
| @我 的提醒 | **独立通知通道**(不靠扩散模型) | 见 §5.3 |

**本项目落点**:主体读扩散(写不放大、大群友好),**把写扩散作为"对比方案 + 何时切换"写进文档和面试话术**,不实际实现写扩散收件箱(避免范围爆炸)。**讲清"我为什么不需要写扩散、什么信号出现我才会上",比硬做一套写扩散更显判断力**(与 11 §4.3 对 seqsvr 的处理同一原则)。

### 4.4 群已读:为什么大群不做逐人已读

"已读回执"在群里是个**写放大炸弹**:
- 群 N 人,每人读 M 条消息 → "谁读了哪条"是 **N × M** 的状态。万人群刷几屏,已读状态量爆炸。
- **微信的选择**:群消息**不显示**逐条已读(只有单聊有已读)。**钉钉**显示"已读 X 人/未读 Y 人"(强协作场景,愿意付这个成本)。
- **本项目策略**:**大群只做"已读人数"聚合,不做"逐人逐条已读"**;单聊/小群可做精确已读。这是基于写放大的有意识取舍。

## 五、核心机制(读扩散下的关键设计)

### 5.1 每人位点在 `UserConversation`,不在消息里

读扩散下消息是共享的,**个性化状态全部外置**到 `UserConversation`(每成员一行):

```
一条群消息(messages 表,1 行,seq=100)
   ├─ 成员 A 的 UserConversation: lastSyncedSeq=100, lastReadSeq=100  → 已读
   ├─ 成员 B 的 UserConversation: lastSyncedSeq=100, lastReadSeq=80   → 未读 20
   └─ 成员 C 的 UserConversation: lastSyncedSeq=60(离线), ...         → 待补拉
```

- 未读数 = 各自 `lastSyncedSeq − lastReadSeq`(13 §5.3),**同一条消息对不同人是不同的未读贡献**——这正是读扩散需要 per-member 位点的原因。

### 5.2 群已读的写放大与缓解

- 逐人已读 = N×M 状态(§4.4)。缓解:
  - 大群**只存聚合**:`已读人数`(一个数,随成员 read 位点推进时增量更新)。
  - 已读上报**批量 + 防抖**:用户滑动浏览不逐条上报,用 `read.uptoSeq` 一次推进到最新可见 seq(类似 13 的游标推进)。
- 在线成员的实时推:读扩散下,发消息时查会话**在线成员**,通过 pub/sub 扇给他们(连接层在 [16](16-深度-连接层与可扩展性.md) 展开);离线成员不推,留待补拉。

### 5.3 @提醒为什么要单独通道

- @某人 是**强提醒**,不能淹没在普通未读里(大群里你可能有 999 未读,但 @你 的那条必须高亮/单独通知)。
- 现状 `Message.mentions` 是 JSONB,存了被 @ 的人。扩散时:**除了正常入会话流,还要给被 @ 的成员单独打一个"提及"标记/通知**(可在 `UserConversation` 加 `hasMention` 或单独的 mention 表)。
- 这是"扩散模型之外的旁路通道":普通消息走读扩散,@提醒额外走一条定向通知,**两者解耦**。

## 六、关键实现(读扩散发送路径)

```ts
async function fanoutGroupMessage(input: { conversationId: string; senderId: bigint; content: string; clientMsgId: string; mentions: bigint[] }) {
  // ① 落库:读扩散只写 1 条(同事务拿 seq,见 11/12)
  const { message } = await persistMessage(input);

  // ② @提醒旁路:给被 @ 成员打标记(§5.3),不影响主消息
  if (input.mentions.length) {
    await markMentions(input.conversationId, message.seq, input.mentions);
  }

  // ③ 实时扇出:只推在线成员(离线靠补拉,13)
  const members = await getMembers(input.conversationId);        // 读会话成员
  const online = await presence.filterOnline(members);           // Redis 连接注册表(15/16)
  publishToConversation(input.conversationId, {                  // pub/sub 扇出(16)
    type: 'message.push', seq: message.seq, serverMsgId: message.id, ...
  }, online);
  // 不给每个成员复制消息副本——这就是"读扩散":写 1 份,读时各自按游标拉
}
```

> 对比写扩散的话,② 处会变成"循环 INSERT N 个成员的 inbox"——那正是写放大来源。读扩散把这步省掉,代价转移到每个成员读时的范围查询(13 §5.1)。

## 七、踩坑案例(真实可复现)

### 坑 1:把 `MessageRef` 当成写扩散收件箱
**复现**:看到 `MessageRef` 以为是每人一份的收件箱,在它上面做位点,结果它只是会话/发送者维度的遗留引用,逻辑全错。**根因**:没认清现状是读扩散、MessageRef 是历史遗留(§三)。**修复**:位点放 `UserConversation`,消息读 `messages`。

### 坑 2:大群做逐人逐条已读 → 状态爆炸
**复现**:万人群每人每条都记已读,`read_receipts` 表瞬间上亿行,写入和查询双崩。**根因**:N×M 写放大(§4.4)。**修复**:大群只做"已读人数"聚合 + 批量游标上报(§5.2)。

### 坑 3:发群消息时给所有成员(含离线)都实时推
**复现**:给离线成员也走实时推,推送通道被无效投递塞满,且离线成员根本收不到还以为送达了。**根因**:没区分在线/离线。**修复**:只推在线成员,离线靠补拉(§6 ③、13)。

### 坑 4:@提醒混在普通未读里被淹没
**复现**:大群 @你 的消息和 999 条普通未读混一起,用户完全没注意到被 @。**根因**:没有独立提醒通道。**修复**:mention 旁路标记 + 高亮/单独通知(§5.3)。

### 坑 5:读扩散下没给会话流加 `(conv, seq)` 索引 → 大群读慢
**复现**:热门大群每个成员补拉都全表扫 messages,DB 被打满。**根因**:缺索引(或没用上 11 建的索引)。**修复**:确保 `idx_messages_conv_seq` 命中 Index Range Scan(13 §5.1)。

## 八、Web vs Native 视角

扩散是**服务端**决策,三端客户端拿到的都是"按自己游标的消息流",所以协议层三端一致。差异在**已读/未读的本地呈现**:

| 维度 | Web | Flutter | 原生 Swift |
|---|---|---|---|
| 已读上报时机 | 会话可见 + 滚动到底(`IntersectionObserver`) | 列表 visible 回调 | `willDisplayCell` / 可见性 |
| 后台已读同步 | 标签页后台难触发,回前台补报 | 前台恢复补报 | 前台恢复补报 |
| 大群未读渲染 | 虚拟列表(react-window)抗大量消息 | ListView.builder 懒加载 | UITableView 复用 |
| @提醒呈现 | 桌面通知(需授权) | 本地/远程推送 | APNs + 角标 |

> 三端的共性:**已读位点都是"用户实际看到"才推进**,而"看到"的判定(cell 可见 / 滚动到底)各端 API 不同,但语义一致——推进 `lastReadSeq`,触发 `read.uptoSeq` 上报。这条贯穿 13/14/15。
