# 03 · 核心流程设计

> 三条主流程的时序:① 文档摄取(ingestion)② RAG 对话 ③ agent 工具编排(进阶)。
> 贯穿 02 的 web 池 / worker 池 / Redis backplane 设计。

---

## 1. 文档摄取流程(上传 → 可检索知识)

这是"上传资料 → 自动处理"的落地。**慢操作全部异步**,进度复用现有运行引擎 + SSE。

### 1.1 时序

```
[Web 进程]                          [Redis/BullMQ]        [Worker 进程]              [外部依赖]
   │
   │ ① POST /documents (multipart)
   │    校验 JWT + 文件类型/大小
   │    存文件到 storagePath
   │    建 Document(status=uploaded)
   │    建 Run(kind=ingestion, refId=docId, status=queued)
   │ ② 入队 ingest job ───────────────►│
   │ ③ 立刻返回 202 {runId, documentId} │
   │                                    │
   │ ④ 前端 GET /runs/{runId}/stream    │
   │    建立 SSE,SUBSCRIBE run:{runId}  │
   │                                    │ ⑤ worker 取出 job
   │                                    │   Run→running, 发 run_started
   │                                    │   ├─ 解析(PDF/docx/md)────────────► 文件
   │                                    │   │  发 step: parsing
   │                                    │   ├─ 切分 chunking
   │                                    │   │  发 step: chunking (n chunks)
   │                                    │   ├─ 批量 embedding ──────────────► Ollama
   │                                    │   │  发 step: embedding (i/n)
   │                                    │   ├─ upsert points ───────────────► Qdrant
   │                                    │   │  写 DocumentChunk 行 ──────────► Postgres
   │                                    │   └─ Document→ready, Run→completed
   │  ◄── Redis PUBLISH run:{runId} ────┤   发 run_completed
   │ ⑥ web 副本收到 → SSE 推给前端       │
```

### 1.2 关键设计点

| 环节 | 设计 | 复用/新增 |
|---|---|---|
| 上传 | `multipart/form-data`,限大小/类型,存盘后只入队不处理 | 新增 |
| 解析 | LangChain loader(`PDFLoader`/`DocxLoader`/`TextLoader`) | 新增(替代现有"只读纯文本") |
| 切分 | `RecursiveCharacterTextSplitter`,按 token,设 size+overlap | 新增(替代"截断 12000 字符") |
| embedding | `OllamaEmbeddings`,**批量**调用,带缓存 | 新增 |
| 入库 | Qdrant upsert(payload 带 user_id)+ Postgres 写 chunk 元数据 | 新增 |
| 进度 | 每步 `RunEvent` 落库 + Redis pub/sub 推 SSE | **复用 run-engine** |
| 失败 | 任一步失败:Run→failed,Document→failed,记 errorMsg,job 可重试 | 复用 FSM + BullMQ 重试 |

### 1.3 双写一致性(Postgres ↔ Qdrant)

摄取要同时写 Qdrant(向量)和 Postgres(chunk 元数据),必须定好顺序与补偿:

- **推荐顺序**:先写 Postgres `DocumentChunk`(拿到 `chunkId`)→ 再 upsert Qdrant(payload 带 `chunk_id`)→ 回填 `qdrantPointId`。
- **失败补偿**:Qdrant upsert 失败 → 整个 job 失败重试;重试时按 `document_id` 先清掉该文档已写的 points(幂等),再重来。
- **删除**:删文档时,Postgres 删 chunks + Qdrant 按 `document_id` filter 删 points,两边都做。

### 1.4 现有"资源整理"逻辑去哪了

现有的"扫目录 + 抓 URL + AI 摘要 + 聚类"不是丢弃,而是**降级为可选的增强步骤**:

- 目录扫描/URL 抓取 → 变成"导入来源"的一种(本地路径/网页),与"用户上传文件"并列。
- AI 摘要/聚类 → 变成摄取完成后的**可选 enrichment**(给文档自动打标签/归类),写入 Document/Chunk 的元数据,供检索过滤。

## 2. RAG 对话流程(和你的资料对话)

这是"个人知识助手"的核心交互。**全在 web 进程内同步完成**(检索快,生成流式)。

### 2.1 时序

```
[Web 进程]                                   [外部依赖]
   │ ① POST /conversations/{id}/messages {query}
   │    校验 JWT,取 userId
   │ ② 存 user message(role=user)──────────► Postgres
   │ ③ embedding(query) ────────────────────► Ollama
   │ ④ Qdrant search(top-k, filter user_id) ─► Qdrant   ★ 强制按 user 过滤
   │ ⑤ 取回 chunk_ids → 从 Postgres 读原文 ──► Postgres
   │ ⑥ 组装 prompt:
   │     system + 检索到的 context + 最近 N 轮历史 + query
   │ ⑦ chat 流式生成 ───────────────────────► Ollama (stream)
   │ ⑧ 边生成边 SSE 逐 token 推给前端
   │ ⑨ 生成完:存 assistant message + citations(引用了哪些 chunk)
   │ ⑩ SSE 发 done(带引用列表)
```

### 2.2 关键设计点

| 环节 | 设计 | 坑(详见 05) |
|---|---|---|
| 检索过滤 | **必须** `filter: user_id == currentUser`,封装在 `shared/rag` 强制注入 | 漏过滤 = 越权泄露他人资料 |
| top-k | 先 k=5~8,后续可加 rerank | k 太大塞爆上下文,太小漏信息 |
| 上下文拼装 | context + 最近 N 轮历史,控制总 token | 超模型上下文窗口要截断/压缩 |
| 引用 | 回答里标注来源 chunk,存 `citationsJson` | RAG 的可信度卖点 |
| 流式 | chat 用 stream,逐 token 经 SSE 推 | 反向代理要关 buffering |
| 历史 | 取该 conversation 最近 N 条 message 拼进 prompt | 长对话需摘要式压缩 |

### 2.3 检索封装(强制隔离)

```ts
// shared/rag/retriever.ts —— 唯一的检索入口,user_id 过滤写死在这里
async retrieve(userId: bigint, query: string, k = 6) {
  const vec = await this.llm.embed(query);
  const hits = await this.qdrant.search('knowledge_chunks', {
    vector: vec,
    limit: k,
    filter: { must: [{ key: 'user_id', match: { value: Number(userId) } }] }, // ★ 强制
  });
  const chunkIds = hits.map(h => h.payload.chunk_id);
  return this.prisma.documentChunk.findMany({ where: { id: { in: chunkIds } } });
}
```

> 禁止任何 controller 自己拼 Qdrant filter——所有检索必须走这个方法,从代码结构上杜绝越权。

## 3. Agent 工具编排流程(进阶,可放后期)

当任务超出"单轮问答"(如"把我上传的论文整理成综述"),用 **自研工具调用循环**(基于 OpenAI tool calling 协议)编排多步。**慢、多步,走 worker + SSE 进度**(同摄取)。**不引入 LangGraph 等 agent 框架**,理由见 01 §5。

### 3.1 工具集(给 agent 的能力)

| 工具(普通函数,JSON in/out) | 作用 |
|---|---|
| `retrieve_knowledge(query)` | 从用户知识库检索(复用 §2.3,自动带 `user_id` 过滤) |
| `list_documents()` | 列出当前用户已上传文档 |
| `summarize_document(docId)` | 对单文档生成摘要(复用现有摘要逻辑) |
| `organize(criteria)` | 按主题聚类(复用现有聚类逻辑) |

工具注册到一个 `toolRegistry`,每个工具声明 OpenAI 兼容的 `tool schema`(name + parameters JSON Schema)。LLM 通过 tool calling 决定调用哪个、传什么参数。

### 3.2 自研循环时序

```
[Web]    POST /agent/tasks {task} → 建 Run(kind=agent_task) → 入队 → 立即返回 runId
[Worker] 取出 job,开始自研循环:

         ┌──────────────────────────────────────────────────────────────┐
         │  loop:                                                        │
         │    ① openai.chat.completions.create({                          │
         │        model, messages: history,                              │
         │        tools: toolRegistry.schemas,                           │
         │      })                                ─────────► Ollama/API   │
         │    ② 若 resp.message.tool_calls 非空:                          │
         │        for each call:                                         │
         │          • emit RunEvent(tool_called)  ─► Redis pub/sub → SSE │
         │          • result = toolRegistry.run(name, args, ctx)         │
         │          • emit RunEvent(tool_result)  ─► SSE                 │
         │          • history.push({ role: 'tool', ..., content: result })│
         │        continue                                               │
         │    ③ 否则(LLM 给出最终回答):                                   │
         │        emit RunEvent(final_answer) → SSE                      │
         │        Run→completed                                          │
         │        break                                                  │
         └──────────────────────────────────────────────────────────────┘
```

约 **200 行可落地**(循环 + 工具注册表 + history 拼装 + 事件发射)。详细骨架见 01 §5.3。

### 3.3 自研循环 vs LangGraph 的取舍(锁定)

| | 自研循环(选) | LangGraph(不选) |
|---|---|---|
| 代码量 | ~200 行清晰 TS | 引一个框架 |
| 与现有 run-engine 配合 | **天然**:每步直接 `runEngine.emit()` 落事件 + SSE | LangGraph 有自己的 checkpoint,会和你的事件溯源**打架** |
| 调试 | 直接读 console + 数据库事件 | 穿透框架抽象,要懂 LangGraph 内部 |
| 状态扩展 | 改循环加分支即可 | 改图节点 |
| 适合范围 | 简单循环(本项目够) | 复杂图结构(分支/合并/并行节点)|

**回头加框架的触发条件**:agent 流程演化成多分支、并行、人工介入等复杂图结构时——单独引 LangGraph,不引 LangChain core(L1/L2/L3 仍不变)。M5 范围内不会触发。

### 3.4 与 run-engine 的关系

- agent 循环**就是一个 `Run`**(`kind=agent_task`)。
- 每一步(`tool_called` / `tool_result` / `final_answer`)都是 `RunEvent`,带 `sequence_no`,落库 + Redis pub/sub 推 SSE。
- 客户端断线重连可从数据库**完整回放** agent 推理轨迹——这是相对 LangGraph "黑盒 checkpoint" 的差异化优势,也是你的简历亮点。

## 4. 三条流程的复用/新增一览

| 流程 | 复用现有资产 | 新增能力 |
|---|---|---|
| 摄取 | run-engine(FSM/事件/SSE)、资源采集骨架 | 上传、解析(loader)、切分、embedding、Qdrant |
| RAG 对话 | Ollama provider、SSE | conversations/messages、检索、流式生成、引用 |
| agent 编排 | run-engine、摘要/聚类逻辑、Ollama | LangGraph 工具循环 |
