## 一、背景与动机:为什么连接层是 IM 最硬的可扩展性问题

### 1.1 先看横向扩展时会真实撞上的墙

```
单台 server 扛不住了,加一台变两副本,前面挂个负载均衡。一般 Web 服务这样就横向扩展了。
但 IM 这么搞会立刻坏掉:
  用户 B 的长连接握手时被 LB 分到了【副本 A】,连接一直挂在副本 A 上。
  现在 C 给 B 发消息,C 的请求被 LB 分到了【副本 B】。
  副本 B 手里有消息,但 B 的连接在副本 A 上 —— 副本 B 根本不认识那条连接,消息发不出去。
→ "连在 A 的用户,消息从 B 怎么送达?" 这条消息就卡死了。
```

普通 HTTP 服务无状态、随便哪个副本处理都行,所以横向扩展很容易。**IM 的长连接是有状态的**(连接固定挂在某个副本上),这让"加机器"从一件简单的事变成一道难题。这就是连接层可扩展性的核心动机。

### 1.2 本质:有状态长连接 与 无状态横向扩展 的矛盾

- 横向扩展的前提是**无状态**(任意副本可处理任意请求)。
- 但长连接**天生有状态**:一条 TCP/WebSocket 连接物理上绑定在某台机器的某个 fd 上,**不能被另一台机器直接操作**。
- 矛盾的化解思路:**把"连接在哪"这个状态从进程内存挪到共享存储(Redis),让副本之间能互相"找到并代投"对方持有的连接**。这就是 backplane(副本间背板)。**核心 insight:连接本身不能共享,但"连接的位置信息"可以共享;副本 B 找不到连接就把消息"转交"给持有连接的副本 A。**

### 1.3 为什么这是面试的可扩展性深挖点

- 它把"你会不会做高并发服务"问到底:长连接保活、C10K/C10M、副本扇出、背压。
- 它是 09/10 里 **Node→Go 成长线**的落点:海量长连接正是 Go 的主场(goroutine + channel),也是"把接入层抽成 Go 网关"唯一真实成立的瓶颈场景。
- **但它是"加深项"不是地基**:纯 Node + Redis pub/sub 就能把多副本可达讲完整(P2);Go 网关是 P5,讲的是"更高并发承载"。两层叙事都要能独立成立。

## 二、概念与底层原理

### 2.1 为什么 IM 用长连接,不用短轮询

| 方案 | 机制 | 问题 |
|---|---|---|
| 短轮询(polling) | 客户端每隔几秒问"有新消息吗" | 延迟高 + 空轮询浪费;消息越实时越浪费 |
| 长轮询(long-polling) | 请求挂住直到有消息或超时 | 比短轮询好,但每条消息一次请求重建,开销大 |
| **WebSocket 长连接(选)** | 一条 TCP 连接双向常开,服务器可主动推 | 实时、省握手;**代价是连接有状态、要保活、要管海量 fd** |

IM 要"服务器主动、实时推送",**长连接是唯一合适选择**。代价就是本文要解决的那些问题(保活、扩展)。

### 2.2 保活:为什么 TCP keepalive 不够,要应用层心跳

- TCP 自带 keepalive,但**默认探测间隔以小时计**(Linux 默认 7200s),且**探测不了应用层假死**(进程卡住但 TCP 还在)。
- 还有**中间设备**:NAT/防火墙/LB 会在连接空闲几十秒~几分钟后**静默回收**映射,两端却都以为连接还在(**半开连接**)。
- 所以必须**应用层心跳**:客户端定期发 ping,服务器回 pong;一段时间收不到 → 判定连接死,主动断开并触发重连。心跳间隔要短于中间设备的空闲回收时间(通常几十秒)。

### 2.3 重连:断线后怎么不丢、不重

- 连接是不可靠的(网络抖动随时断)。重连后**断连期间的消息**靠 [13](13-深度-下行可靠-离线增量补拉.md) 的机制补:`lastSyncedSeq` + `/sync` + gap 检测。
- **连接层只负责"重新建立连接 + 触发补拉"**,可靠性由 seq/游标保证。这是分层:连接层管"通道",可靠性层管"内容不丢"。
- SSE 场景用 `Last-Event-ID` 头按事件 id 续传(agent-server 已有);WebSocket 用 seq 游标续传——**同一种"按序号断点续传"范式**(11 §2.3、13 §1.3)。

### 2.4 backplane:副本间的"消息背板"

解决 §1.1 的核心组件。**backplane = 让多个副本之间能互相转发下行消息的通道**:

```
副本 B 要给"连在副本 A 的 B 设备"发消息:
  副本 B → publish 到 Redis 频道 → 副本 A 订阅到 → 副本 A 用本地 fd 投递给连接
```

- 实现常用 **Redis pub/sub**(Socket.io 的 `socket.io-redis-adapter` 即此)。
- **它是"事件落库 + pub/sub 实时扇出 + 断线按 seq 回放"三件套的一环**——pub/sub 负责实时,落库 + 回放负责不丢(因为 pub/sub 会丢,§4.2)。这套和 agent-server 的 SSE backplane(02 §6.1)**完全同构**,直接复用。

### 2.5 C10K / C10M:海量连接的底层约束

- **C10K 问题**:单机扛 1 万并发连接。早年用"一连接一线程"的模型在这就崩了(线程栈内存 + 上下文切换)。解法是 **I/O 多路复用(epoll/kqueue)** + 事件驱动——一个线程管海量连接。Node 的 libuv、Go 的 netpoller 底层都是 epoll。
- **C10M**:单机百万连接,瓶颈转移到内存(每连接的缓冲区)、fd 上限、协议栈。
- 关联到选型:**Node 单线程事件循环**能扛不少连接,但 CPU 密集(大量序列化/扇出)时单线程会成瓶颈;**Go 的 goroutine(轻量,栈 2KB 起)+ netpoller** 在"海量连接 + 每连接少量计算"时调度更省、能用多核——这是 §4.3"为什么接入层用 Go"的底层依据。

## 三、现状的坑

- **单实例 Socket.io,连接态在进程内存**:`io.to(socketId)` / 内存 `Map` 只在本副本有效。一旦多副本,§1.1 的"跨副本送不达"立刻发生。
- **没有 Redis adapter / backplane**:副本之间无法互相代投。
- **保活/重连**:Socket.io 自带一定心跳与重连,但**重连后的"按 seq 补拉"没接上**(13 的游标缺失)→ 重连了也补不齐。
- **无背压/限流**:下行突发时没有保护,易被打爆。

## 四、各方案分析

### 4.1 副本间通信:用什么做 backplane

| 方案 | 机制 | 取舍 |
|---|---|---|
| **Redis pub/sub(选,P2)** | 副本订阅频道,发布即扇出 | 简单、低延迟、与现有 Redis 复用;**尽力而为会丢**(§4.2),靠落库+回放兜底 |
| 消息队列(Kafka/NATS) | 持久化的发布订阅 | 不丢、可回放;但重、运维成本高,IM 实时下行用它偏重 |
| 节点间 gossip / 直连 | 副本互相直连广播 | 副本数多时 N² 连接,复杂 |

**选 Redis pub/sub**:实时扇出要的是低延迟,可靠性已由"落库 + seq 回放"在另一层保证,**不需要 backplane 自己持久化**。这正是"职责分层"的体现:pub/sub 管快,DB+seq 管不丢。

### 4.2 为什么 pub/sub 必须配"落库 + 回放"双保险

- Redis pub/sub 是 **fire-and-forget**:发布时没有订阅者在线,消息**直接丢弃**,不重发、不持久化。副本重启/订阅抖动的瞬间,扇出的消息就没了。
- 所以**绝不能把 pub/sub 当可靠投递**。正确结构:
  ```
  消息落库(权威,不丢) ──┬─► pub/sub 实时扇出(快,可能丢)
                          └─► 接收方按 seq 发现 gap → /sync 补拉(兜底,补回 pub/sub 丢的)
  ```
- **双保险**:pub/sub 丢了,gap 检测 + 补拉一定补回。这条是 09/10 反复强调的"同 agent-server 02 §6.1"。**面试问"pub/sub 不是会丢吗"时,这就是答案。**

### 4.3 Go 接入网关:为什么、何时、怎么无状态

**为什么 Go(§2.5 的落地)**:海量长连接 + 每连接少量计算,goroutine(2KB 起栈、M:N 调度到多核)+ channel 管理连接天然契合;Node 单线程在扇出/编解码 CPU 密集时会成瓶颈。

**何时上(触发信号)**:长连接数 / 下行扇出成为**真瓶颈**时(Node 副本 CPU 打满、连接数撞单进程上限)。**没有这个信号就不上**——P5 加深项,时间紧直接砍(09 §8.1)。

**怎么保证无状态(INV-7)**:
- 网关**不存任何业务态**,连接态(用户→连接映射)全在 Redis 注册表(15)。
- 网关只做:长连接生命周期、心跳、JWT 校验(验 our-chat 签发的 token)、下行扇出(订阅 pub/sub)、限流背压。
- **业务仍在 Node**:网关把上行透传给 Node(gRPC/内部 HTTP),Node 处理后下行经 pub/sub 回到网关。这样网关可任意加副本(无状态),Node 业务不动。

### 4.4 背压与限流:不被下行打爆

- **背压(backpressure)**:下行速度 > 客户端消费速度时,数据在服务端缓冲堆积 → 内存涨 → OOM。必须有背压:缓冲满了就**降速/丢弃低优先级/断开慢消费者**,而不是无限缓冲。
- **限流(rate limit)**:上行用令牌桶限制单连接/单用户的消息速率,防刷防滥用。
- 选错(无背压)的代价:一个慢客户端 + 高速下行,服务端缓冲无限涨,**一个连接拖垮整个进程**。Go 网关用 channel 有界缓冲天然表达背压;Node 要显式管 `socket.bufferedAmount`。

## 五、核心机制

### 5.1 pub/sub backplane 下行流程(P2,纯 Node)

```ts
// 发送方副本:消息落库后(11/12),实时扇出
async function fanout(conv: string, payload: { seq: bigint; serverMsgId: bigint; ... }) {
  const members = await getOnlineMembers(conv);              // 查注册表得在线成员设备(15)
  for (const d of members) {
    if (d.replica === SELF) localEmit(d.socketId, payload);  // 本副本直接投
    else redis.publish(`replica:${d.replica}`, JSON.stringify({ socketId: d.socketId, payload })); // 跨副本
  }
}
// 每个副本启动时订阅自己的频道
redis.subscribe(`replica:${SELF}`, (msg) => {
  const { socketId, payload } = JSON.parse(msg);
  localEmit(socketId, payload);                              // 用本地 fd 投递
});
```

> 简化版可直接用 `socket.io-redis-adapter`(它把 `io.to(room).emit` 透明地跨副本广播)。自己实现一版能讲清机制,用 adapter 省事——面试讲"我理解它内部是 pub/sub 代投"即可。

### 5.2 心跳协议

```
客户端每 25s 发 ping → 服务器回 pong,并刷新该连接的注册表 TTL(15 §5.4)
服务器侧:60s 内没收到任何 ping → 判定连接死 → 关闭 + 从注册表摘除
客户端侧:发了 ping 但 N 秒没 pong → 主动断开 → 触发重连 + catchUp(13)
```

间隔取舍:太长则中间设备回收/假死发现慢;太短则心跳流量与唤醒耗电(移动端尤其)。常见 25~30s。

### 5.3 重连后按 seq 回放

```ts
socket.on('reconnect', async () => {
  await reauth();                                            // 重新校验 JWT
  for (const conv of activeConversations()) await catchUp(conv);  // 按 lastSyncedSeq 补拉(13 §6.3)
});
```

连接层只触发,补齐逻辑全在 13。**连接层与可靠性层解耦**:换 Go 网关也不改这套。

### 5.4 Go 网关连接管理(P5 概念)

```
每条连接一个 goroutine 读、一个 goroutine 写(或读写复用),通过有界 channel 解耦:
  readLoop  ──► 上行 channel ──► 透传给 Node 业务(gRPC)
  下行 channel ──► writeLoop ──► 客户端     (channel 满 = 背压,§4.4)
连接态不在 goroutine 里,在 Redis;网关重启,客户端重连重建,业务不受影响(无状态)。
```

## 六、关键实现要点(部署)

- **反向代理**:Nginx/Ingress 必须为 WebSocket 配 `Upgrade`/`Connection` 头转发,且**关闭对 SSE/WS 的 buffering**(`proxy_buffering off`)——否则流式/实时全被缓冲掉(§七坑1)。
- **粘性会话**:用 Redis adapter 后**不需要**会话粘性(任意副本可代投);若没有 backplane 又多副本,才被迫开 sticky session(下策)。
- **Redis 高可用**:注册表 + pub/sub 都依赖 Redis,生产要主从/哨兵或集群,避免单点。

## 七、踩坑案例(真实可复现)

### 坑 1:反代 buffering 把流式/实时掐住
**复现**:Nginx 默认 `proxy_buffering on`,WebSocket 升级失败或 SSE 消息被攒着不下发,前端"卡住不动"。**根因**:代理缓冲。**修复**:WS 配 Upgrade 头,SSE/WS 关 buffering(§六、对应 09 §8.3)。

### 坑 2:把 pub/sub 当可靠投递 → 偶发丢消息
**复现**:副本重启瞬间扇出的消息没人订阅,直接丢;客户端没补拉机制 → 永久缺。**根因**:pub/sub fire-and-forget。**修复**:落库 + gap 补拉双保险(§4.2)。

### 坑 3:多副本但连接态在内存 → 跨副本送不达
**复现**:加到两副本,A 副本用户收不到 B 副本发起的消息。**根因**:没共享注册表 + 无 backplane。**修复**:注册表入 Redis + pub/sub 代投(§1.1、§5.1)。

### 坑 4:只靠 TCP keepalive → 假死连接堆积
**复现**:移动端切后台,NAT 回收了映射,服务端 TCP 仍显示 ESTABLISHED,一直往死连接推。**根因**:无应用层心跳。**修复**:应用层 ping/pong + TTL 摘除(§2.2、§5.2)。

### 坑 5:无背压,慢客户端拖垮进程
**复现**:一个弱网客户端 + 高频下行,服务端发送缓冲无限堆积,内存涨到 OOM,**拖垮整个副本上所有连接**。**根因**:无背压。**修复**:有界缓冲 + 慢消费者降速/断开(§4.4)。

### 坑 6:Go 网关 goroutine 无上限 → fd/内存爆
**复现**:连接暴涨,每连接起多个无节制 goroutine,fd 撞上限、内存爆。**根因**:无连接数上限/无背压。**修复**:连接数配额 + 有界 channel + fd ulimit 调优(§4.3、§4.4)。

## 八、Web vs Native 视角

| 维度 | Web | Flutter | 原生 Swift |
|---|---|---|---|
| 长连接载体 | WebSocket(浏览器 API) | WebSocket / socket_io_client | URLSessionWebSocketTask / Starscream |
| 后台保活 | **标签页后台基本无法维持连接**,息屏/切走即断 | 系统挂起 App,连接断 | 系统挂起,连接断;靠 APNs 唤醒 |
| 重连触发 | `online` / `visibilitychange` | 前台恢复 | `didBecomeActive` |
| 心跳定时器 | 后台被节流,不可靠 → 回前台立即重连 + catchUp | 恢复后重连 | 恢复后重连 |
| 离线消息送达 | 无后台进程,**纯靠回前台补拉**(13 §八) | 推送通道唤醒 + 补拉 | APNs 唤醒 + 补拉 |

> **结构性差异**:浏览器**没有常驻后台**,Web 的"长连接"只在标签页前台活着,因此 Web 端的实时性更依赖"回前台快速重连 + seq 补拉";原生有系统推送(APNs/FCM)唤醒,能在后台收提醒再补拉。**所以连接层设计必须让"实时推"和"补拉"都成立——Web 偏补拉、原生偏推送唤醒,但底层都收敛到同一套 seq 续传。**
