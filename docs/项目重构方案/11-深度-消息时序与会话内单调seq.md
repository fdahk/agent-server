# 11 · 深度 · 消息时序与会话内单调 seq

> 对应 [10 目标架构](10-目标架构.md) 的 **INV-1**(同一会话内 seq 严格单调、不跳号、不重号)。
> 这是消息可靠性核心的**地基**:seq 不对,后面的 ack 去重、离线补拉、gap 检测、多端一致全部建在沙子上。
> 本文是深度文档的**样板**,深度标准:**先把"为什么必须考虑这个点"讲到第一性原理,再讲底层机制,再逐方案拆解取舍**,最后落到实现/踩坑/面试。

---

## 一、背景与动机:这个点为什么存在

### 1.1 先看一个会真实发生的 bug

不引入 seq、就按现状的 `timestamp` 排序,下面这个场景**几乎必然发生**:

```
群里三个人,网络都正常,时间都在同一秒内:
  Alice 发:  "在吗"            -> 服务器记 timestamp = 14:30:05
  Bob   发:  "在的,怎么了"     -> 服务器记 timestamp = 14:30:05
  Alice 发:  "帮我看个 bug"     -> 服务器记 timestamp = 14:30:05
```

三条消息时间戳**字节级相同**(现状精度是秒,见 §3.1)。当客户端按 `ORDER BY timestamp DESC` 拉取时,数据库对"排序键完全相同"的行**不保证任何固定顺序**(SQL 标准:相同排序键的相对顺序 undefined)。于是可能渲染成:

```
Bob:   在的,怎么了        ← Bob 在回答一句还没出现的问题
Alice: 帮我看个 bug
Alice: 在吗
```

**这不是"偶尔抖一下"**:同一秒内的任意并发消息都会中招;甚至同一个用户连发两条,刷新一次顺序就可能变。聊天记录的顺序错乱是 IM 最致命的体验 bug——它直接让产品**不可信**。

这就是"为什么必须考虑时序"的最直接答案:**消息的顺序是 IM 的正确性底线,而现状根本没有一个可靠的顺序来源。**

### 1.2 再往下一层:这其实是"分布式系统的事件定序"问题

为什么一个聊天软件会有"定序"这么硬核的问题?因为 **IM 本质上是一个分布式系统**:

```
多个客户端(各自有时钟)  ──┐
                          ├──► 一条消息流,需要所有参与者看到"同一个顺序"
多个服务器副本(各自有时钟)─┘
```

分布式系统里,"两个事件谁先谁后"是一个**没有免费答案**的问题——这是 Leslie Lamport 1978 年那篇《Time, Clocks, and the Ordering of Events in a Distributed System》奠基的领域(分布式系统被引用最多的论文之一)。它的核心结论用在 IM 上就是:

> **你不能靠各节点的物理时钟来给跨节点的事件定序**,因为没有任何机制保证不同机器的时钟一致。要得到一个所有人都认同的顺序,必须引入一个**逻辑顺序**,而不是物理时间。

seq 就是这个逻辑顺序。所以"加 seq"不是一个工程小优化,而是**把一个分布式定序问题,收敛成一个有确定答案的问题**。能在面试里把"消息排序"上升到"这是分布式事件定序"这一层,就已经和"我用了个自增字段"拉开了档次。

### 1.3 为什么定序是整个可靠性体系的地基(下游依赖链)

时序不是孤立的一个点,它是后面所有可靠性机制的**前置条件**。把依赖链画出来就明白为什么必须先做它:

```
            ┌─────────────────────────────────────────────┐
            │  会话内单调 seq(本文,INV-1)               │  ← 地基
            └───────────────┬─────────────────────────────┘
              ┌─────────────┼──────────────┬───────────────┐
              ▼             ▼              ▼               ▼
        排序(顺序对)  gap 检测       离线增量补拉      已读位点
        12/13 依赖     "我漏了吗"      WHERE seq>X       lastReadSeq
                       靠 seq 连续      (13 文档)        (15 文档)
                       (13 文档)
```

- **没有连续的 seq,就无法做 gap 检测**:客户端怎么知道自己漏了消息?只有当序号是"会话内连续 1,2,3…"时,收到 5 之后直接来 8,才能**确定**漏了 6、7。
- **没有单调数值游标,就无法做增量补拉**:`WHERE seq > lastSyncedSeq` 这种范围查询,要求游标可比较、单调。
- **没有可比较游标,已读位点也做不了**:"我读到哪了"本质也是一个 seq 游标。

所以 09/10 把它列为 P1 的**第一**件事,且其它深度点都标注"依赖本文"。**先做时序,不是因为它最显眼,而是因为它是别的东西的前提。**

### 1.4 底层原理:物理时钟为什么不可信(分布式系统第一课)

§1.2 说"不能靠物理时钟",这里必须讲清**到底是物理时钟的哪些性质让它不可用**——这是面试深挖"为什么不用时间戳"时,能不能答到第三层的关键。

物理时钟有两种,都不能直接拿来给消息定序:

**① 墙上时钟(wall-clock,`Date.now()` / `now()` 拿到的)**——会"跳":
- **时钟偏移(clock skew)**:两台服务器的时钟在同一时刻读数不同。机房里 NTP 同步后偏差通常几毫秒到几十毫秒,**但这已经足够让"B 机器后发的消息"时间戳早于"A 机器先发的消息"**。
- **时钟漂移(clock drift)**:晶振温度等原因,时钟走快/走慢,两次同步之间持续累积偏差。
- **NTP 回拨**:NTP 校时发现本地快了,会**把时钟往回调**。于是"后发生的事件"可能拿到**更小**的时间戳——时间倒流。靠它排序直接错乱。
- **闰秒(leap second)**:UTC 偶尔插入一秒,某些系统会让同一秒出现两次,或让时钟暂停。

**② 单调时钟(monotonic clock,如 `process.hrtime` / `CLOCK_MONOTONIC`)**——不会回拨,但:
- **它只在单台机器内单调**,不同机器的单调时钟没有可比的零点,**跨机器无意义**。
- 它度量的是"开机以来过了多久",不是"现在几点",**无法跨进程/重启对齐**。

**结论**:墙上时钟会跳(skew/drift/回拨/闰秒),单调时钟跨不了机器。**没有任何物理时钟能给"跨客户端、跨副本的消息"提供一个所有人都认同的全序。** 这就是为什么必须用逻辑顺序(seq)。

> 面试杀招:被问"为什么不用时间戳",不要只答"同毫秒会冲突"(那只是表象),要答"物理时钟在分布式下有 skew/drift/回拨,根本不存在一个可信的全局物理时间;定序必须用逻辑时钟"。这是第三层。

## 二、概念与底层原理

### 2.1 三种"顺序",只有一种能用

| 顺序 | 定义 | 谁产生 | 能排序吗 | 为什么 |
|---|---|---|---|---|
| 发送时间(send time) | 用户点发送那一刻 | **客户端**时钟 | ❌ | 客户端时钟可错/可篡改,多端不一致 |
| 服务器接收时间(recv time) | 服务器收到的物理时间 | **服务器**时钟 | ⚠️ | 单机近似可用;多副本有 skew/回拨(§1.4) |
| 逻辑顺序(logical seq) | 服务器分配的会话内递增序号 | 服务器**逻辑分配** | ✅ | 与物理时间脱钩,人为制造全序 |

### 2.2 逻辑时钟 vs 物理时钟,seq 是哪一种

- **物理时钟**:试图反映"真实时间",受硬件/同步影响,不可信(§1.4)。
- **逻辑时钟(Lamport)**:不关心"几点",只关心"谁先谁后"。给每个事件一个递增计数器,**计数器大的就是后发生的**。seq 就是逻辑时钟的一种。

Lamport 逻辑时钟有两种形态,要能区分:

| 形态 | 能力 | IM 用哪个 |
|---|---|---|
| **标量逻辑时钟(Lamport clock)** | 给出一个**全序**(任意两个事件可比) | ✅ seq 属于这类 |
| **向量时钟(vector clock)** | 给出**偏序**(只能判断有因果关系的事件,并发事件不可比) | ❌ 偏序不够,见 §4.6 |

IM 一个会话里,用户要看到一条**线性的聊天记录**——必须是**全序**(任意两条消息有确定先后)。向量时钟只能告诉你"这两条没有因果关系",但聊天记录不能渲染成"这两条顺序未定"。所以 IM 用**标量序号(seq)拿全序**,不用向量时钟。

### 2.3 核心 insight:把会话变成"单写者日志"

seq 能简单可靠的根本原因,是一个架构 insight:

> **让一个会话的所有消息都流经唯一一个序列化点(single point of serialization),在那里一个一个分配序号。**

一旦所有写入被"漏斗"成单点串行,全序就**免费得到了**——因为单点天然知道"我先处理了谁、后处理了谁"。这等价于把一个会话变成一条 **单写者(single-writer)的追加日志**。

这个模式在系统领域到处都是,理解它能让你瞬间看懂很多东西:

| 系统 | "单写者日志"的体现 | 序号 |
|---|---|---|
| 数据库 WAL(预写日志) | 所有修改先顺序写一条日志 | LSN(Log Sequence Number) |
| Kafka 分区 | 一个 partition 内消息顺序追加 | offset |
| Raft 共识 | leader 把命令顺序写进 replicated log | log index |
| **本项目** | 一个 conversation 的消息顺序追加 | **seq** |

**类比记忆**:`seq ≈ 会话内的 Kafka offset`,`会话 ≈ partition`。本项目做的事,就是"给每个会话维护一条单写者日志,序号是 seq"。面试时这个类比极有杀伤力。

### 2.4 为什么是 per-conversation(会话内),不是全局

限定词是 **per-conversation**:seq 在一个会话内 1,2,3… 递增,不同会话各自独立、互不相关。

为什么不全局单调?

- **排序/gap/补拉都只在单会话内进行**,跨会话从不需要比较"会话 A 的第 5 条和会话 B 的第 5 条谁先"——这种比较没有业务意义。
- **全局单调 = 所有会话争抢同一个计数器**,这个计数器成为全系统写入热点,白白制造瓶颈。
- 单写者日志的"写者"粒度选在**会话**这一层,正好:并发隔离在会话之间(不同会话完全并行),串行只发生在会话内部(本来也必须串行才能定序)。

**per-conversation 是"够用的最小单调性"**,全局单调是过度设计——主流 IM 几乎都选会话/用户维度,不选全局。

## 三、现状的坑(基于真实 schema)

现状权威:`our-chat/server/prisma/schema.prisma`。三个真实的洞:

### 3.1 洞一:用秒级 `timestamp` 排序

```prisma
model Message {
  id        BigInt   @id @default(autoincrement())
  timestamp DateTime @default(now()) @db.Timestamptz(0)   // ← (0) = 0 位小数 = 秒级精度
  @@index([conversationId, timestamp(sort: Desc)], map: "idx_messages_conv_ts")
}
```

- `@db.Timestamptz(0)` 的 `(0)` 是**小数秒位数**,即**精度只到秒**。同一秒内所有消息时间戳完全相同 → 排序退化为未定义顺序(§1.1 的 bug)。
- `now()` 取事务开始时间,**同一事务/同一瞬间的多条消息可拿到同值**。
- 即使提到毫秒精度,**多副本时钟仍有 skew/回拨**(§1.4),跨副本时序不可靠。

### 3.2 洞二:已有全局自增 `id`,但它替代不了 seq

会有人问:"`Message.id` 是 `BigInt autoincrement`,本来就单调,直接拿它排序不行?"——**不行,且必须讲清为什么**:

| 维度 | 全局 autoincrement `id` | 会话内 seq |
|---|---|---|
| 单调范围 | 全局(所有会话共用一条序列) | 会话内 |
| 会话内是否连续 | ❌ **有大量空洞**:会话 A 相邻两条消息 id 可能是 100、580,中间夹着别的会话 | ✅ 连续 1,2,3… |
| 能否 gap 检测 | ❌ 收到 id=100 后来 id=580,**无法判断**中间是漏了还是只是别的会话占用了号 | ✅ 收到 5 后来 8,**确定**漏了 6、7 |
| 补拉游标语义 | ⚠️ `id>X` 的 X 是全局值,客户端要存全局位点,跨会话语义混乱 | ✅ 每会话独立游标,干净 |

**根因**:全局自增是**全系统单写者日志**的序号,它在"全局"这个粒度连续,但在"单会话"这个粒度**必然有空洞**。而 gap 检测要的是"单会话内连续"。粒度不匹配,所以全局 id 做不了 gap 检测——这是必须引入 per-conversation seq 的硬理由。

### 3.3 洞三:已读位点是字符串 messageId,无法做范围查询

```prisma
model UserConversation {
  lastReadMessageId String?  @map("last_read_message_id") @db.VarChar(50)  // 字符串,沿用 MongoDB ObjectId 时代
}
```

- 类型上 `Message.id` 是 `BigInt`,而 `lastReadMessageId` 是 `VarChar(50)`——**对不上**(指向旧 MessageRef.id)。
- 致命的是:**字符串 id 无法做 `WHERE seq > X` 的范围查询**。要实现"拉我没读/没同步的消息",必须有一个**可比较、单调的数值游标**。这正是 13(离线增量同步)被卡住的根因。

## 四、各方案深度分析(逐个拆机制 + 失败模式 + 适用边界)

这一节是"为什么选 A"的全部含金量。每个方案讲清**怎么工作、怎么坏、什么时候才该用**,而不是甩个表。

### 4.1 方案 A:会话行上的 DB 计数器(本项目选)

**机制**:在 `conversations` 表加一列 `next_seq`,发消息时在**同一事务**内:

```sql
UPDATE conversations SET next_seq = next_seq + 1 WHERE id = :conv RETURNING next_seq;
-- 用拿到的 next_seq 作为本条消息的 seq,紧接着 INSERT
```

`UPDATE` 会对该会话行加**行级排他锁**,并发的两条消息被串行化(机制细节见 §五)。

- **正确性**:行锁保证不重号;同事务保证不跳号(回滚则 +1 也回滚)。强一致,无需对账。
- **失败模式**:几乎没有正确性失败模式;唯一"问题"是**同一会话的写入被串行**(吞吐受单行锁限制)。
- **适用边界**:单会话写入 QPS 在"正常人类聊天"量级(哪怕千人群)完全够。**校招项目规模 100% 落在此区间。**
- **代价**:同会话高频写时,行锁是瓶颈——但这要到弹幕级才显现。

### 4.2 方案 B:Redis 段式预分配 + 异步落库

**机制**:用 Redis `INCR conv:{id}:seq` 拿号(内存自增,极快),消息再异步批量落库。为减少 Redis 往返,可"段式"分配——一次 `INCRBY 100` 拿一段号在内存里发。

- **为什么更快**:拿号从"DB 行锁事务"变成"内存自增",同会话写入不再被 DB 行锁串行卡住。
- **失败模式(关键)**:
  - **跳号**:Redis 发了 seq=5,消息落库前进程崩 → seq=5 永远没有对应消息 → **会话内永久空洞** → gap 检测误判"漏消息",触发无意义补拉,甚至客户端永远在等一条不存在的 5。
  - **丢号 / 倒退**:Redis 自身重启且没持久化(或 AOF 没刷盘),计数器回退 → **重号** → 两条消息同 seq → 覆盖/错乱。
  - 段式预分配让上面两个问题更严重:崩一次丢一整段。
- **要补的工程**:Redis 持久化策略(AOF everysec 仍可能丢)、**以 DB 为权威的崩溃对账**(对应 10 文档 INV-8)、空洞填补/墓碑机制。一整套额外复杂度。
- **适用边界**:**单个会话**写入高到 DB 行锁扛不住(十万人直播弹幕群)。这时拿"可容忍偶发空洞 + 对账"换吞吐才划算。

### 4.3 方案 C:独立序列号服务(seqsvr,微信式)

**机制**:把"分配 seq"抽成一个**独立服务**。核心优化是**段缓存(section cache)**:序列号服务从持久化存储一次预取一大段号(如 +10000)缓存在内存,之后在内存里飞快分配,**只有用完一段才回存储取下一段并持久化"已分配到哪"**。这样把"每条消息一次存储写"摊薄成"每一万条一次存储写"。

- **为什么微信需要它**:微信是**每个用户**一个递增 seq(用于多端增量同步),用户数十亿、消息量极大,DB 行锁/单 Redis 都扛不住,必须独立服务 + 段缓存 + 高可用(主备/路由)。
- **失败模式**:段缓存崩溃恢复时,会**跳过整段未用的号**(为保证不重号,恢复后从"已持久化的下一段"开始)——即**用"可接受的跳号"换"绝不重号"**。这个取舍正是 seqsvr 设计的精髓。
- **适用边界**:**互联网级规模**。校招项目用它 = 杀鸡用牛刀,而且**讲不出真实压测数据**反而像背架构。
- **本项目的正确姿势**:**不实现,但要能讲**——"我知道微信用 seqsvr + 段缓存把存储写摊薄、用跳号换不重号,但我的规模 DB 行锁就够,过早上 seqsvr 是过度设计"。**能讲清"我为什么不用",比硬抄一个更显判断力。**

### 4.4 方案 D:全局序列 / Snowflake(为什么解决不了)

- **PG 全局 sequence(`nextval`)/ Snowflake ID**:能生成全局单调(或趋势单调)的 id。
- **为什么不行**:它们给的是**全局**序号,在单会话粒度**必然有空洞**(同 §3.2 的全局自增 id)。Snowflake 还只是"趋势递增"(含时间戳位,跨节点甚至可能局部乱序)。**做不了会话内 gap 检测。** 方向错了,不是性能问题。

### 4.5 方案 E:混合逻辑时钟 HLC(为什么 overkill)

- **HLC(Hybrid Logical Clock)**:把物理时间和逻辑计数器揉进一个值,既趋近真实时间又保证单调,CockroachDB 等分布式 DB 用它做跨节点事务定序。
- **为什么 IM 不需要**:HLC 解决的是"**无中心、跨节点**还要近似物理时间"的难题。但 IM 的一个会话**有天然的中心化串行点**(会话行 / seqsvr),根本不需要无中心定序。用 HLC = 为一个你没有的约束(去中心)付复杂度。**适用边界不匹配。**

### 4.6 方案 F:客户端向量时钟(为什么不行)

- **向量时钟**:每个客户端各持一个计数器向量,能判断因果关系。
- **为什么不行**:它给的是**偏序**——并发消息之间"不可比"。但聊天记录必须是**线性全序**(§2.2),不能渲染成"这两条顺序未定"。而且向量随设备数膨胀。**偏序 ≠ 全序,根上就不满足需求。**

### 4.7 决策矩阵

| 方案 | 全序 | 会话内连续(可 gap 检测) | 崩溃不跳/不重 | 复杂度 | 适用规模 |
|---|---|---|---|---|---|
| **A DB 计数器** | ✅ | ✅ | ✅(同事务) | 低 | 中小~大群,**本项目** |
| B Redis 段式 | ✅ | ⚠️(崩溃有空洞) | ❌ 需对账 | 中 | 单会话超高频 |
| C seqsvr 段缓存 | ✅ | ⚠️(跳号换不重号) | ⚠️ 跳号 | 高 | 互联网级 |
| D 全局 seq/Snowflake | ⚠️ | ❌ 有空洞 | — | 低 | 不适用(方向错) |
| E HLC | ✅ | ✅ | ✅ | 高 | 去中心分布式 DB |
| F 向量时钟 | ❌ 偏序 | ❌ | — | 高 | 不适用(只偏序) |

**本项目锁定 A**:正确性最省心(强一致、无对账)、复杂度最低、规模完全够用;B/C 只在出现**单会话写入真瓶颈**时才有 ROI,而那是 P5 级加深。关键能力不是"实现了最快的",而是**能讲清每个方案的机制、失败模式和适用边界,并解释我的规模为什么选 A**。

## 五、核心机制:行锁 + MVCC 怎么保证 INV-1

方案 A 的正确性全靠 PostgreSQL 两个机制,必须能讲到机制级(否则面试一问"凭什么不重号"就卡住)。

### 5.1 行级排他锁怎么把并发 UPDATE 串行化

```sql
UPDATE conversations SET next_seq = next_seq + 1 WHERE id = :conv RETURNING next_seq;
```

PG 执行 `UPDATE` 时,会对命中的那一行加**行级排他锁(`FOR UPDATE` 隐含)**:

```
事务 T1: UPDATE 会话行  → 拿到行锁 → 读 next_seq=4 → 写 5 → (锁持有到 commit)
事务 T2: UPDATE 同一行  → 行锁被 T1 占用,T2 阻塞等待
                          ↓ T1 commit,释放锁
                        → T2 被唤醒,**重新读最新值** next_seq=5 → 写 6
```

关键点(机制级):
- 行锁在**事务提交/回滚时才释放**,不是 UPDATE 语句结束就放。所以"拿号→落库"整个事务期间,别的消息拿不到这行。
- T2 被唤醒后会**重新读取该行的最新版本**(读到 T1 写的 5),不是用它阻塞前的旧快照。**这保证了 T2 拿到 6 而不是又拿 5**——这就是"不重号"的根。
- 不同会话是不同的行,锁互不影响 → **会话间完全并行**,只有**会话内串行**(这正是我们要的)。

> 补充:即使你的事务隔离级别是 `READ COMMITTED`(PG 默认),`UPDATE` 也会"重新读取并锁定"被并发修改的行(PG 的 `EvalPlanQual` 机制),所以不用刻意提到 `SERIALIZABLE`。这点能答出来是加分。

### 5.2 同事务原子性怎么保证"不跳号"

```
BEGIN;
  UPDATE conversations SET next_seq = next_seq + 1 ... RETURNING next_seq;  -- 拿到 5
  INSERT INTO messages (..., seq) VALUES (..., 5);                          -- 落库
COMMIT;   -- 要么 5 既被消耗也被写入;要么整体回滚,5 还回去
```

靠的是 ACID 的 **A(原子性)** 和 **D(持久性)**:
- 若 INSERT 失败 → 事务回滚 → `next_seq` 的 +1 **一并回滚**(回到 4)→ 下一条消息重新拿 5。**没有空洞。**
- 若 COMMIT 成功 → seq=5 与对应消息**同时持久化**。

**反例(拆事务的代价)**:先单独事务 `UPDATE...RETURNING` 提交(next_seq 变 5),再开新事务 INSERT 失败 → 5 被消耗但无消息 → **永久空洞 5** → 客户端 gap 检测误判漏消息。**这就是 §四方案 B 跳号问题的本质,也是"为什么拿号和落库必须同事务"的答案。**

## 六、关键实现

### 6.1 Schema 增量(对应 10 文档 §5.1)

```prisma
model Conversation {
  nextSeq BigInt @default(0) @map("next_seq")   // 该会话 seq 的权威自增源
}

model Message {
  seq         BigInt   @map("seq")                          // 会话内单调序号
  clientMsgId String?  @map("client_msg_id") @db.VarChar(64) // 幂等键(12 文档用,一并加)
  // 排序基准从 timestamp 切到 seq;timestamp 退为展示用,不删
  @@index([conversationId, seq(sort: Desc)], map: "idx_messages_conv_seq")
  @@unique([conversationId, senderId, clientMsgId], map: "uniq_msg_idem")  // 12 幂等去重
}

model UserConversation {
  lastSyncedSeq BigInt @default(0) @map("last_synced_seq")  // 已同步游标(13 用)
  lastReadSeq   BigInt @default(0) @map("last_read_seq")    // 已读游标(15 用)
}
```

### 6.2 发送路径伪代码(Node)

```ts
async function persistMessage(input: {
  conversationId: string; senderId: bigint; content: string; clientMsgId: string;
}) {
  return prisma.$transaction(async (tx) => {
    // ① 幂等查重(12 详述):命中直接返回首次结果,不消耗 seq
    const existed = await tx.message.findUnique({
      where: { uniq_msg_idem: {
        conversationId: input.conversationId, senderId: input.senderId, clientMsgId: input.clientMsgId,
      }},
    });
    if (existed) return { message: existed, deduped: true };

    // ② 原子拿号 + 落库,同一事务(INV-1,§五机制)
    const [{ next_seq }] = await tx.$queryRaw<{ next_seq: bigint }[]>`
      UPDATE conversations SET next_seq = next_seq + 1
      WHERE id = ${input.conversationId} RETURNING next_seq`;
    const message = await tx.message.create({
      data: { ...input, seq: next_seq, status: 'sent' },
    });
    return { message, deduped: false };
  });
}
```

> 唯一约束是去重最终防线:即使 ① 因并发竞态漏判,② 的 INSERT 也会撞 `uniq_msg_idem`,捕获后返回已存在记录(12 展开)。

### 6.3 数据迁移与回填

```sql
-- ① 会话内"时间升序 + id 升序"回填单调 seq(id 兜底:同秒消息按入库顺序,§3.1)
WITH ordered AS (
  SELECT id, row_number() OVER (
           PARTITION BY conversation_id ORDER BY timestamp ASC, id ASC
         ) AS rn
  FROM messages
)
UPDATE messages m SET seq = o.rn FROM ordered o WHERE m.id = o.id;

-- ② next_seq 置为各会话已用最大 seq
UPDATE conversations c
   SET next_seq = COALESCE((SELECT MAX(seq) FROM messages WHERE conversation_id = c.id), 0);

-- ③ 已读游标:把字符串 last_read_message_id 翻译成对应消息的 seq(略)
```

**回填顺序的关键**:用 `timestamp ASC, id ASC` 而非仅 timestamp——现状 timestamp 精度只到秒(§3.1),同秒消息必须用全局递增 `id` 做稳定 tiebreak,否则回填出的 seq 顺序不确定。

## 七、踩坑案例(真实可复现)

### 坑 1:`Timestamptz(0)` 秒级精度 → 同秒乱序
**复现**:两端 100ms 内连发 5 条,按 `(conv, timestamp DESC)` 查,顺序每次刷新都可能不同。**根因**:精度 `(0)` + `now()`。**修复**:切 seq 排序。这正是 09 §P0"改造前 vs 改造后"的最佳演示素材。

### 坑 2:拿号与落库分事务 → 永久空洞
**复现**:把 `UPDATE...RETURNING` 与 `INSERT` 拆两个事务,中间抛异常 → next_seq 涨了但无消息 → 客户端补拉时卡在等一条不存在的 seq。**修复**:同事务(§5.2)。

### 坑 3:灰度期直接删旧 timestamp 索引 → 慢查询
**复现**:迁移时立刻删 `idx_messages_conv_ts`,但仍有按 timestamp 排序的旧查询没切干净 → 全表扫描。**修复**:先加 seq 索引、双跑、切流验证、最后才删旧索引(删索引不可逆,对应 CLAUDE.md 风险动作)。

### 坑 4:回填只按 timestamp → 同秒 seq 顺序错乱
**复现**:回填 `ORDER BY timestamp ASC`(无 id tiebreak),同秒消息 row_number 不确定 → 回填顺序与真实发送顺序不符。**修复**:加 `, id ASC`(§6.3)。

### 坑 5:`BigInt` 的 JS 序列化与精度
**复现**:`JSON.stringify(seq)` 抛 `TypeError: Do not know how to serialize a BigInt`;若图省事 `Number(seq)`,超 `2⁵³` 后**丢精度** → seq 比较错误。**修复**:全链路用 string 传 seq,客户端按 string/BigInt 解析,**绝不转 Number**。

## 八、Web vs Native 视角

排序/gap 逻辑三端同构(都按 seq),差异在**整数精度与后台断连**:

| 维度 | Web(React) | Flutter | 原生 Swift |
|---|---|---|---|
| seq 类型 | JS `number` >2⁵³ 丢精度,**必须 BigInt/string** | Dart `int` 是 64 位,直接装 BIGINT,无精度问题 | `Int64` 直接对应,无精度问题 |
| 本地有序存储 | IndexedDB,seq 建索引 | sqflite/Drift,seq 列索引 | Core Data/SQLite,seq 索引 |
| gap 检测时机 | 收 push 时比对本地 maxSeq | 同 | 同 |
| 进后台断连 | 切走/息屏 WS 可能断,回前台用 lastSyncedSeq 补拉 | 后台被挂起,恢复补拉 | 后台挂起更激进,**必须**靠 seq 游标补拉 |

> **Web 的 seq 精度是真坑**:BIGINT 超 `2⁵³` 后 JS `number` 丢精度致比较错误;Flutter/Swift 原生 64 位整数天然没此问题。这是"同协议三端实现不同"的典型,讲多端一致(15)时可带出。
