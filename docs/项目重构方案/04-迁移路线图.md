# 04 · 迁移路线图

> 先给"保留/移植/丢弃"清单,再给分阶段里程碑。每阶段标注**可复用的现有代码**,避免重造。

---

## 1. 保留 / 移植 / 丢弃清单

### ✅ 移植(逻辑跟语言无关,搬进 Node,别重造)

| 来源 | 目标 | 备注 |
|---|---|---|
| Java FSM `AgentRunStatus` + 流转 | `shared/run-engine` 状态机 | QUEUED→RUNNING→COMPLETED/FAILED |
| Java 事件溯源(agent_run_events + sequence_no) | `RunEvent` + run-engine | 加 Redis pub/sub backplane |
| Java `SseEmitterHub` | NestJS `@Sse()` + Redis 扇出 | 升级为多副本可用 |
| Node 资源采集(目录扫描/URL 抓取/cheerio) | `ingestion` 的"导入来源"子能力 | 现成可用 |
| Node Ollama provider(模型解析/失败降级) | `shared/llm` | 现成可用 |
| Node 各类 prompt(计划/摘要/聚类/报告) | `agent`/enrichment | 现成可用,降级为可选增强 |

### ✅ 保留(数据/约定)

- `users` / `agent_runs` / `agent_run_events` 三表的**结构思路**(用 Prisma 重建,`agent_runs` 泛化为 `Run`)。
- JWT 认证机制(换成 Node 的 passport-jwt 实现,协议不变)。

### ❌ 丢弃

| 丢弃项 | 原因 |
|---|---|
| 整个 `apps/java-server`(Spring/MyBatis/jjwt) | 统一 Node,消除 polyglot |
| Flyway | 换 Prisma Migrate |
| RabbitMQ | 单体无跨服务消息需求,BullMQ 覆盖 |
| `X-Internal-Token` 内部网关调用 | 合并后变进程内调用 |
| `{noop}` 明文种子密码 | 换 bcrypt + 真实注册 |
| "截断 12000 字符"的伪切分 | 换 RecursiveCharacterTextSplitter |
| "只扫服务器本地目录"作为唯一入口 | 换成用户上传为主、目录/URL 为辅 |

## 2. 分阶段里程碑

每个 Mx 都是**可独立验收**的。前期(M0–M4)是 MVP,M5–M6 是加分项。

### M0 · 脚手架与基础设施(地基)

- 新建单一 NestJS 工程(或在现有 node-server 上扩);确立 `modules/` + `shared/` 结构。
- `docker-compose`:postgres + redis + qdrant + ollama(删 mysql/rabbitmq)。
- 接 Prisma(连 postgres)、ioredis、Qdrant client。
- **接 `shared/llm` 的 OpenAI 兼容客户端**(详见 `01` §7):用官方 `openai` SDK,`LLM_BASE_URL/MODEL/API_KEY` 全走环境变量,默认指向本地 Ollama。
- **删除** `apps/java-server`,monorepo 收敛为单应用。
- **验收**:`docker compose up` 全起;`/health` 通;Prisma migrate 建表成功;`shared/llm` 能用 OpenAI SDK 调通 Ollama(返回一次 chat completion + 一次 embedding);切 `.env` 指向某第三方 API 也能跑通(端到端验证抽象层有效)。

### M1 · 认证(用户隔离的入口)

- Prisma 建 `User`;实现 **注册**(bcrypt)+ 登录(签 JWT)。
- NestJS JwtAuthGuard 全局守卫;`@CurrentUser()` 拿 userId。
- **复用**:Java 的 JWT 签发/校验逻辑(移植为 passport-jwt)。
- **验收**:注册→登录→拿 token→访问受保护接口通;无 token 被拒。

### M2 · 运行引擎(复用资产,先搭好进度通道)

- 移植 FSM + 事件溯源到 `shared/run-engine`;Prisma 建 `Run`/`RunEvent`。
- 接 BullMQ;实现 `start:web` / `start:worker` 双入口。
- SSE:`@Sse()` + Redis pub/sub backplane + 断线重连回放。
- **验收**:造一个假 job,worker 跑,前端 SSE 实时收到事件;杀掉重连能回放历史事件;起 2 个 web 副本,事件仍能送达。

### M3 · 文档摄取(核心新能力之一)

- 上传接口(multipart,限大小/类型)+ `Document`/`DocumentChunk` 表。
- 摄取 worker:LangChain loader 解析(PDF/docx/md)→ splitter 切分 → Ollama 批量 embedding → Qdrant upsert + 写 chunk。
- 进度走 M2 的 run-engine;失败重试 + 双写补偿。
- **复用**:资源采集骨架(目录/URL 作为额外来源)。
- **验收**:上传一个 PDF,SSE 看到 parsing→chunking→embedding→ready;Qdrant 里有 points(payload 带 user_id);Postgres 有 chunk 行。

### M4 · RAG 对话(核心新能力之二,MVP 完成点)

- `Conversation`/`Message` 表 + 会话接口。
- `shared/rag` 检索(**强制 user_id 过滤**)→ 拼上下文 + 历史 → Ollama 流式生成 → 逐 token SSE。
- 存 assistant 回答 + citations。
- **验收**:上传文档后提问,得到**基于自己资料**的流式回答 + 引用来源;另一个用户问,检索不到你的资料(隔离验证)。

> **到 M4,"上传资料 → 和资料对话"的 ChatGPT 式 MVP 已经成立。** 后续是加分。

### M5 · Agent 工具编排(进阶,简历亮点)

- **自研工具调用循环**(基于 OpenAI tool calling,详见 01 §5.3 + 03 §3.2),约 200 行。**不引入 LangGraph/任何 agent 框架**。
- 工具注册表:`retrieve_knowledge` / `list_documents` / `summarize_document` / `organize`,每个工具声明 OpenAI 兼容的 tool schema。
- 多步推理走 worker + run-engine 进度;循环每步直接调 `runEngine.emit()` 落 `RunEvent`,Redis pub/sub 推 SSE。
- **复用**:现有摘要/聚类逻辑作为工具实现;run-engine 直接接住事件流,无需新机制。
- **验收**:下达"把我上传的 X 类文档整理成综述",agent 多步完成,全程可在 SSE 看到 `tool_called`/`tool_result`/`final_answer` 轨迹;客户端断线重连能从数据库回放整条推理轨迹。

### M6 · 打磨与可观测(收尾)

- 文档管理接口(列表/删除,删时清 Postgres+Qdrant)。
- enrichment(摄取后自动打标签/归类,供检索过滤)。
- 基础可观测:结构化日志、运行耗时统计、(可选)简单 eval 检索召回。
- **验收**:能管理知识库;有基本的运行指标。

## 3. 依赖与并行关系

```
M0 ──► M1 ──► M2 ──┬──► M3 ──► M4 ──► M5 ──► M6
                   │
        (M2 的 run-engine 是 M3/M5 的进度基础)
```

- **关键路径**:M0→M1→M2→M3→M4(MVP)。
- M2 必须先于 M3/M5(它们的进度依赖 run-engine)。
- M5/M6 可按时间裁剪;M4 之后项目已可演示。

## 4. 每阶段的"最小可演示"原则

每个里程碑结束都应能**对外演示一句话能讲清的东西**:
- M1:"能注册登录了"
- M2:"后台任务能实时看进度,断线能重连"
- M3:"能上传 PDF 并自动入库,看得到处理进度"
- M4:"能和我上传的资料对话,带引用,且只能看到自己的"
- M5:"能让 agent 自动整理我的资料"
