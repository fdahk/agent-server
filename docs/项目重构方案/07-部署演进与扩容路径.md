# 07 · 部署演进与扩容路径

> 本文回答:**"多副本扩容用 Docker + k8s 是最佳方案吗?"** —— 我的诚实评估,以及分阶段的部署建议。
> 前置:02 §6 已确立"无状态 web 池 + worker 池 + Redis 队列/Pub/Sub backplane"的架构形态。本文讲**用什么工具把它跑起来**。

---

## 1. 先回答你的问题:**不,k8s 不是"最佳",至少现在不是**

**短回答:**

> **当前阶段(M0–M4 MVP + 校招演示):用 Docker Compose,单机扩副本就够,k8s 不要上。**
> **k8s 留到 M6 之后作为「能力展示」,用 k3d/k3s 跑一份 manifest 就够,生产不必真用。**

详细原因下面展开。先把这个判断说在最前面,免得文档读到一半还不知道结论。

---

## 2. 部署工具的真实选项(2026)

| 方案 | 适合 | 复杂度 | 我对本项目的判断 |
|---|---|---|---|
| **Docker Compose**(单机) | 1 台 VM,几个服务,solo 项目 | 低 | ✅ **本项目默认方案** |
| **Docker Compose 多机**(swarm-like) | 跨机,但 Compose 原生不强 | 中 | ⚠️ 能玩但生态淡 |
| **Docker Swarm** | Compose 思维 + 多机编排 | 中 | ❌ 生态停滞,被 k8s 淘汰中 |
| **k3s / k3d**(轻量 k8s) | 想要 k8s API 又不想运维大集群 | 中高 | ✅ **能力展示用** |
| **k8s 自建** | 生产、多节点、有运维团队 | 极高 | ❌ 个人项目过度 |
| **托管 k8s**(阿里云 ACK / 腾讯 TKE / GKE / EKS) | 同上但不用管控制面 | 高 | ⚠️ 想体验可,贵 |
| **PaaS**(Render / Fly.io / Railway / 阿里云 SAE) | 推代码即部署 | 低 | ⚠️ 简单但简历价值低 |
| **PM2 cluster** | 单 VM、无容器化、要利用多核 | 低 | ⚠️ 容器时代的过渡方案 |

## 3. 评估 k8s:对本项目是不是过度?

诚实拆解 k8s 的**真实成本** vs **真实收益**(对一个 solo校招 项目而言):

### 3.1 k8s 给你的(真实收益)

- **多节点编排** —— 但你现在通常只有 1 台服务器,这个能力闲置。
- **自动重启 / 副本自愈** —— Docker `restart: unless-stopped` 已经覆盖 90%。
- **声明式部署** —— 真好,但 Compose 也是声明式。
- **滚动更新** —— Compose `up -d` + 健康检查可以达到 80% 效果。
- **服务发现** —— Compose 内置(`service_name` 即 DNS)。
- **简历信号** —— 这是 k8s 在校招中**唯一无可替代的真实价值**。

### 3.2 k8s 要你付的(真实成本)

- **集群运维**:即使托管,你也要管 namespace / RBAC / network policy / ingress controller。
- **Manifest 生态**:Deployment / Service / ConfigMap / Secret / Ingress / HPA / PVC,每个都要写。
- **持久化**:Postgres/Qdrant/Redis 在 k8s 上做 StatefulSet + PV,门槛不低,生产风险高。
- **TLS / Ingress / cert-manager** 一整套要配。
- **Helm**:不学就靠手写 YAML,学了又是新栈。
- **调试**:`kubectl describe / logs / exec` + 网络/DNS 排障比 Compose 麻烦多了。
- **机器成本**:k8s control plane + 至少 2 节点 worker,$30–100/月起步,而单 VM Compose 可能 $5–20。

### 3.3 校招视角的真相

> **腾讯/字节这级别面试,听到"我用了 k8s",**下一个问题**永远是"你怎么做的 ingress?HPA 怎么配的?StatefulSet 数据库存储怎么处理的?滚更时长连接怎么不掉?"** 答不上来反而扣分。

vs.

> **"我用 Compose 部了单机多副本 + Nginx LB,设计上是 k8s 兼容的(无状态 web + 工作节点 worker + 外部状态),我也写了 k8s manifest 在 k3d 跑通,但生产不用是因为单机够 + 简化运维"** —— 这是**判断力**,远比"清单堆砌"加分。

## 5. 阶段 1:Docker Compose 单机部署(默认方案)

### 5.1 拓扑

```
                    Nginx(反代 + TLS + SSE 兼容)
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
   web × 2 副本      worker × M 副本    (按需扩缩)
   start:server        start:worker
        └────┬───────────┴─────┐
             ▼                  ▼
        Postgres            Redis(队列+pub/sub)
        Qdrant              Ollama(GPU 主机/本地)
```

### 5.2 关键 Compose 片段

```yaml
# docker-compose.yml(节选)
services:
  web:
    image: agent-server:${TAG}
    command: ["node", "dist/main.js", "--role=web"]
    environment:
      ROLE: web
      LLM_BASE_URL: ${LLM_BASE_URL}
      REDIS_URL: redis://redis:6379
      DATABASE_URL: postgres://...
    deploy:
      replicas: 2                # ← 副本数
    depends_on: [postgres, redis, qdrant]
    networks: [portal-net]

  worker:
    image: agent-server:${TAG}   # ★ 同一个镜像
    command: ["node", "dist/main.js", "--role=worker"]
    environment:
      ROLE: worker
      # 同 web,共享 Redis/Postgres
    deploy:
      replicas: 4                # ← 重活就开大
    depends_on: [postgres, redis, qdrant]
    networks: [portal-net]

  nginx:
    image: nginx:alpine
    volumes: ["./nginx.conf:/etc/nginx/nginx.conf:ro"]
    ports: ["443:443", "80:80"]
    networks: [portal-net]

  postgres: { image: postgres:16, ... }
  redis:    { image: redis:7-alpine, ... }
  qdrant:   { image: qdrant/qdrant:latest, ... }

networks:
  portal-net:
    driver: bridge
```

**扩缩命令**:

```bash
docker compose up -d                              # 起来
docker compose up -d --scale worker=8             # 摄取压力上来,扩 worker
docker compose up -d --scale web=4                # 并发上来,扩 web
docker compose up -d --no-deps --build web        # 滚动更新 web(注:不是真零停机)
```

### 5.3 Nginx 负载均衡(把同一服务名解析到多副本)

```nginx
upstream web_backend {
    server web:3000;             # ← Compose DNS 解析到所有 web 副本(轮询)
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name agent.yourdomain.com;
    ssl_certificate     /etc/nginx/certs/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/privkey.pem;

    location /runs/ {            # ← SSE 路由特殊处理
        proxy_pass http://web_backend;
        proxy_http_version 1.1;
        proxy_buffering off;     # ★ 流式必须关
        proxy_read_timeout 1h;   # SSE 长连接
        proxy_set_header Connection '';
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /conversations/ {   # ← chat 流式也走这个
        proxy_pass http://web_backend;
        proxy_buffering off;
        proxy_http_version 1.1;
    }

    location / {
        proxy_pass http://web_backend;
        proxy_http_version 1.1;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

呼应早期讨论的 nginx 反向代理 + resolver/upstream 路由原理。

### 5.4 阶段 1 已实现的能力

| 能力 | 实现 |
|---|---|
| 横向扩容(单机) | `--scale` 多副本 |
| 服务发现 | Compose 内置 DNS |
| 负载均衡 | Nginx upstream 轮询 |
| 健康检查 + 重启 | `healthcheck` + `restart: unless-stopped` |
| 日志 | `docker compose logs -f` + 文件持久 |
| 配置外置 | `.env` + ConfigMap-style env vars |
| 滚动更新 | `up -d --no-deps web`(非零停机,但够用) |

**够 MVP 和校招演示用了**。

## 6. 阶段 2:K8s 作为能力展示(M6 之后,可选)

**目标**:不用真跑生产 k8s,但要能讲出"我的应用是 k8s-native 设计"。

### 6.1 用 k3d 在本地跑一遍

`k3d`(k3s 的 Docker 封装)能在你的 Mac 上**几分钟拉起一个真 k8s 集群**:

```bash
brew install k3d
k3d cluster create agent --port 443:443@loadbalancer
kubectl apply -f k8s/
```

**关键收益**:k8s manifests 不是写来玩的,是真跑通的——这才有讲故事的底气。

### 6.2 Manifest 草案(以 web 为例)

```yaml
# k8s/web.yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: agent-web }
spec:
  replicas: 2
  selector: { matchLabels: { app: agent, role: web } }
  template:
    metadata: { labels: { app: agent, role: web } }
    spec:
      containers:
        - name: web
          image: ghcr.io/you/agent-server:latest
          args: ["node", "dist/main.js", "--role=web"]
          env:
            - { name: ROLE, value: web }
            - { name: REDIS_URL, value: redis://redis:6379 }
            - { name: DATABASE_URL, valueFrom: { secretKeyRef: { name: app-secrets, key: db-url } } }
            - { name: LLM_BASE_URL, valueFrom: { configMapKeyRef: { name: app-config, key: llm-base-url } } }
          ports: [{ containerPort: 3000 }]
          readinessProbe: { httpGet: { path: /health, port: 3000 } }
          livenessProbe:  { httpGet: { path: /health, port: 3000 }, periodSeconds: 30 }
---
apiVersion: v1
kind: Service
metadata: { name: agent-web }
spec:
  selector: { app: agent, role: web }
  ports: [{ port: 80, targetPort: 3000 }]
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: agent-web }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: agent-web }
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource: { name: cpu, target: { type: Utilization, averageUtilization: 70 } }
```

类似的写 `agent-worker.yaml`(Deployment + 无 Service,只消费队列)、`postgres-statefulset.yaml`(或干脆用外部 DB)、`redis-statefulset.yaml`、`qdrant-statefulset.yaml`、`ingress.yaml`(Nginx Ingress Controller + cert-manager)。

### 6.3 阶段 2 的简历讲法

> "Compose 是我的生产部署,k8s manifest 是同一架构的兼容验证——HPA 配 CPU 70% 阈值,2~10 副本弹性,readinessProbe / livenessProbe 都写了。生产没切是因为单机 Compose 已经满足 SLA,引 k8s 当前是负收益。"

这就是**判断力 + 能力**双重信号。

## 7. CI/CD 路径(沿用既有 GitHub Actions 模式)

| 阶段 | 路径 |
|---|---|
| 阶段 1(Compose) | GitHub Actions 构建镜像 → push GHCR → SSH 服务器拉镜像 → `docker compose up -d`(具体见早期讨论的 deploy.yml 形态) |
| 阶段 2(k8s) | GitHub Actions → 构建镜像 → `kubectl set image` 或 `kubectl apply -f k8s/`(用 kubeconfig secret) |

镜像两个阶段**完全相同**——这是"k8s-portable design"的实际意义。

## 8. 何时真要升级到 k8s(触发条件清单)

任何一条满足才考虑:

- [ ] 单机 CPU/内存常态化 80% 以上,需要真正多机
- [ ] 需要**跨可用区高可用**(单机宕机不能影响服务)
- [ ] 团队多人协作,需要 RBAC + 命名空间隔离
- [ ] 业务有强**蓝绿/金丝雀部署**需求(Compose 做不到优雅)
- [ ] 入职环境要求 k8s 经验

**你校招阶段一条都不满足**。所以阶段 1 + 阶段 2(showcase)就够。

---

## 9. 跨主机扩展(如果某天单机真不够)

如果一台机扛不住,有这几条不上 k8s 的路径(按推荐度):

1. **垂直扩**:换更大的 VM。最简单。
2. **拆服务到第二台机**:Postgres/Redis/Qdrant 移到专用机,app 服务器跑 web/worker。Compose 改 `host: server2.local`。
3. **Compose remote context**:`docker context create remote --docker host=ssh://server2`,可以在一台机指挥多台。
4. 实在还不够 → 上 k8s。

实践中 1+2 能扛到月活几十万,你校招阶段触达不到。

## 10. 总结决策表

| 项 | 阶段 1(默认) | 阶段 2(showcase) |
|---|---|---|
| 编排 | Docker Compose | k3d / k3s |
| 反向代理 | Nginx | Nginx Ingress Controller |
| 副本扩容 | `--scale` | Deployment.replicas + HPA |
| 服务发现 | Compose DNS | k8s Service |
| 配置 | `.env` + env vars | ConfigMap + Secret |
| 持久化 | Compose volumes | StatefulSet + PV(或外部) |
| 滚动更新 | `up -d --no-deps`(粗糙) | Deployment rolling update |
| TLS | Certbot + Nginx | cert-manager |
| 监控 | docker logs + 简单 metrics | Prometheus + Grafana(可选) |
| 部署成本 | 极低 | 学习 1–2 周,运维持续 |
