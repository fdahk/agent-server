# 06 · 前端接入与多端规划

> 本文规划 agent-server 的前端形态:**作为 our-chat 的"系统助手"集成 + Flutter 移动端扩展**。
> 核心定位:**agent-server 是纯后端 API,不绑定任何前端**;以下两个前端是它的"消费者",可以独立演化。

---

## 1. 总定位

```
        ┌──────────────────────────────────────┐
        │           agent-server               │
        │     纯后端 API(REST + SSE)         │
        │  /auth  /documents  /conversations    │
        │  /messages  /runs/{id}/stream  /agent │
        └─────────┬──────────────────┬─────────┘
                  │                  │
            HTTP/SSE/JWT       HTTP/SSE/JWT
                  │                  │
        ┌─────────▼─────────┐ ┌──────▼──────────┐
        │  our-chat (web)   │ │ Flutter (mobile)│
        │ React,作为系统助手 │ │ 移动端独立 app   │
        │  集成进 IM 联系人   │ │ 或 our-chat 端  │
        └───────────────────┘ └─────────────────┘
```

**关键原则**:
- agent-server **不知道也不在乎**前端是什么。它只暴露 API。
- 两个前端**完全独立**演进,共享同一套 API 契约。
- our-chat 与 agent-server **不合并代码库**(沿用前期决定:两个独立项目,API 集成)。

## 2. 前端 ↔ agent-server 的契约面(API 边界)

这是两个前端共同的契约,**先把它定下来**,前端就可以独立开发。

### 2.1 REST 端点(M0–M5 涵盖范围)

| 端点 | 用途 | 阶段 |
|---|---|---|
| `POST /auth/register` | 注册 | M1 |
| `POST /auth/login` | 登录返回 JWT | M1 |
| `POST /documents`(multipart) | 上传文件,返回 documentId + runId | M3 |
| `GET  /documents` | 列出当前用户文档 | M3 |
| `DELETE /documents/{id}` | 删文档(Postgres + Qdrant 都清) | M6 |
| `POST /conversations` | 新建会话 | M4 |
| `GET  /conversations` | 列出会话 | M4 |
| `POST /conversations/{id}/messages` | 发问,**流式返回 token**(SSE) | M4 |
| `GET  /conversations/{id}/messages` | 历史消息 + 引用 | M4 |
| `POST /agent/tasks` | 启动 agent 复合任务,返回 runId | M5 |
| `GET  /runs/{runId}/stream` | SSE 监听运行进度(摄取/agent 通用) | M2+ |
| `GET  /runs/{runId}` | 拿运行快照(用于断线重连后回放) | M2+ |

### 2.2 鉴权约定

- 所有受保护端点要求 `Authorization: Bearer <jwt>`。
- JWT 字段:`sub`(userId)、`username`、`role`、`exp`。
- 401 = token 无效/过期;403 = 越权(如查别人的文档)。

### 2.3 SSE 流式约定

- `Content-Type: text/event-stream`
- 每条事件:`event: <type>\ndata: <json>\n\n`,事件类型对齐 `RunEvent.eventType`。
- **断线重连**:客户端用 `Last-Event-ID`(或自定义 `?since=<sequenceNo>`),后端从数据库回放该序号之后的事件 → 衔接实时流。
- chat 流式回答:每个 token 一个 `event: token`,结束一个 `event: done`(带 citations)。

### 2.4 错误格式(统一)

```json
{ "code": "DOCUMENT_NOT_FOUND", "message": "...", "details": {} }
```

> 这套契约用 `@nestjs/swagger` 自动生成 OpenAPI,前端可选 codegen 类型(见 §7)。

## 3. 跨项目鉴权:our-chat ↔ agent-server

这是 our-chat 集成最关键的一道题。三个选项:

| 方案 | 体验 | 改动 | 我的建议 |
|---|---|---|---|
| **A. 独立账号** | 用户在 our-chat 和 agent-server 各登录一次 | 零 | ❌ 体验差,不要做 |
| **B. agent-server 为 IdP**(身份源) | 用户用 agent-server 账号登录 our-chat | our-chat 改 auth 实现 | 适合 agent-server 是主入口的场景 |
| **C. 共享 JWT 密钥**(共担密钥) | 一边签的 token,另一边能验 | 双方约定 secret/算法 + JWT 字段 | ✅ **推荐**:改动最小,两个项目仍独立 |

### 推荐方案 C 的落法

- 两个项目共享同一个 **JWT 签名密钥**(对称 HS256)或 agent-server 暴露 **JWKS**(非对称 RS256,更安全)。
- our-chat 现有的登录流程不变;它签发的 JWT,agent-server 的 `JwtAuthGuard` 直接能验。
- agent-server 的 `/auth` 仍保留(M1 还要做),用于:① Flutter 独立移动场景直接用;② 后期支持邮箱注册等独立用户路径。
- **用户身份同步**:第一次 our-chat 用户访问 agent-server 时,agent-server 按 JWT 里的 `sub`/`username` 在本地 `User` 表里 `findOrCreate`(zero-touch 同步)。

### 落地约束

- 双方 JWT 算法、密钥、issuer、字段名必须**严格一致**——写到 `docs/项目重构方案/契约/` 一个公共文件里(本期可放到本文档附录)。
- 密钥从环境变量读,代码里别写死。
- Refresh token 策略:可选,初期可以不做(长 expiry token + 401 强制重登)。

## 4. Web 端:在 our-chat 内做"系统助手"

### 4.1 形态

在 our-chat 的联系人列表里,**置顶一个虚拟联系人 "AI 助手"**(类似微信的"文件传输助手")。点进去 = 一个特殊的会话窗口,但消息流走 agent-server 而非 socket.io。

```
┌───────── our-chat 联系人列表 ──────────┐
│  📌 AI 助手 ← 虚拟联系人,点进去打开 ↓ │
│  👤 张三                                │
│  👤 李四                                │
│  ...                                    │
└─────────────────────────────────────┘
                  │
                  ▼
┌──────────── AI 助手会话窗口 ───────────────┐
│  顶栏:[会话切换 ▼] [+ 新会话] [📚 我的知识库] │
│  ─────────────────────────────────────  │
│  [消息流,渲染 user/assistant + 引用来源]    │
│  ─────────────────────────────────────  │
│  [输入框] [📎 上传文档] [发送]               │
└──────────────────────────────────────────┘
```

### 4.2 our-chat 侧的具体改动(估算)

| 改动 | 范围 |
|---|---|
| 联系人列表注入"AI 助手"虚拟项 | 小:加一个 hard-coded entry |
| 新组件 `AgentChatPanel`(替代 IM 消息窗口的内容区) | 中:RAG chat UI + 文件上传 |
| 新 axios 实例 `agentClient`(指向 agent-server)+ 拦截器加 JWT | 小 |
| 文档列表/管理侧栏(可选) | 中 |
| SSE 流式 token 渲染 | 中 |

> 这些改动**全在 our-chat 仓库内**,agent-server 仓库不动。

### 4.3 复用 our-chat 的现有能力

- antd 组件库 → 直接用
- redux store → AgentChatPanel 自己的状态可挂在它下面
- 现有 axios 拦截器 / 401 处理 → 复用
- WebRTC、socket.io → AI 助手用不到,与之并存

## 5. Flutter 端:移动扩展规划

### 5.1 形态选择

| 选项 | 说明 | 适合 |
|---|---|---|
| **A. 独立"AI 助手 App"** | Flutter 单独做一个 agent 客户端,不包含 our-chat IM | 想快速演示 + 简化范围 |
| **B. Flutter 重做整个 our-chat 移动端 + 内嵌 AI 助手** | 把 our-chat 移植到 Flutter,顺带集成助手 | 体现"web+mobile 全栈" + IM 完整体验 |

**建议**:从 A 起步(范围可控,2–3 周可演示),时间允许再考虑 B。

### 5.2 关键技术点

| 关注点 | Flutter 方案 | 备注 |
|---|---|---|
| HTTP/JWT | `dio` + 拦截器统一加 `Authorization` | 401 → 自动跳登录 |
| **SSE** | `flutter_client_sse` 或 `dio` + `ResponseType.stream` 手动解析 | Dart 原生 SSE 支持比浏览器粗糙,需要自己处理 `data:` 行分割与 reconnect |
| 多 tab/后台 SSE | iOS/Android 后台会断 SSE,**回前台用 `Last-Event-ID` 重连**(契约 §2.3 已支持) | 移动端必须设计断线恢复 |
| 文件上传 | `file_picker` 选文件 → `MultipartFile` POST | iOS/Android 权限要处理 |
| 流式 chat UI | `StreamBuilder` 渲染逐 token 文本;输入框 + "正在思考"状态 | UX 细节多 |
| 安全存储 | `flutter_secure_storage` 存 JWT(Keychain/Keystore) | 不要塞 SharedPreferences |
| 路由 | `go_router`(声明式,Web 类似) | 与浏览器路由习惯一致 |
| 状态管理 | `riverpod`(基于你既有 Flutter 经验) | 类 React Hooks |
| API 类型 | 用 `freezed` + `json_serializable` 生成 DTO,可结合 OpenAPI codegen | 见 §7 |

### 5.3 移动端的 SSE 是真坑

> 这一条单独拎出来——Flutter SSE 比浏览器粗糙,**端到端要专门测**:

- iOS 应用进后台 ~30 秒后系统就可能切断 socket → 必须在前台恢复时重连。
- 弱网下 dio stream 可能"挂着但不收数据",需要**心跳 + 超时**(后端每 15s 发一个 `event: heartbeat`,前端超过 30s 没收到任何事件就主动重连)。
- 重连必须带 `Last-Event-ID`(契约层已支持),否则会丢事件 + 重复事件。

后端做的事(已经在 02/03 涵盖):事件落库 + `sequence_no`,允许从任意 `sequenceNo` 之后回放。

### 5.4 Web 与 Flutter 共用 API 的天然好处

- 同一套契约 → 后端只维护一份
- 同一套 SSE 重连协议 → 经验复用
- 一份 OpenAPI → 两端各自 codegen

## 6. 鉴权与多端账号矩阵

| 场景 | 鉴权来源 | 备注 |
|---|---|---|
| 浏览器进 our-chat → 用 AI 助手 | our-chat 签的 JWT(方案 C 共享密钥) | 用户感知不到 agent-server 的存在 |
| Flutter 独立 AI 助手 App | 直接调 agent-server `/auth/login` | 移动端用户的"主账号"是 agent-server 的 |
| Flutter 版 our-chat(可选 B) | 同 web our-chat 路径 | 用 our-chat 账号登录 |

如果 web 和 Flutter 都允许同一用户使用,**最简单的做法**:用户用 our-chat 账号登录(web 在 our-chat 内,Flutter 调一个 our-chat 的登录 API,或干脆 Flutter 自己也直接走 agent-server `/auth`,只要双方账号能对得上即可——这就是为什么 §3 推荐方案 C)。

## 7. 跨端类型与契约共享(可选,加分)

| 选项 | 做法 | 何时上 |
|---|---|---|
| **手写**(初期) | 各端各自维护 TS / Dart 类型,手工同步 | M0–M4 完全够 |
| **OpenAPI codegen** | `@nestjs/swagger` 输出 OpenAPI → web 用 `openapi-typescript`,Flutter 用 `openapi-generator` 生成 Dart 客户端 | 接口稳定后(M4 之后) |
| **共享 schema 包** | 用 zod / json-schema 共享,各端引用 | 重度生产场景,本项目不需要 |

**建议**:M0–M4 手写,M5/M6 时上 OpenAPI codegen 当作"工程化升级"的故事。

## 10. 简历叙事

> "我设计了一个纯后端 AI agent 平台,前端解耦——通过 OpenAPI 契约和共享 JWT 密钥,**同时支持 web(集成进 IM 项目作为系统助手)和 Flutter 移动端**两条消费路径。后端无感知前端形态;新增第三种客户端只需对接同一套 API。"

这个故事三块亮点:
1. **后端纯 API 化的设计判断力**(不是"我做了一个全栈项目",而是"我让后端可以被多种前端独立消费")
2. **跨项目 SSO 实现**(共享密钥 / JWKS,实践 OAuth 资源服务器模式)
3. **多端 SSE 流式难题的处理**(尤其 Flutter 端的断线重连 + 心跳)
