## 1. 服务拆分:合并为单一 Node 模块化单体

**结论:删掉 Java,把它的能力搬进一个 Node 服务,与现有 Node 网关合并成一个进程。**

理由(已在前期讨论确认,这里固化):

- 当初拆 Java Core ↔ Node Gateway 的**唯一**理由是"Java 干平台、Node 干 AI"的语言分工。语言统一成 Node 后,这条 `X-Internal-Token` 的内部 HTTP 调用就是**纯开销**:多一次网络往返、多一套服务间鉴权、多一层序列化,换不来任何收益(你没有独立扩缩容需求)。
- 合并后,原"gateway 调用"从 HTTP 降级为**进程内方法注入**,一整类故障(网络抖动 / token 失配 / 序列化错误)直接消失。

**保留边界(关键)**:虽然合成一个进程,但**AI/检索能力仍封装为独立模块 + 清晰接口**(镜像原 `AgentAiGateway` 端口)。这样:① 将来真要独立扩缩容(AI 推理是慢操作),能原样抽成独立服务;② 面试时"我懂服务边界、也懂何时不该拆"的故事还在,而且比"我拆了微服务"更显成熟。

**何时不该这么选**:如果 AI 推理需要独立的 GPU 机器池、独立扩缩容节奏,或团队按服务分工——那才值得拆。你现在都不满足。

### 1.1 单体 ≠ 不能横向扩容:用 web 池 + worker 池

"合并成单体会不会丢横向扩容能力?" —— **不会**。单体做成**无状态**就能多副本横向扩容。单体真正放弃的是**差异化扩容**(只扩重的、不扩轻的),而这件事用 **worker 角色拆分**解决,不用拆微服务。

**做法:同一份代码库,两种启动角色,各自独立扩副本。**

```
        ┌──────────── Load Balancer ───────────┐
   ┌────▼─────────┐                  ┌──────────▼──────────┐
   │  Web 池 × N   │   Redis(队列)   │   Worker 池 × M      │
   │ auth/chat/    │◄───────────────►│ 消费 BullMQ:         │
   │ upload/SSE/   │                 │ 解析/切分/embedding/ │
   │ 检索           │                 │ 入 Qdrant            │
   └────┬──────────┘                 └──────────┬──────────┘
        │  (start:web)   同一份代码   (start:worker)         │
        ▼                                                   ▼
   Postgres / Qdrant                                  Ollama(GPU,独立)
```

- **轻的(auth/编排/chat)放 web 池**,无状态,扩 N 副本扛并发。
- **重的(摄取:解析+编排 embedding)放 worker 池**,压力大就把副本从 2 扩到 10,独立于 web。
- **最重的推理(embedding/生成)本就在 Ollama**(独立进程),扩它是 GPU 层的事,和应用架构正交。

**为什么不拆成两个微服务**:把 auth/编排 和 资源处理拆成独立服务,等于把刚删掉的 `X-Internal-Token` 内部 HTTP 边界**原样重建**——重新引入服务间网络/鉴权/序列化 + 跨服务数据一致性 + 双份部署运维,而"独立扩容"worker 角色拆分已经免费给了。付微服务的税,买不到额外收益。

**保留逃生口**:`shared/rag`、`shared/llm`、`agent` 都在清晰接口后面,将来真出现瓶颈可原样抽成独立服务——即业界默认的 "Monolith First,有真瓶颈再抽"。

**何时真要拆服务**:重计算搬进了你自己的进程(不再靠 Ollama)且需独立 GPU 节奏 / 不同子系统 SLA 截然不同需故障隔离 / 重新 polyglot(如 Python rerank)/ 团队按服务分工。你现在都不满足。

> NestJS 用 `NestFactory.createApplicationContext()` 起一个无 HTTP 监听的 worker 进程,与 web 共享全部 module。部署拓扑详见 `02` 第 6 节。

## 2. Web 框架:NestJS(保留),可降级 Fastify

**结论:用 NestJS,单服务。** ⟲ **修正**:前期我曾倾向 Fastify,理由是"你在逃离 Spring 的仪式感"。在看清实际重构范围后我改回 NestJS,原因如下——

**为什么改判**:这次重构不是写一个薄 API,而是要落地**多个子系统**:auth、文档摄取、向量检索、对话、agent 编排、SSE 流式、文件上传、队列。这是一个会持续长大的多模块应用。在这种规模下:

- NestJS 的 module / DI / Guard / Interceptor / `@Sse()` / ValidationPipe / Swagger 提供的结构**有真实价值**,不是负担。
- 你 Java 侧的 DDD 分层(`interfaces→application→domain→infrastructure`)**可以 1:1 映射**到 NestJS module,移植是机械的,不用重新设计。
- 现有 Node 侧本来就是 NestJS,延续它零迁移成本。

**关键澄清**:你想"降的复杂度"是 **polyglot(Java+Node 双语言)** 的复杂度,**不是** NestJS 的结构。NestJS 的仪式感(装饰器/DI)比 Spring+MyBatis+Flyway+双语言**轻一个量级**。砍掉 Java 已经解决了 80% 的心智负担。


|         | NestJS(选)              | Fastify(备选)            |
| ------- | ---------------------- | ---------------------- |
| 结构      | 框架给(module/guard/pipe) | 自己搭(plugin 封装 + 自选 DI) |
| 多模块大应用  | ✅ 强项                   | ⚠️ 要自律维护结构             |
| Java 移植 | 1:1 映射,机械              | 要重新组织                  |
| 心智负担    | 中(装饰器/DI)              | 低                      |
| 中国公司常见度 | 高(字节等大量使用)             | 中                      |


**这是全方案里唯一可翻转的决策**:如果你强烈偏好极简、愿意自己搭结构、不介意手写 auth/DI,那换 Fastify 也成立。翻转代价仅限于"自己实现 NestJS 替你做的那部分脚手架",其余架构/数据模型/流程全不变。

**何时该选 Fastify**:项目确定不会长大、追求极致轻量与启动速度、或你想用"全部自研架构"作为简历叙事。

## 3. ORM:Prisma

**结论:用 Prisma。** ⟲ **修正**:前期我曾倾向 Drizzle,理由是"RAG 项目要 pgvector,而 Prisma 的 pgvector 支持弱"。**这个前提被 Qdrant 的引入推翻了**——

**为什么改判**:你决定用 **Qdrant 存向量**,那么 **Postgres 就只存纯关系数据**(users / documents 元数据 / chunks 元数据 / conversations / messages / runs / events)。Postgres 里**根本没有 pgvector**,Prisma 的那个短板**直接消失**。于是 Prisma 的优势(顶级 DX、类型安全、迁移体验、你本来就偏好)全部成立,没有任何掣肘。


|           | Prisma(选)          | Drizzle(备选) |
| --------- | ------------------ | ----------- |
| DX / 类型安全 | 顶级                 | 很好,更贴 SQL   |
| 迁移        | Prisma Migrate,体验好 | drizzle-kit |
| pgvector  | 弱(但**本项目用不到**)     | 强           |
| 复杂原生 SQL  | 一般(需 `$queryRaw`)  | 强           |


**何时该回 Drizzle**:如果你后来决定**不用 Qdrant、改用 pgvector**(把向量也塞进 Postgres),那立刻换 Drizzle——这是它俩的真正分水岭。

## 4. 向量库:Qdrant

**结论:用 Qdrant。**(你的选择,我认可,并给出对比依据。)

Qdrant 是 Rust 写的专用向量库,对本项目是好选择:

- **过滤能力强**:支持 payload 过滤(`filter: must user_id == X`),这对**多租户隔离**至关重要——每个用户只能检索自己的 chunk。
- **TS 客户端成熟**:`@qdrant/js-client-rest`,也有 `@langchain/qdrant` 集成,接 LangChain 顺。
- **运维简单**:Docker 一个镜像起,自带 Web Dashboard,本地开发友好。
- **简历信号好**:用专用向量库比"在 Postgres 里塞 pgvector"更能体现"我搭过真正的向量检索基础设施"。


| 方案            | 优点                    | 缺点         | 适合                     |
| ------------- | --------------------- | ---------- | ---------------------- |
| **Qdrant**(选) | 过滤强、专用、Dashboard、扩展性好 | 多一个组件要运维   | 想展示真实向量基础设施            |
| pgvector      | 一个库搞定、运维最省            | 大规模/复杂过滤偏弱 | 极简、向量量小                |
| Weaviate      | 功能全、自带混合检索            | 偏重、概念多     | 企业级、需要内置 hybrid search |
| Milvus        | 超大规模                  | 运维重        | 亿级向量,你用不到              |


**何时不该选 Qdrant**:如果你想把组件数压到最少、向量量很小(几万级),pgvector + Drizzle 反而更省心。这是和 §3 ORM 决策联动的一组选择。

## 5. RAG/Agent:**分层选型,不绑单一框架** ⟲ 修正

**结论:LangChain 不是框架,只作工具包;检索层和 agent 循环自研。**

⚠️ 这里要正面修正之前对"用 LangChain"的默认推荐。之前我锚定了"既然做 RAG/agent 就用 LangChain"这个先验,没有真正再推一遍。重做后的判断是:**一个 RAG/agent 应用其实有 4 个可独立选型的层,把它们捆给 LangChain 是默认行为而非最优选择**。

### 5.1 分层选型表

| 层 | 干什么 | 选 | 不选 LangChain 的核心原因 |
|---|---|---|---|
| **L1 文档插件** | PDF/docx/web 加载 + 切分 | **LangChain 单包**:`@langchain/community`(loader)+ `@langchain/textsplitters` | 这是 LangChain 在 TS 端唯一明显的生态红利,**单独 import 不引 chain 抽象** |
| **L2 模型调用** | chat/stream/tool/embedding | **裸 `openai` SDK**(OpenAI 兼容,M0 已锁) | 协议层已统一,LangChain 再包一层 = 负价值 |
| **L3 检索** | embed query → Qdrant 搜 → 拼上下文 | **自写在 `shared/rag`** + `@qdrant/js-client-rest` 裸客户端 | LangChain 的 `QdrantVectorStore` 会**遮蔽 `user_id` 过滤**这个安全关键点;且本来就 50 行 |
| **L4 Agent 循环** | 多步推理 + 工具调用 + 进度可审计 | **自研循环 + 复用现有 run-engine**(FSM + 事件溯源) | LangGraph 的 checkpoint 会和你自有审计层打架;自研循环是 ~200 行清晰代码 |

### 5.2 LangChain 工具包怎么用(不引入 chain)

```ts
// ingestion worker —— 只 import 工具,不 import chain/runnable
import { PDFLoader }    from '@langchain/community/document_loaders/fs/pdf';
import { DocxLoader }   from '@langchain/community/document_loaders/fs/docx';
import { RecursiveCharacterTextSplitter } from '@langchain/textsplitters';

const docs   = await new PDFLoader(filePath).load();
const splits = await new RecursiveCharacterTextSplitter({
  chunkSize: 500, chunkOverlap: 50,
}).splitDocuments(docs);
// 之后:用 openai SDK 算 embedding → 用 qdrant 客户端 upsert(自己写,见 03)
```

**关键约束**:不 import `@langchain/core`/`Runnable`/`RunnableSequence`;不 import `@langchain/qdrant`;不 import `@langchain/openai` 包装。这些"chain 抽象"层在本项目里是负价值。

### 5.3 自研 Agent 循环的形态(L4)

不引入任何 agent 框架(LangGraph/Mastra 等都不引),用一个清晰的循环 + 你已有的 run-engine 落事件:

```ts
// agent/agent-runner.ts —— 概念骨架,~200 行可落地
async function runAgentLoop(task: string, userId: bigint, runId: string) {
  const ctx = { runId, userId };
  let history: Message[] = [{ role: 'user', content: task }];

  while (true) {
    // 1. 调 LLM,带工具定义(OpenAI tool calling)
    const resp = await llm.chat.completions.create({
      model, messages: buildPrompt(history), tools: toolSchemas, stream: false,
    });
    const msg = resp.choices[0].message;
    history.push(msg);

    // 2. 若 LLM 决定调工具
    if (msg.tool_calls?.length) {
      for (const call of msg.tool_calls) {
        await runEngine.emit(ctx, 'tool_called', { name: call.function.name, args: call.function.arguments });
        const result = await toolRegistry.run(call.function.name, JSON.parse(call.function.arguments), ctx);
        await runEngine.emit(ctx, 'tool_result', { name: call.function.name, result });
        history.push({ role: 'tool', tool_call_id: call.id, content: JSON.stringify(result) });
      }
      continue;
    }

    // 3. 无工具调用 = 最终回答
    await runEngine.emit(ctx, 'final_answer', { content: msg.content });
    break;
  }
}
```

工具集(M5 落地):`retrieve_knowledge` / `list_documents` / `summarize_document` / `organize`——每个都是普通函数,参数和返回 JSON 即可,**不依赖任何框架**。

### 5.4 为什么这个决策更值得讲

| 普通版本 | 你的版本 |
|---|---|
| "我用了 LangChain 做 RAG" | "我评估了 LangChain.js,**只用了它的 loader/splitter 工具包**,检索强制 `user_id` 过滤握在 `shared/rag` 里,agent 循环用 OpenAI tool calling 自研 200 行,**复用我已有的 FSM + 事件溯源**做审计——既享生态又不被框架接管" |

这是**分层选型 + 知道何时不该用框架**的故事,比"我用了 X 框架"高一档。

### 5.5 何时回头加框架

- LangGraph:如果你后续要做**复杂的图结构 agent**(分支/合并/并行节点),自写循环开始痛了——再引,只引 LangGraph,不引 LangChain core。
- 但 M5 范围内的"工具调用循环",自研足够,不需要框架。

## 6. 异步队列:BullMQ(顺便让 Redis 复活)

**结论:用 BullMQ 做摄取流水线的异步队列;删除 RabbitMQ。**

- 文档摄取(解析→切分→embedding)是**慢操作**,必须异步:上传接口立即返回,后台 worker 处理,进度走 SSE。
- 你本来就要跑 Redis,BullMQ 是 Redis 原生队列,**正好把"声明了却没用的 Redis"变成承重组件**(队列后端 + embedding 缓存 + 会话)。这本身就是一个好的重构叙事:"把 infra theater 变成真实负载"。
- **删 RabbitMQ**:它当初是为"服务间异步派发"准备的,单体化后这个理由消失。单体里用 BullMQ 覆盖所有异步,不为一个队列再背一个 broker。


|     | BullMQ(选)        | RabbitMQ(删)       |
| --- | ---------------- | ----------------- |
| 依赖  | 复用已有 Redis       | 额外 broker         |
| 能力  | 任务队列、重试、进度、限流、调度 | 复杂路由、fan-out、多消费者 |
| 适合  | 单服务异步任务          | 跨服务事件总线           |


**何时该回 RabbitMQ/Kafka**:将来真要做跨服务 fan-out、多消费者路由、强投递保证或事件流——你现在都不需要。

## 7. 模型层:OpenAI 兼容接口前置抽象(关键架构决策)

**结论:从 M0 起,所有 LLM/embedding 调用只走 OpenAI 兼容协议;Ollama 本地开发 + 第三方 API 生产,通过配置一键切换,业务代码零改动。**

### 7.1 为什么必须前置做这层抽象

- 你已明确"未来用第三方 API 取代 Ollama"。如果业务代码一开始就直接绑 Ollama 私有 API,**未来切换 = 全应用搜替换 + 逐处验证**;前置抽象则是改 `.env` 的事。
- 业内 2026 年事实标准已收敛到 OpenAI 协议:**Ollama、vLLM、TGI、SGLang、DeepSeek、通义、智谱、豆包、硅基流动、Groq、Together、Fireworks、OpenAI、Anthropic(经兼容层)** 全都暴露 `/v1/chat/completions` 和 `/v1/embeddings`。一套客户端通吃。
- 换 API 后,你进程内不再做任何 AI 计算——这反过来巩固了**单 Node 单体**的选择(无 Python ML 重活可言)。

### 7.2 怎么落

```ts
// shared/llm/openai-compatible.client.ts —— 唯一的模型层入口
const llm = new OpenAI({
  baseURL: env.LLM_BASE_URL,     // Ollama: http://localhost:11434/v1
                                 // 第三方: https://api.deepseek.com/v1 等
  apiKey:  env.LLM_API_KEY,      // Ollama 可填 'ollama' 占位
});
// chat:  llm.chat.completions.create({ model, messages, stream: true })
// embed: llm.embeddings.create({ model, input })
```

- **chat 默认开 stream**,逐 token 经 SSE 推前端。
- **embedding 批处理**(`input` 接数组),减少往返。
- 用官方 `openai` SDK 即可,不用 Ollama 私有客户端;LangChain 集成也都基于这个协议。

### 7.3 配置矩阵(切换示例)

| 场景 | `LLM_BASE_URL` | `LLM_CHAT_MODEL` | `LLM_EMBED_MODEL` | `LLM_API_KEY` |
|---|---|---|---|---|
| 本地开发(默认) | `http://localhost:11434/v1` | `qwen2.5:7b` | `nomic-embed-text` | `ollama` |
| DeepSeek | `https://api.deepseek.com/v1` | `deepseek-chat` | (用别家 embed) | 真实 key |
| 硅基流动 | `https://api.siliconflow.cn/v1` | `Qwen/Qwen2.5-72B-Instruct` | `BAAI/bge-m3` | 真实 key |
| 通义 / 智谱 | 各自的 OpenAI 兼容入口 | 各自模型名 | 各自 embed | 真实 key |
| 自托管 vLLM | `http://gpu-host:8000/v1` | 模型名 | (TEI 单独) | 任意 |

### 7.4 与其他决策的连锁

- **维度坑(05 §1.1)**:换 embedding 模型 = 向量维度可能变 = Qdrant collection 要重建。**collection 名建议带模型标识**(如 `chunks_nomic768`、`chunks_bgem3_1024`),环境变量切换时校验维度。
- **chat 与 embed 可以分别配置**:有些第三方 chat 强但 embed 不提供,允许"chat 用 A、embed 用 B"组合。`shared/llm` 暴露两个 client 实例即可。
- **降级策略保留**:沿用现有 Ollama provider 的"失败降级"思路,封装在新客户端里(超时、重试、出错时返回兜底文本/向量)。

## 8. 这套选型如何回应你的两个诉求

1. **降复杂度**:5 个语言/框架(Java/Spring/MyBatis/Flyway/Nest)→ 1 套 Node 栈(Nest + Prisma);2 个进程 → 1 个;2 个中间件(Redis 摆设 + RabbitMQ 摆设)→ 1 个真实承重的 Redis。复杂度实质性下降。
2. **引新技术补能力**:Qdrant + LangChain/LangGraph + embedding + BullMQ 摄取流水线,把缺失的 RAG 主线补齐。
